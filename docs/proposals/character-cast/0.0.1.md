# Lute Character & Cast — Capability Proposal

- **Version:** 0.0.1
- **Status:** Draft (pre-implementation).
- **Scope:** the normative **language contract** for character identity, display label, costume,
  name-reveal, and voice-join in a `.lute` scenario — as a **capability plugin** (`character/cast`).
  Covers the character **registry** (provider), the `cast:` **frontmatter**, the `scene.cast.<id>.*`
  **state**, the compile-time **resolution** of sprite / display-name / voiceKey, the `seal`/
  `reveal`/`wear` **directives**, and **migration** of legacy content.
- **Out of scope:** the compiler/runtime **implementation** (deferred; a future Rust core — this
  document specifies *what* it must produce, not *how*). The design **rationale + audit trail** is
  [`design.md`](design.md).
- **Depends on:** Lute Scenario DSL `^0.0.1` ([`../scenario-dsl/0.0.1.md`](../scenario-dsl/0.0.1.md))
  with the §6.1 *plugin-contributed meta keys* amendment and the `as` line attribute; the Lute
  Plugin System `^0.0.1` ([`../plugin-system/0.0.1.md`](../plugin-system/0.0.1.md)) with the
  `frontmatter` export and `assetkinds` (§6.9); plus a `CH` asset kind (asset/core capability)
  with segments `characterId`/`costume`/`emotion`/`variant`.
- **Reference example:** [`../../examples/date-minigame.lute`](../../examples/date-minigame.lute)
  (cast usage to be reflected there when examples are refreshed).

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHOULD**, **SHOULD NOT**, **MAY** are to be
interpreted as in RFC 2119.

---

## 1. Abstract

A content line's speaker id (`:<id>{…}: text`) carries **only** the stable `characterId` (identity —
the key for sprite lookup, voice/line joins, and state). Three further concerns are modelled
separately, never packed into the speaker id:

1. **display label** — the name the dialog box shows (`???`, a mask, or the real name);
2. **costume** — an appearance variant of the same identity (the `CH` asset kind's `costume` segment, plugin-system §6.9);
3. **name-reveal** — the `???`→real-name transition, modelled as first-class state that flips once
   and affects all later lines.

All mutable cast state is `enum`/`bool` (satisfying the language's `number|bool|enum` state types,
`../scenario-dsl/0.0.1.md` §9.3); label strings are **static** configuration; and everything reduces
to flat command records + CEL, preserving the language invariants (total, non-Turing).

## 2. Relationship & dependencies (normative)

This capability is a plugin. It REQUIRES:

- **Language §6.1 plugin-contributed meta keys** — so `cast:` is a legal frontmatter key owned by
  this plugin.
- **Language content-line `as` attribute** — the one-off display-label override (§6).
- **Plugin-system `frontmatter` export** — so this plugin can declare and validate the `cast:` key
  (`../plugin-system/0.0.1.md`).
- **Plugin-system providers, state templates, directives, and named lowering hooks** — for the
  registry (§3), the `scene.cast.<id>.*` template (§5), and the directives/resolution (§7–§8).

A conforming consumer (checker/LSP/compiler) MUST validate a document against the single resolved
capability snapshot that includes this plugin.

## 3. Character registry (provider)

The registry is global, static character identity data, supplied as a plugin **provider**
(snapshot-first, `../plugin-system/0.0.1.md` §10). Per character id:

```yaml
bianca:
  name: { ko: "비앙카", en: "Bianca" }   # REQUIRED — canonical display name per language
  costumes:                               # REQUIRED — each costume carries its OWN emotion/variant domain
    - { key: default,  emotions: [neutral, content, shy], variants: [0, 1] }
    - { key: waitress, emotions: [content, delighted, surprised], variants: [0, 1] }
    - { key: casual,   emotions: [neutral, shy], variants: [0] }
  defaultCostume: default                 # REQUIRED — a costume `key`
ann:
  name: { ko: "앤", en: "Ann" }
  costumes: [ { key: default, emotions: [delighted, puzzled], variants: [0, 1] },
              { key: child,   emotions: [delighted, puzzled], variants: [0] } ]
  defaultCostume: default
  costumeVoiceBank: { child: ann_child }  # OPTIONAL — per-costume static voice bank (§7.3)
fixer: { kind: player }                   # POV/protagonist — no sprite, name = {{userName}} (§3.1)
store_staff: { kind: npc, name: { ko: "점원", en: "Staff" } }   # spriteless NPC (§3.1)
emotionGroups:                            # shared; the CH assetKind `emotionGroup` fallback reads this
  joy: { rep: delighted, members: [delighted, content] }
```

| Field | Req. | Rule |
|---|---|---|
| `kind` | MAY | `character` (default) \| `player` \| `npc` \| `narrator` — see §3.1 |
| `name` | MUST* | map language code → display string (*omitted for `player`, which renders `{{userName}}`) |
| `costumes` | MUST* | list of `{ key, emotions[], variants[] }`; MUST include a `default` key (*omitted for `player` / spriteless `npc`) |
| `defaultCostume` | MUST* | a costume `key` (*n/a for spriteless) |
| `costumeVoiceBank` | MAY | map costume key → static voice-bank id (§7.3) |
| `emotionGroups` | MAY | top-level (not per-character): `{ group: { rep, members[] } }` for the `emotionGroup` fallback |

The `character` id domain is owned by this provider. A content-line/`::auto`/`cast:` reference to an id
absent from a **fresh** registry snapshot is a static error; absent from a **stale** snapshot is a
*catalog-stale* diagnostic, not a hard error (`../plugin-system/0.0.1.md` §10).

### 3.1 Speaker kinds (character · player · npc · narrator)

`kind` distinguishes four speaker kinds:

- **`character`** (default) — a full sprite-bearing character (name + costumes); §5–§9 apply.
- **`player`** — the POV/protagonist. Its **id is project-defined** (Idola uses `fixer`; other games
  use e.g. `user`, `doctor`, `adventurer`), named per scene by the `pov` frontmatter. It has **no
  sprite** and **no `cast` state**; its display label renders as the runtime **`{{userName}}`** (the
  player's own name, not a registry name), and its `voiceKey`/lineId normalize to the `pov` speaker
  id so all player lines share a counter. A `player` line with `delivery="thought"` is the player's
  inner **monologue**; `seal`/`reveal`/`wear` do not apply.
- **`npc`** — a **spriteless** speaker (no `costumes`): `:store_staff` emits a speaker+text line
  with **no sprite** and no stage bookkeeping (an off-stage / bit-part voice). It MAY carry a `name`
  (else the id is the label). Ad-hoc unregistered ids are a lint (register even lightly).
- **`narrator`** — a reserved **speakerless** kind: `:narrator: text` renders neutral narration
  with **no speaker name and no sprite** (this replaces bare prose). No `cast` state; seal/reveal/wear
  and costume do not apply.

**Monologue is not player-only.** Any non-`narrator` speaker MAY carry `delivery="thought"`; for a
non-`player` character it renders as that character's own inner **monologue** (its voice, not the
player's), per the DSL 0.1.0 §12 `delivery`→role derivation table. Only `narrator` forbids a
`delivery` attribute (a static error).

A `character` with an empty costume set is likewise **spriteless** (voice-only). Only sprite-bearing
kinds resolve a `CH` assetId (§7.1); player / npc / narrator / spriteless skip sprite resolution.

## 4. `cast:` frontmatter (plugin-owned)

`cast:` is a frontmatter meta key owned by this plugin (legal via language §6.1 + plugin-system
`frontmatter` export). It seeds per-episode initial per-character state and static labels:

```yaml
cast:
  bianca:
    costume: waitress                 # OPTIONAL enum ∈ registry.costumes; default registry.defaultCostume
    sealed: true                      # OPTIONAL bool; default false
    maskAs: "은발의 종업원"           # OPTIONAL static string; default "???"
  takeru: { costume: default }         # sealed omitted → false (real name); maskAs unused
```

- Each key MUST be a `character` id present in the registry (§3); an unknown id is a static error.
- `costume` (if present) MUST be a member of that character's `registry.costumes`.
- `sealed` and `costume` seed the corresponding **state** (§5). `maskAs` is **static** (not state).
- A character not listed in `cast:` uses defaults (`costume = defaultCostume`, `sealed = false`,
  `maskAs = "???"`).

## 5. Cast state (normative)

This plugin declares, as a state template keyed by the registry's `character` id domain
(`../plugin-system/0.0.1.md` §6.3/§7.4), for every referenced character `<id>`:

- `scene.cast.<id>.costume : enum[<registry.costumes for id>]` — default `registry.defaultCostume`,
  overridden by `cast:` (§4).
- `scene.cast.<id>.sealed : bool` — default `false`, overridden by `cast:` (§4).

Both are `enum`/`bool`, satisfying the language state types (§9.3). They are **`scene`-scoped**
(episode lifetime; reset at episode end, re-materializing defaults). Because `sealed` defaults
`false`, a later episode that omits a character from `cast:` shows the real name automatically;
cross-episode "already known" needs no persisted flag. A rare multi-episode masked arc MAY bind the
mask to a declared `run.*` fact instead (author-declared; outside the auto-declared template).

`maskAs` is **static** per-character-per-episode data from `cast:` (or the default `"???"`); it is
**not** a state path and MUST NOT be written by `::set`.

Cast state is mutated only by `::set` (`../scenario-dsl/0.0.1.md` §7.3.4) or its sugar (§8). No
`unset`/`null`/delete operation is introduced.

## 6. Content-line display-label override (`as`)

The content line gains an OPTIONAL `as` attribute (a `String`): a one-off display-label override
for that single line. It does not change state. It has the highest precedence in name resolution
(§7.2). Absent `as`, the resolution falls to state + registry.

```lute
:bianca{code="0075" as="그 목소리"}: (from behind the door) 잠깐만요.
```

## 7. Resolution (normative)

For each content line `:<id>{code, emotion?, variant?, as?}:` (and, for sprite/costume, each `::auto`), a
conforming compiler MUST resolve three outputs.

### 7.1 Sprite assetId
Resolve via the `CH` **asset kind** (plugin-system §6.9): compose its segments with
`characterId = <id>`, `costume =` the character's **current** `scene.cast.<id>.costume` (carried in
stage state), `emotion =` the prior sprite emotion / `neutral`, and `variant = 0` by default. The id
string (e.g. `CH.bianca.waitress.content.0`) is that asset kind's serialization; missing assets
resolve through the kind's declared `fallback`. The current costume MUST apply to **dialogue**
sprites as well as `::auto`, not only where costume is set.

**Only sprite-bearing kinds** (`character` with costumes) resolve a sprite; `player` / `npc` /
spriteless skip §7.1. `emotion` and `variant` validate against the **current character+costume's**
declared `emotions`/`variants` (per-costume domains, §3), not a flat enum; an out-of-domain
`emotion`/`variant` resolves through the `CH` assetKind's declared `fallback` (plugin-system §6.9),
not a raw `neutral`.

**IR contract.** A conforming compiler carries the resolved current costume on the emitted `sprite`
IR record — the `sprite` record and the stage fork/join slots include a `costume` field (compile-IR
0.1.0 addendum). The runtime engine renders from that field and MUST NOT reconstruct the costume by
replaying `scene.cast.<id>.costume` `::set` records.

### 7.2 Display label
Precedence, first match wins:

1. `kind: narrator` → **no speaker name** (neutral narration);
2. `kind: player` → the runtime **`{{userName}}`** (never a registry name or mask);
3. the line's `as` attribute (§6), if present;
4. else if `scene.cast.<id>.sealed == true` → `maskAs` (static label; default `"???"`);
5. else → `registry.name[lang]` — or, for a `kind: npc` without a `name`, the id itself.

### 7.3 voiceKey (path-stable)
`voiceKey = {voiceBank}-{code}` where `voiceBank` is a **static, compile-time constant**:

- `registry.costumeVoiceBank[costume]` **only when** the character's costume is statically
  determinable at that line (from `cast:` or a dominating static assignment);
- otherwise `characterId`.

`voiceKey` MUST NOT be derived from mutable `scene.cast.<id>.costume` in a way that varies by
branch/timeline path — voice joins MUST be path-stable. (Sprites MAY vary per path; they are not
join keys.) If a coded line's `voiceBank` would be path-dependent or ambiguous, it is a static error
(§9); the author pins it or accepts `characterId`. With no `costumeVoiceBank`, `voiceKey =
{characterId}-{code}`.

**Static determinability (conservative).** Whether a costume is statically determinable at a line
reuses the compiler's conservative control-flow-graph join (compile-IR **D9**): the costume is
treated as known only where every path reaching the line joins to the same value. A costume that is
*factually* identical along all paths but that the conservative join cannot prove equal MAY still be
rejected as path-dependent (§9) — this is **expected behavior**, not a defect; the author pins the
costume or accepts `characterId`.

For a `kind: player` speaker, `voiceKey` normalizes to the single **project-defined** player id named
by the scene's `pov` frontmatter (e.g. Idola's `fixer`), so all player lines share a counter.

## 8. Directives (plugin-provided sugar)

Thin, pure desugaring to `::set` (§7.3.4) — no new semantics; the primary authoring surface:

| Directive | Desugars to |
|---|---|
| `::seal{character="bianca"}` | `::set{scene.cast.bianca.sealed = true}` |
| `::reveal{character="bianca"}` | `::set{scene.cast.bianca.sealed = false}` |
| `::wear{character="bianca" costume="casual"}` | `::set{scene.cast.bianca.costume = 'casual'}` |

Attribute schemas: `character` (REQUIRED, `providerRef: character`); `::wear` `costume` (REQUIRED,
`enum` ∈ that character's `registry.costumes`). Raw `::set{scene.cast.*}` remains the low-level
escape hatch; conditional changes use `<match>`/`<branch>` arms containing these.

**Lowering precedence.** `::seal`/`::reveal`/`::wear` declare `lower: {kind: set}` in the plugin
manifest, so a conforming compiler desugars each to `::set` records **at compile time**. This
compile-time `set` lowering takes precedence over the generic non-core-directive → `plugin`-record
passthrough (`../plugin-system/0.0.1.md`): these directives MUST NOT survive to the IR as opaque
`plugin` records.

## 9. Diagnostics (normative lints)

- **Forgot-to-seal:** a character that undergoes a `::reveal` (a `sealed` true→false transition)
  but first appears without being sealed (`sealed: true` in `cast:` or a dominating `::seal`) → a
  warning (possible name spoiler).
- **Path-dependent voiceBank:** a coded line whose `voiceBank` is path-dependent/ambiguous (§7.3)
  → error.
- **Unknown costume / character:** `costume` ∉ `registry.costumes`, or a `character` ∉ registry →
  error (fresh snapshot) or *catalog-stale* diagnostic (stale snapshot).
- **`<match on="scene.cast.<id>.sealed">`** is a `bool` — a finite domain; ordinary exhaustiveness
  applies (no `unset`/null case).

## 10. Migration (non-breaking)

Legacy bard authoring is rewritten by the formatter with deprecation warnings; `code=` values are
**never** changed (voice joins stay stable).

| Legacy | Rewritten | voiceKey |
|---|---|---|
| `:???-bianca{code=…}: …` | `:bianca{code=…}: …` + `cast.bianca.sealed: true` (or `::seal`) | unchanged (`bianca-…`) |
| `:???{code=…}: …` (no id) | requires an explicit id — flagged for author fix | n/a |
| `:ann_child{code=…}: …` | `:ann{code=…}: …` + `cast.ann.costume: child` + `registry.ann.costumeVoiceBank.child = ann_child` | preserved (`ann_child-…`) |
| `:nera_formal{code=…}: …` | `:nera{code=…}: …` + `cast.nera.costume: formal` + `costumeVoiceBank.formal = nera_formal` | preserved (`nera_formal-…`) |

The `-`/`_` id-delimiter conventions and the "post-hyphen id" rule are removed; the content-line
speaker id is identity only.

## 11. Conformance

A conforming document/registry under this proposal:

- references only registry-known `character` ids and, for `costume`, only that character's
  `registry.costumes`;
- keeps the content-line speaker id (`:<id>{…}:`) free of delimiter-encoded label/costume;
- mutates cast state only through `::set`/`::seal`/`::reveal`/`::wear` (enum/bool assignments; no
  `unset`);
- resolves sprite/label/voiceKey per §7, with **path-stable** voiceKey (§7.3).

A conforming consumer resolves these against the one capability snapshot (registry provider + cast
plugin) and reports the §9 diagnostics.

## Appendix A — Worked example

```lute
---
character: bianca
season: 1
episode: 1
profile: story
cast:
  bianca: { costume: waitress, sealed: true, maskAs: "은발의 종업원" }
---

## Shot 1.
::auto{character="bianca" anchor="center" action="fade-in-up"}
:bianca{code="0010" emotion="content"}: 이쪽으로 오세요, 손님.
/* label "은발의 종업원" · sprite CH.bianca.waitress.content.0 · voiceKey bianca-0010 */

## Shot 4.
::reveal{character="bianca"}
:bianca{code="0110" emotion="delighted"}: 저는 비앙카예요.
/* label "비앙카" (registry) · all later lines follow automatically */

::wear{character="bianca" costume="casual"}
:bianca{code="0120" emotion="shy"}: 편한 옷으로 갈아입었어요.
/* sprite CH.bianca.casual.shy.0 · voiceKey bianca-0120 (costume-independent) */
```

## Appendix B — Open items

- **Multi-stage reveal** (`???`→nickname→real): served today by the per-line `as` override; promote
  `sealed : bool` to a staged `enum` only if a scene needs native multi-stage.
- **Plugin id / ownership:** `idola.cast` vs a generic `lute.cast` — decided at packaging; lute core
  stays generic, the VN profile activates it.
- **Collection state** (inventory/sets/maps) is unrelated and out of scope
  (`../scenario-dsl/0.0.1.md` state types remain `number|bool|enum`).

## Appendix C — Errata (2026-07-06, Lute DSL 0.1.0)

This proposal remains at **0.0.1**; the following corrective errata align it with the approved Lute
DSL 0.1.0 revision ([`2026-07-06-lute-dsl-0.1.0-design.md`](../../superpowers/specs/2026-07-06-lute-dsl-0.1.0-design.md)),
which introduces the breaking scenario-dsl 0.1.0 ([`../scenario-dsl/0.1.0.md`](../scenario-dsl/0.1.0.md)).
These items are corrective, not new design.

1. **Player normalization term dropped (§3.1, §7.3).** Player `voiceKey`/lineId normalization is now
   defined solely in terms of the `pov` speaker id; the earlier undefined alias wording is removed.
2. **Lowering precedence (§8).** A plugin directive whose manifest declares `lower: {kind: set}`
   desugars to `::set` records at compile time, taking precedence over the generic
   non-core-directive → `plugin`-record passthrough. `::seal`/`::reveal`/`::wear` declare it.
3. **Monologue is no longer player-only (§3.1).** A non-`player` speaker's `delivery="thought"` is
   that character's own inner monologue, per the DSL 0.1.0 §12 `delivery`→role derivation table
   (narrator forbids a `delivery` attribute).
4. **voiceBank static determinability (§7.3).** Costume determinability reuses the compiler's
   conservative CFG join (compile-IR **D9**); a path-joined costume that is factually identical MAY
   still be conservatively rejected — expected behavior, not a defect.
5. **Costume rendering contract (§7.1).** The IR carries the resolved costume on `sprite` records
   (compile-IR 0.1.0 addendum); the engine renders from that field and does not reconstruct costume
   from `::set` records.
6. **Content-line syntax (throughout).** All `.lute` examples use the DSL 0.1.0 content-line form
   `:speaker{attrs}: text`; the legacy `:line[…]` bracket form is removed from every example.
