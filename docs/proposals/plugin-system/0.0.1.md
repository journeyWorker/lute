# Lute Plugin System — Capability & Manifest Proposal

- **Version:** 0.0.1
- **Status:** Draft (pre-implementation).
- **Scope:** the normative **formats and semantics** of the capability/plugin system — plugin
  packages, the YAML **manifest schemas**, capability **resolution**, the resolved **capability
  snapshot**, and the data↔code boundary that bounds what a plugin MAY declare.
- **Out of scope:** the authoring **language** ([`../scenario-dsl/0.0.1.md`](../scenario-dsl/0.0.1.md));
  the compiler/AST/validation/LSP **architecture** ([`../../architecture.md`](../../architecture.md));
  the human-facing **overview** ([`../../plugin-system.md`](../../plugin-system.md), which is the
  rationale/author guide — this document is the normative source of truth).
- **Targets:** Lute Scenario DSL `^0.0.1`.
- **Reference example:** [`../../examples/date-minigame.lute`](../../examples/date-minigame.lute).

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHOULD**, **SHOULD NOT**, **MAY** are to be
interpreted as in RFC 2119.

---

## 1. Abstract

The Lute language has a **fixed grammar** (`../scenario-dsl/0.0.1.md`). A plugin extends only the
*vocabulary and capability surfaces* the grammar refers to — directive names, attributes, enum
domains, state shapes, providers, bridge signatures, definitions, and diagnostics — **never** the
grammar and **never** behavior. Every declaration in a plugin is **data**, validated against the
schemas in this document and resolved, together with a selected profile, into a single
deterministic **capability snapshot** that the checker, LSP, and compiler all consume. A plugin
that needs new control flow, a new lowering algorithm, or any other *behavior* is not expressible
as data and is out of scope for a plugin (§3.2, §12).

## 2. Status & versioning

This proposal is `0.0.1`, versioned `MAJOR.MINOR.PATCH` independently of the language. Pre-`1.0.0`
minor versions MAY make breaking changes.

- A `plugin.yaml` declares its own `version` (the plugin's semver) and a `depends` range against
  other plugins (§5).
- The **resolved capability snapshot** carries a `version` field, the **`capabilityVersion`** — a
  content hash over the resolved set of plugin ids+versions, their options, and the active profile
  (§13). Parser, checker, LSP, and compiler artifacts MUST be stamped with the `capabilityVersion`
  they were produced against and MUST refuse to mix mismatched stamps.

## 3. Design invariants (normative)

1. **Data, not grammar.** A plugin MUST NOT introduce grammar productions, new bracket forms, new
   block kinds, or any change to line classification (`../scenario-dsl/0.0.1.md` §4.3). A plugin
   adds names and schemas that fill positions the grammar already defines (a `Directive` ident, an
   attribute value domain, a `<match on>` subject path, a `defs` entry).
2. **Data, not behavior.** A capability is registrable as plugin data **iff all** hold: (1) fixed
   syntax; (2) validation is local to its attributes plus catalog lookup; (3) lowering is a finite
   attributes→records mapping; (4) it introduces no new control flow; (5) it requires no
   cross-sibling or global reasoning beyond declared resource conflicts; (6) it changes no AST
   shape; (7) it adds no ordering-sensitive interpretation beyond the existing timeline/`wait`
   model. If **any** condition is false, the capability is **code** — a change to the compiler core
   — and MUST NOT be expressed as a plugin (§12).
3. **Deterministic resolution.** Given the same installed plugins, selected profile, and scene
   frontmatter, capability resolution (§11) MUST produce a byte-identical capability snapshot and
   `capabilityVersion`. Resolution MUST NOT depend on file-system iteration order, wall-clock, or
   network.
4. **Snapshot-first.** Validation and compilation MUST consult pinned **provider snapshots**
   (§10), never a live or remote catalog. Missing required snapshot data is a build error, never a
   network fetch.

## 4. Plugin package

A plugin is a directory whose entry is a single `plugin.yaml` (§5). The entry's `exports` map
(§5) names which sub-directories the loader reads; the loader MUST ignore any directory not listed
in `exports`.

```text
plugins/<id>/
  plugin.yaml          # the manifest entry (REQUIRED)
  directives/*.yaml     # directive declarations            (§6.1)
  state/
    shapes.yaml         # reusable state shapes              (§6.2)
    templates.yaml      # structured path templates          (§6.3)
  providers/*.yaml      # id registries (catalogs)           (§6.4)
  bridge/*.yaml         # bridge capabilities                (§6.5)
  defs/*.yaml           # shared typed-CEL definitions       (§6.6)
  docs/*.md             # human docs (non-normative)         (§6.7)
  frontmatter/*.yaml    # plugin-owned meta keys                (§6.8)
  assetkinds/*.yaml     # asset id templates                    (§6.9)
```

Loader rules (normative):

- The loader MUST read only the directories named in `exports`, MUST sort files within a directory
  by byte-wise path order before merging, and MUST reject the package if two declarations across
  the package share an id within the same kind (directive name, shape name, provider name, def
  name, bridge service+operation).
- Merge MUST be order-independent: if the resolved result depends on file-sort order, the package
  is non-conforming (a duplicate-id or conflicting-declaration error MUST be raised instead).

## 5. `plugin.yaml` (manifest entry)

```yaml
id: idola.minigame          # REQUIRED — reverse-dotted plugin id (§14)
version: 0.1.0              # REQUIRED — semver of this plugin
kind: capability           # REQUIRED — only "capability" is defined in 0.0.1
depends:                   # OPTIONAL — other plugins this one requires
  - id: lute.core
    range: "^0.0.1"
exports:                   # REQUIRED — which sub-directories the loader reads
  directives: directives/
  state: state/
  providers: providers/
  bridge: bridge/
  defs: defs/
  docs: docs/
  frontmatter: frontmatter/
  assetkinds: assetkinds/
options:                   # OPTIONAL — typed activation options (§5.1)
  - name: resultScope
    type: { enum: [scene, run] }
    default: scene
  - name: allowedKinds
    type: { list: { enum: [rhythm, puzzle, timing] } }
    default: [rhythm, puzzle, timing]
```

| Field | Req. | Type | Rule |
|---|---|---|---|
| `id` | MUST | reverse-dotted ident (§14) | globally unique; the only handle consumers reference |
| `version` | MUST | semver | the plugin's own version |
| `kind` | MUST | `capability` | the only kind in 0.0.1; unknown kinds are an error |
| `depends[]` | MAY | `{id, range}` | `range` is a semver range; the dependency MUST resolve in the closure (§11) or it is an error |
| `exports` | MUST | map<exportKind, dirpath> | keys MUST be from the closed set in §4; a listed dir that is absent is an error |
| `options[]` | MAY | `Option[]` (§5.1) | activation options |

### 5.1 Options

```
Option ::= { name: Ident, type: Type (§7), default?: <literal of type> }
```

An option `name` MUST be unique within the plugin. A `default` (if present) MUST be a literal
valid for `type`. Option **values** are supplied at activation (§11) and become part of the
capability snapshot — and therefore part of `capabilityVersion` (§13). Activation MUST reject an
option value that is not valid for its declared `type`, and MUST reject an unknown option name.

## 6. Manifest files (exports)

Each export kind has a normative schema. All examples are YAML.

### 6.1 `directives/*.yaml` — directive declarations

A directive declaration is the schema for one `::name` staging/bridge directive (the language
defines `Directive ::= "::" Ident Attrs?`; this file gives that ident meaning). See §8 for the
full directive schema.

### 6.2 `state/shapes.yaml` — state shapes

```yaml
stateShapes:
  - name: minigameResult
    fields:
      - { name: score,    type: number, default: 0 }
      - { name: rank,     type: { enum: [fail, bronze, silver, gold] }, default: fail }
      - { name: cleared,  type: bool,   default: false }
      - { name: attempts, type: number, default: 0 }
```

A `stateShape` is a reusable typed record. `name` MUST be unique. Each field has a `type` (§7) and
an OPTIONAL `default`. A shape is referenced from a directive's `state.declares[].shape` (§8). A
field with a `default` is **always-assigned** when its slot is declared (consistent with language
§9.3/§9.4); a field without a default is **maybe-unset** until written.

### 6.3 `state/templates.yaml` — structured path templates

Machine-readable manifests MUST express variable state paths as **structured segments**, never as
`$name` string interpolation. Documentation MAY use `$name` shorthand (e.g.
`scene.minigame.$resultKey.rank`); a manifest MUST instead use a segment list (§7.4).

```yaml
stateTemplates:
  - name: minigameSlot
    scope: scene
    path: [minigame, { fromAttr: { name: resultKey, slotType: localId } }]
    shape: minigameResult
```

### 6.4 `providers/*.yaml` — id registries

A provider names an **open** registry of ids resolved from a catalog (e.g. character ids, asset
ids, minigame ids). The file declares the provider; the **values** come from a pinned snapshot
(§10), not from this file.

```yaml
providers:
  - name: minigameId
    idShape: "Ident"          # OPTIONAL — a validation shape for ids
    snapshot: minigame-id     # the provider snapshot key this resolves against (§10)
```

`name` MUST be unique and is the target of a `providerRef` attribute type (§7).

### 6.5 `bridge/*.yaml` — bridge capabilities

```yaml
bridgeCapabilities:
  - service: minigame
    operation: play
    replay: recorded          # recorded | deterministic | none
    result:                   # OPTIONAL — declared result fields a directive may bind via fromBridgeResult (§8)
      - { name: score,   type: number }
      - { name: rank,    type: { enum: [fail, bronze, silver, gold] } }
      - { name: cleared, type: bool }
```

A bridge capability is a typed runtime call the engine owns. `service`+`operation` MUST be unique.
A directive invokes it via `bridge: { service, operation }` (§8). The DSL emits only data; the
engine executes the bridge; story control-flow observes only the **declared state** a directive
writes (§9), never raw bridge output.

### 6.6 `defs/*.yaml` — shared definitions

`defs` are named, typed CEL values, exactly as in language §8.1 (`@ref` macro expansion). A plugin
MAY export shared `defs`; def `name`s MUST be unique across the resolved closure (duplicate def
names are an error — no silent shadow, consistent with language §9.2).

### 6.7 `docs/*.md`

Human documentation surfaced by the LSP (hover). Non-normative.

### 6.8 `frontmatter/*.yaml` — plugin-owned meta keys

A plugin MAY declare top-level frontmatter (`meta`) keys it owns, so authors configure the plugin
declaratively in a scene's `---` block (language §6.1). Each declared key has a `key` name and a
`schema` (using the §7 type system); the checker validates the scene's block against it. A meta key
owned by no active plugin is a static error, and two active plugins MUST NOT declare the same key.
Frontmatter values are **static** configuration — they MAY seed state defaults (via a state
template, §6.3), but are not themselves mutable state.

```yaml
frontmatter:
  - key: cast                       # the top-level meta key this plugin owns
    schema: { map: { key: string, value: { record: [ { name: costume, type: string }, { name: sealed, type: bool, default: false }, { name: maskAs, type: string, default: "???" } ] } } }   # key = characterId, validated by character-cast §4
```

Declared keys + schemas are part of the capability snapshot (§13) and the `capabilityVersion` hash.

### 6.9 `assetkinds/*.yaml` — asset id templates

An **asset id is a structured, ordered list of segments** — the dotted string
`CH.bianca.waitress.content.0` is only its serialization (same principle as structured state paths,
§6.3/§7.4). An `assetKind` declares that segment schema, so composition (segments → id),
decomposition (id → segments), and per-segment validation/completion/hover/go-to-def all derive from
one datum instead of hardcoded `compose`/`parse` code.

```yaml
assetKinds:
  - kind: CH                         # namespace / prefix
    sep: "."                         # serialization separator (default ".")
    segments:                        # ordered; each is a const or a typed slot (§7)
      - { name: prefix,      const: CH }
      - { name: characterId, type: { providerRef: character } }
      - { name: costume,     type: string }     # per-character domain validated by the owning capability
      - { name: emotion,     type: { enum: [...] } }
      - { name: variant,     type: number }
    fallback: [emotionGroup, neutral, variant0]  # ordered named hooks (below)
    persistence: scene               # e.g. bg=scene-persistent, sfx=fire-and-forget
```

- **Composition / decomposition are data:** join/split by `sep`, type-check each segment (the §3.2
  data-vs-code conditions hold). The compiler composes; the LSP parses an id back to segments and
  validates/completes/hovers each one against its declared type + the pinned provider snapshot (§10).
- **`fallback` is an ordered list of named hooks** from a closed core registry (like lowering hooks,
  §8.2), for missing assets (e.g. emotion-group rep → neutral → variant 0). Each fallback hook MUST
  declare its **input/output segment schema** and MUST be **deterministic over the pinned asset
  snapshot**, so the LSP can generate candidate ids and diagnostics offline. An opaque fallback hook
  (no declared schema / non-deterministic) is non-conforming — it would merely move the magic from
  id strings into an uninspectable resolver.
- A capability that owns an asset domain declares its kinds here; a kind referencing another
  capability's provider (e.g. `CH` → the `character` provider) composes across plugins.
- **Two resolve modes.** A `resolve: compose` kind (default) builds the id by joining `segments`
  (CH). A `resolve: query` kind instead **matches a provider catalog by declared fields** — authored
  attrs map to catalog fields (optionally through an `aliases` map), returning an existing entry's
  id; a missing match relaxes through the kind's ordered `fallback` hooks. Query mode covers Bard's
  `mood → bgm`, `spaceId/view/timeOfDay/variation → bg` (location alias + prefer-default), and fuzzy
  `name → sfx`. Query-mode kinds MAY still declare `segments` so authored ids decompose/validate.

```yaml
assetKinds:
  - kind: BGM
    resolve: query
    provider: bgm
    match: [ { attr: mood, field: mood } ]          # authored attr → catalog field
  - kind: BG
    resolve: query
    provider: backgrounds
    aliases: { location: locationAlias }            # attr → canonical id via an alias provider
    match: [ { attr: location, field: spaceId, via: locationAlias },
             { attr: time, field: timeOfDay }, { attr: view, field: view },
             { attr: variation, field: variation } ]
    fallback: [ dropVariation, areaKind, preferAfternoon, anyView ]   # ordered named hooks
```

**Resolution precedence (normative).** For any asset-bearing directive:

1. an **explicit authored `assetId`** (author/enricher-provided) **wins**: it MUST validate and
   **decompose** against the assetKind (segment shape + provider-existence) and **bypasses**
   compose/query inference. This is Bard's primary escape hatch for `bg`/`cut`/`sfx`/`video`.
2. else **compose** (segment kinds) or **query** (query kinds) runs;
3. else the kind's ordered **`fallback`** hooks run;
4. a resolved id MAY carry a **canonical redirect** (`canonicalAssetId`: slot id → physical file) —
   a data-declared redirect table, not code.

An unresolved asset is a diagnostic; an authored `PLACEHOLDER_*` sentinel resolves to itself and is
surfaced as a warning (Bard's "fill later" convention).

## 7. The manifest type system

Every typed position (option, attribute, shape field, bridge result, asset-kind segment, frontmatter field) uses one
`Type`:

```
Type ::= "bool" | "number" | "string"
       | { enum: [<literal>, ...] }
       | { list: Type }
       | { record: [ Field, ... ] }             (* object with named typed fields *)
       | { map: { key: Type, value: Type } }    (* dictionary *)
       | { enumFromOption: <optionName> }      (* attribute types only *)
       | { providerRef: <providerName> }       (* an authored/static id; any typed position *)
       | { slotId: { namespace: <statePath> } }(* attribute types only *)
Field ::= { name: Ident, type: Type, default?: <literal>, required?: bool }
```

- `enumFromOption` resolves its domain from a plugin **option** value at activation; the option
  MUST be a `list` of scalars or an `enum`.
- `providerRef` validates a value (attribute, asset-kind segment, or frontmatter field) as an id
  present in the named provider's snapshot (§10). An id absent from a *fresh* snapshot is an error;
  absent from a *stale* snapshot is a "catalog stale" diagnostic, not a hard error (§10).
- `slotId` declares/refers to a state-slot id in `namespace`; see §9.
- A **state shape field** MAY additionally use `shape: <stateShapeName>` (a nested shape
  reference); attribute types MAY NOT.
- `record` is an inline object with named typed `Field`s (an unnamed `stateShape`); `map` is a
  homogeneous dictionary. Both compose (a `map` of `record`s, a `list` of `record`s) and back
  richer schemas such as `frontmatter` keys (§6.8) and nested config.

### 7.4 Structured paths

A state `path` is an ordered list whose elements are each:

```
PathSegment ::= <literal string>                       (* a fixed key *)
              | { fromAttr: { name: <attrName>, slotType?: <slotKind> } }
              | <fieldName>                             (* a shape field, trailing *)
```

A `fromAttr` segment binds to the runtime value of the directive attribute `name`. Manifests MUST
NOT use `$`-interpolated path strings (§6.3).

## 8. Directive declaration (normative)

```yaml
directives:
  - name: minigame            # REQUIRED — the ::name ident; unique in plugin, namespaced in snapshot
    layer: bridge             # OPTIONAL — categorization (e.g. staging | bridge)
    attrs:                    # REQUIRED — attribute schema
      - { name: kind,      required: true, type: { enumFromOption: allowedKinds } }
      - { name: id,        required: true, type: { providerRef: minigameId } }
      - { name: resultKey, required: true, type: { slotId: { namespace: scene.minigame } } }
      - { name: wait,      type: bool, default: true }
    semantics: [ "writes.sceneState", "bridgeCall" ]   # OPTIONAL — closed-vocabulary flags (§8.1)
    state:                    # OPTIONAL — slots this directive declares
      declares:
        - scope: scene
          path: [minigame, { fromAttr: { name: resultKey, slotType: localId } }]
          shape: minigameResult
    effects:                  # OPTIONAL — declared state writes
      writes:
        - { scope: scene, path: [minigame, { fromAttr: { name: resultKey } }, rank],  value: { fromBridgeResult: rank } }
        - { scope: scene, path: [minigame, { fromAttr: { name: resultKey } }, score], value: { fromBridgeResult: score } }
        - { scope: scene, path: [minigame, { fromAttr: { name: resultKey } }, attempts], value: { op: increment, by: 1 } }
    bridge: { service: minigame, operation: play }     # OPTIONAL — bridge binding (§6.5)
    lower: { kind: builtin, name: bridgeMinigame }     # REQUIRED — lowering (§8.2)
```

| Field | Req. | Rule |
|---|---|---|
| `name` | MUST | the `::name` ident; unique within the plugin; namespaced by plugin id in the snapshot |
| `layer` | MAY | informative categorization |
| `attrs[]` | MUST | each `{name, required?, type (§7), default?}`; attr `name` unique within the directive; a missing `required` attr at a use site is a static error; a `default` MUST be valid for `type` |
| `semantics[]` | MAY | flags from the **closed vocabulary** (§8.1); unknown flags are an error |
| `state.declares[]` | MAY | `{scope, path (§7.4), shape}`; declares a slot conforming to language §9; `scope` ∈ `scene\|run\|user` (a plugin MUST NOT declare into `app`, language §9.5) |
| `effects.writes[]` | MAY | `{scope, path, value}`; `value` ∈ `{fromBridgeResult: <field>}` \| `{op: increment, by: N}` \| `<literal>`; every written path MUST be declared (here or in the imported schema) |
| `bridge` | MAY | `{service, operation}` referencing a declared bridge capability (§6.5) |
| `lower` | MUST | §8.2 |

### 8.1 Semantics flags (closed vocabulary)

`semantics` values are drawn from a closed set the compiler core understands, e.g.
`writes.sceneState`, `writes.characterState`, `reads.onStage`, `mayExitCharacter`, `usesAnchor`,
`isExit`, `isStateful`, `mutatesScene`, `requiresAnchor`, `cancelsPrevious`, `bridgeCall`. The set
is owned by the core; a plugin MUST NOT invent flag names (an unknown flag is an error). Flags
declare *which* core behavior a directive participates in; the core owns *how* (§12).

### 8.2 Lowering

```
lower ::= { record: <recordType>, fields: { ... } }   (* declarative: one fixed record *)
        | { kind: "builtin", name: <hookName> }        (* a narrow, named core hook *)
```

- A directive whose lowering is a finite attributes→single-record mapping MUST use the declarative
  `record` form (data).
- A directive whose lowering reads prior commands, pairs show/hide, allocates timeline tracks,
  emits a variable number of records, does anchor/lifecycle bookkeeping, or inspects siblings MUST
  reference a **named builtin hook** from the core's closed registry. `name` MUST resolve to a
  registered hook. There is **no** inline code in a manifest. **Adding a hook is a core code
  change, not plugin registration** (§12).

## 9. State, slots & bridge results

- A directive that declares a slot (`state.declares[]`) MUST do so with a `slotId` attribute (§7)
  whose `namespace` matches the declared `scope`+`path` prefix. The slot's fields come from its
  `shape` (§6.2).
- A declared slot conforms to language §9: every read/write of its paths is subject to
  definite-assignment (§9.4 of the language). Shape fields with `default` are always-assigned;
  fields without are maybe-unset until written.
- **Bridge result timing.** A directive with `bridge` and `wait: true` (the schema default for a
  blocking bridge) produces writes that **dominate** an immediately following read. With
  `wait="false"` the writes do **not** dominate: because result slots are typically shape-defaulted
  (e.g. `rank: fail`), a following `<match>` is **not** `E-MAYBE-UNSET` — it reads the **default**,
  not the produced outcome. The checker MUST report this as a *result-read-before-produced*
  (stale-default) diagnostic, distinct from `E-MAYBE-UNSET`.

## 10. Providers & the snapshot artifact

Compiler/checker correctness MUST NOT depend on a live catalog. A provider resolves against a
pinned snapshot artifact:

```
ProviderSnapshot ::= {
  manifestVersion,   # the capabilityVersion this snapshot was built for
  providerVersion,   # the snapshot's own version/id
  generatedAt,       # timestamp (informative)
  sourceRefs,        # provenance of the underlying catalogs
  entries            # the resolved ids, keyed by provider name
}
```

```ts
interface Provider<T> {
  snapshot(): ProviderSnapshot<T>;          // sync, deterministic — what everyone validates against
  refresh?(): Promise<ProviderSnapshot<T>>; // async; editor/build-prep only
}
```

- The **compiler** MUST consume a pinned snapshot and MUST fail if required data is missing; it
  MUST NOT block on the network. An explicit `catalog refresh` precedes a build.
- The **LSP** MUST load the same snapshot immediately, MAY refresh asynchronously, MUST surface
  snapshot age, and when offline MUST keep the stale snapshot and emit a *catalog-stale*
  diagnostic rather than false *unknown-id* errors.
- The **parser** MUST NOT call providers; only the checker consults snapshots.

## 11. Profiles & activation (normative)

A **profile** is a root-level capability selector (language §6.1.1). The project declares a profile
graph and a `defaultProfile`:

```yaml
profiles:
  global:                       # reserved name; inherited by every profile
    plugins: { lute.core: true }
    lint: { unknownDirective: error }
  story:
    plugins: { idola.vn: true }
  date:
    extends: story
    plugins: { idola.date: { phoneSurface: enabled } }
  date-minigame:
    extends: date
    plugins:
      idola.minigame: { resultScope: scene, allowedKinds: [rhythm, timing] }

defaultProfile: story
```

- `plugins` is a **map** from plugin id to a typed option object. Presence of a key **activates**
  that plugin; `true` normalizes to an empty option object (defaults). There is **no** `plugins.use`
  list.
- The reserved profile `global` MUST be applied before any other profile.
- `extends` names a single parent profile; the `extends` chain MUST be acyclic (a cycle is an
  error).
- A scene selects a profile via frontmatter `profile:` (absent ⇒ `defaultProfile`) and MAY add
  scene-local `plugins:` (language §6.1). Activation is **additive**; 0.0.1 defines **no**
  scene-local deactivation. To exclude a plugin, do not inherit a profile that activates it.

### 11.1 Resolution order (deterministic)

The capability snapshot is resolved by applying, in this exact order:

1. language-required core plugin(s) (`lute.core`);
2. `profiles.global`;
3. the selected profile's `extends` chain, **parent first**;
4. the selected profile itself;
5. scene-local `plugins`;
6. the dependency closure (`depends`, §5) of everything activated above.

### 11.2 Merge semantics

When the same plugin's options are set at multiple layers (§11.1), later layers override earlier:

- **scalar** option values override;
- **map** option values deep-merge;
- **list** option values **replace** by default, unless the option's schema declares a different
  merge strategy.

The result is exactly one option object per active plugin, and exactly one capability snapshot
(§13). A reference to a directive/attr/id from an installed-but-**inactive** plugin MUST be a
diagnostic with fix-its ("change profile" / "activate plugin"), never silently accepted syntax.

## 12. The data↔code boundary (normative)

This restates invariant §3.2 as a conformance gate. The seductive cases — "have her leave
naturally", "keep the same pose unless mood changes", "hit this SFX exactly as the line appears",
"hide whoever is no longer speaking" — *feel* like vocabulary but are compiler **behavior**; they
MUST NOT be encoded as plugin data. They are served, if at all, by a **named builtin lowering hook**
(§8.2) added to the core.

**Golden-test gate.** Every directive declaration MUST be accompanied by a golden test
(DSL input → CheckedIr → command records → diagnostics). **If a directive cannot get a clean golden
test, it is behavior, not data**, and does not belong in a plugin.

## 13. The capability snapshot (resolved artifact)

Resolution (§11) produces one immutable snapshot consumed identically by checker, LSP, and
compiler:

```ts
capabilitySnapshot = {
  version,            // capabilityVersion — content hash over the fields below
  plugins,            // resolved plugin ids + versions + option objects
  profiles,           // the profile graph; "global" reserved
  enums,              // emotion / mood / volume / anchor / vfxType / musicAction / …
  namespaces,         // id shapes + ownership: character.id, bg.id, music.id, action.id
  providers,          // provider declarations + bound snapshot keys (§10)
  stateShapes,        // §6.2
  stateTemplates,     // §6.3
  assetKinds,         // §6.9 — asset id segment templates + fallback; persistence, preview
  directives,         // §8, namespaced by plugin id
  bridgeCapabilities, // §6.5
  blockKinds,         // branch/choice/match/when/otherwise/timeline/track (structural; language-owned)
  layers,             // visual/audio lanes: exclusivity, z-order, persistence
  transitions,        // reusable easing/fade/timing schemas
  actionSemantics,    // isExit / isStateful / mutatesScene / requiresAnchor / cancelsPrevious
  diagnostics,        // per-attr messages, deprecations, aliases, fixits
  frontmatter,        // §6.8 — plugin-owned meta key names + schemas
}
```

`capabilityVersion` MUST be a deterministic hash over the resolved plugin ids+versions, their
option objects, the active profile, and the bound provider snapshot versions. Every generated
artifact (validation tables, completion data, tree-sitter grammar, lowering tables) MUST be stamped
with the `capabilityVersion` it targets; a consumer MUST refuse mismatched stamps.

## 14. Reserved names & collisions

- A plugin `id` is a reverse-dotted identifier (`vendor.pack`, e.g. `lute.core`, `idola.minigame`).
  `lute.*` is reserved for the language's own core capability.
- A directive `name`, tag, or block kind MUST NOT collide with a language-reserved term
  (`../scenario-dsl/0.0.1.md` §10: `cut`, `scene`, the §7.5 timing attrs, etc.).
- The timing attribute keys `duration`, `delay`, `wait` (and the timeline-only `at`) are reserved
  across all directives (language §7.5); a plugin MUST NOT redefine them.

## 15. Conformance

A **conforming plugin**:

- has a valid `plugin.yaml` (§5) and exports only declared directories (§4);
- declares no grammar and no behavior (§3.1, §3.2, §12);
- gives every directive `attrs` + a `lower` (§8) and a passing golden test (§12);
- uses structured paths, not `$`-interpolation (§6.3/§7.4);
- introduces no duplicate ids within a kind and no merge-order dependence (§4);
- resolves, under §11, into a capability snapshot without unresolved `depends`, cycles, or
  inactive-plugin references.

A **conforming consumer** (checker, LSP, compiler) validates a document against exactly the one
capability snapshot resolved in §11 and refuses artifacts whose `capabilityVersion` does not match.

## Appendix A — Worked manifest (`idola.minigame`)

The reference example [`../../examples/date-minigame.lute`](../../examples/date-minigame.lute)
activates this plugin via the `date-minigame` profile.

```yaml
# plugin.yaml
id: idola.minigame
version: 0.1.0
kind: capability
depends: [ { id: lute.core, range: "^0.0.1" } ]
exports: { directives: directives/, state: state/, providers: providers/, bridge: bridge/, defs: defs/, docs: docs/ }
options:
  - { name: resultScope, type: { enum: [scene, run] }, default: scene }
  - { name: allowedKinds, type: { list: { enum: [rhythm, puzzle, timing] } }, default: [rhythm, puzzle, timing] }
```

```yaml
# state/shapes.yaml
stateShapes:
  - name: minigameResult
    fields:
      - { name: score,    type: number, default: 0 }
      - { name: rank,     type: { enum: [fail, bronze, silver, gold] }, default: fail }
      - { name: cleared,  type: bool,   default: false }
      - { name: attempts, type: number, default: 0 }
```

```yaml
# directives/minigame.yaml
directives:
  - name: minigame
    layer: bridge
    attrs:
      - { name: kind,      required: true, type: { enumFromOption: allowedKinds } }
      - { name: id,        required: true, type: { providerRef: minigameId } }
      - { name: resultKey, required: true, type: { slotId: { namespace: scene.minigame } } }
      - { name: wait,      type: bool, default: true }
    semantics: [ "writes.sceneState", "bridgeCall" ]
    state:
      declares:
        - { scope: scene, path: [minigame, { fromAttr: { name: resultKey, slotType: localId } }], shape: minigameResult }
    effects:
      writes:
        - { scope: scene, path: [minigame, { fromAttr: { name: resultKey } }, score],    value: { fromBridgeResult: score } }
        - { scope: scene, path: [minigame, { fromAttr: { name: resultKey } }, rank],     value: { fromBridgeResult: rank } }
        - { scope: scene, path: [minigame, { fromAttr: { name: resultKey } }, cleared],  value: { fromBridgeResult: cleared } }
        - { scope: scene, path: [minigame, { fromAttr: { name: resultKey } }, attempts], value: { op: increment, by: 1 } }
    bridge: { service: minigame, operation: play }
    lower: { kind: builtin, name: bridgeMinigame }
```

The scene then writes nothing by hand — it reads the declared slot:

```lute
::minigame{kind="rhythm" id="bianca_service_01" resultKey="service01" wait="true"}

<match on="scene.minigame.service01.rank">
  <when test="$ == 'gold'"> ... </when>
  <otherwise> ... </otherwise>
</match>
```
