# Lute Scenario DSL — Language Proposal

- **Version:** 0.0.1
- **Status:** Draft (pre-implementation; revised after a grammar↔example consistency audit)
- **Scope:** the authoring **language** only — lexical structure, grammar, and normative
  semantics of a `.lute` scenario document.
- **Out of scope:** implementation architecture (AST shapes, compiler passes, auto-injection,
  validation core / LSP, capability manifest, runtime engine). Those are specified separately in
  [`../../scenario-dsl-spec.md`](../../scenario-dsl-spec.md).
- **Reference example:** [`../../examples/bianca-s01ep02.lute`](../../examples/bianca-s01ep02.lute).

The key words **MUST**, **MUST NOT**, **SHOULD**, **MAY** are to be interpreted as in RFC 2119.

---

## 1. Abstract

The Lute Scenario DSL is a line-oriented language for authoring visual-novel episodes. A document
interleaves three syntactic layers — **content** (prose + dialogue), **staging** (single-line
directives), and **logic** (nesting blocks: branching, matching, and a multi-track timeline) —
and embeds [CEL](https://cel.dev) expressions for conditions. The language is **total**: every
construct MUST reduce to a finite, ordered sequence of command records plus CEL expression
strings. It is deliberately **not** Turing-complete.

## 2. Status & versioning

This is `0.0.1`, the first draft, prior to implementation. Versions are `MAJOR.MINOR.PATCH`.
Pre-`1.0.0` minor versions MAY make breaking changes. A document MAY declare the language version
it targets via `meta.lang` (§6.1); absent that, an implementation assumes the latest it supports.

## 3. Design invariants (normative)

1. **Total, not Turing-complete.** No construct introduces unbounded iteration, recursion, or
   user-defined functions with bodies. All conditions are CEL, which is terminating and
   side-effect-free.
2. **Reduces to data.** Every document MUST be reducible at build time to a flat, ordered
   command-record sequence + CEL strings. A construct that cannot be so reduced is **not** part of
   this language.
3. **Layer legibility.** The three layers are distinguished by leading syntax so a reader can
   classify any line at a glance (§4.3).

## 4. Lexical structure

### 4.1 Lines, whitespace, blank lines

A document is a sequence of lines separated by U+000A. Most productions are **line-anchored**:
they match from the first non-whitespace character of a line to its end. Leading whitespace on a
line is **insignificant** (the language is not indentation-scoped). A **blank line** (empty or
whitespace-only) is **trivia**: it MUST be ignored except as a separator between prose paragraphs.

### 4.2 Comments are trivia

A comment starts with `/*` and ends at the next `*/`. Comments are stripped as trivia before body
line classification (§4.3); they are **not** nodes. They MAY appear standalone, inline, trailing,
or multi-line anywhere in the body.

Comments are not recognized inside quoted `String` / `CelString` values (§4.4). They do apply
inside content text before `Text` is formed: after comment stripping, the remaining `:line` /
`:mono` / prose body is opaque to end of line (§4.4). Comments do not nest; an unterminated
comment is a parse error. Frontmatter uses YAML's own comment rules, not this body comment syntax.

### 4.3 Line classification (normative precedence)

The leading `---` **frontmatter** envelope (§6.1) is recognized first and is not a body line; it is
peeled (delimiter-to-delimiter) before body comment stripping and classification begin. After
comments and leading whitespace are stripped, a non-blank **body** line is classified by the
**first matching** rule in this order (most specific first; **prose is the fallback**):

1. `## ` — shot heading (§6.3)
2. `# ` — document title (§6.2)
3. `::set{` — assignment directive (§7.3.4)
4. `::` *ident* — staging directive (§7.2)
5. `:line[` — dialogue (§7.1)
6. `:mono:` — monologue (§7.1)
7. `<` *ident* / `</` — logic / timeline block open or close (§7.3, §7.4)
8. otherwise — **prose** (§7.1)

This order is normative: e.g. `::set{` MUST be tried before the generic `::`-directive rule. (There
is **no** `:::` construct in the body — `meta` is frontmatter (§6.1) and the former `:::route` is
the `<branch>`/`<match>` nesting (§7.3); `:::` is not part of the grammar.)

### 4.4 Terminals

```
Ident     ::= [A-Za-z] [A-Za-z0-9_-]*
Integer   ::= [0-9]+
Speaker   ::= Ident                         (* a character id *)
String     ::= '"' ( <any char except '"' or '\'> | '\' <any char> )* '"'
Path      ::= ("scene" | "run" | "user" | "app") ("." Ident)+   (* a declared state path, §9 *)
Text      ::= <the rest of the line, verbatim, to EOL>   (* opaque; never re-scanned *)
Ref       ::= "@" Ident ( "(" CelArgs ")" )?
CelString ::= '"' <a CEL expression> '"'
```

- **`String`** is double-quoted with backslash escaping; it MUST NOT contain a raw newline.
- **`CelString`** is a double-quoted **CEL expression**. CEL's own string literals inside it use
  single quotes (`'blunt'`) or escaped double quotes; thus `test="$ == 'blunt'"` is well-formed.
  A `Ref` (`@name` / `@name(args)`) MAY appear inside a `CelString` and is expanded to inline CEL
  before evaluation (§8.1). The distinguished token `$` MAY appear only inside a `<match>` (§8.2).
- **`Text`** is opaque to end of line: once a `:line`/`:mono`/prose body begins, the remainder of
  the line is literal (so `(`, `?`, `<`, `:`, quotes are not special there). Empty `Text` is
  permitted (e.g. `:line[fixer]{code="0010"}: ...`).
- **Quoted-string boundaries are respected before structural scanning.** A lexer MUST tokenize
  `String` / `CelString` quotes before classifying lines or scanning tag/attr structure, so
  structural-looking characters inside a quoted value (`<`, `{`, `}`, `:` — e.g. `test="$ < 3"` or
  a CEL map literal) are content, not delimiters.

### 4.5 Attributes

```
Attrs ::= "{" ( Attr ( WS Attr )* )? "}"
Attr  ::= Ident "=" String | Ident "=" Ref | Ident      (* bare Ident ⇒ boolean true *)
```

Attribute order is **not significant**. All attribute **values** are a `String` or a `Ref`; there
is **no** `{ expression }` interpolation (preserving invariant §3.1). An implementation coerces a
`String` value to the attribute's declared type.

## 5. Layers (overview)

| Layer | Syntax | Members |
|---|---|---|
| **Content** | `:line[name]{…}: text`, `:mono: text`, bare prose | §7.1 |
| **Staging** (leaf) | `::name{…}` | §7.2, Appendix A |
| **Logic / timeline** (nesting) | `<tag>…</tag>` | §7.3, §7.4 |

**Bracket rule (normative):** a construct with children MUST use the `<tag>…</tag>` form (and MUST
be explicitly closed); a single-line construct MUST use a `::`-directive or `:`-content form.

## 6. Document structure

```
Document  ::= Meta DocItem*
DocItem   ::= Title | ShotBlock
```

Trivia (blank lines, comments) MAY appear between any of these.

### 6.1 `meta` (frontmatter)

```
Meta      ::= "---" NEWLINE YamlBody "---" NEWLINE
```

`Meta` is **YAML frontmatter** and MUST be the document's first construct (it precedes all body
trivia and nodes). `YamlBody` is an **opaque YAML block** (the implementation parses it as YAML).
This is the repo-wide `---` frontmatter convention; there is no `:::`-fenced meta form. `meta` MUST
define `character`, `season`, `episode`; it MAY define `mode`, `pov`, `title`, `lang`, `profile`,
`plugins`, `uses`, and inline `state` (§9) and `defs` (§8.1) blocks.

- `lang` here is the **DSL language-version** pin (§2) — distinct from the `app.lang` UI-language
  *state* (§9). They coexist; one is a version target, the other is game state.
- `profile` selects the root-level capability profile used to resolve active plugins (§6.1.1).
- `plugins` MAY add scene-local plugin activations/options on top of the selected profile (§6.1.1).
- `uses` imports the shared state schema (§9.2). Most scenes carry only `character`/`season`/
  `episode` + `profile` + `uses` (and at most a few `scene.*` declarations); the `run`/`user`/`app`
  schema and shared `defs` live in the imported schema document, not per scene.

#### 6.1.1 Capability profile activation

`profile` is a root-level scene capability selector, not a plugin-local field. If absent, the
project's `defaultProfile` applies. The reserved profile name `global` is inherited by every other
profile before that profile's own settings are applied.

`plugins` is a map from plugin id to a typed option object. Presence of a plugin id activates that
plugin for this scene; an empty option object means active with default options. The shorthand
`true` MAY be accepted and normalized to an empty option object. There is no `plugins.use` list in
v0.0.1.

```
profile: date-minigame
plugins:
  idola.minigame:
    resultScope: scene
    allowedKinds: [rhythm, timing]
```

Effective capabilities are resolved in this order: `global` profile → selected profile's `extends`
chain (parent first) → selected profile → scene-local `plugins`. Plugin activation is additive;
v0.0.1 does not define scene-local plugin disabling. The checker, LSP, and compiler MUST validate
the document against the same resolved capability snapshot.

### 6.2 Title

```
Title ::= "# " Text
```

An optional document-level H1 title MAY appear after `Meta` and before the first shot.

### 6.3 Shots

```
ShotBlock   ::= ShotHeading Node*
ShotHeading ::= "## " ( "Shot" | "Scene" ) WS Integer "." ( WS Text )?
              | "## " ( "프롤로그" | "에필로그" ) ( WS Text )?
```

A shot heading opens a shot. Shots are flat (not nested), ordered, and are the document's
sequence boundaries. Shot integers SHOULD be ascending; gaps are permitted. `Shot` and `Scene`
are equivalent heading keywords.

## 7. Grammar (nodes)

```
Node ::= Prose | Line | Mono | Directive | Set | Branch | Match | Timeline
```

(Comments and blank lines are trivia, §4.1–4.2, not nodes.)

### 7.1 Content

```
Line  ::= ":line[" Speaker "]" Attrs? ":" WS Text
Mono  ::= ":mono:" WS Text
Prose ::= Text                          (* the fallback classification, §4.3 *)
```

`Line` attributes are content metadata (e.g. `code`, `emotion`, `variant`, `action`); their value
domains are vocabulary, not grammar (Appendix A).

### 7.2 Staging directive

```
Directive ::= "::" Ident Attrs?
```

A single-line leaf. The `Ident` and attribute meanings are **vocabulary** (Appendix A), extensible
without grammar change. Staging timing attributes (§7.5) MAY appear on staging directives; `at` is
valid only for timeline clips.

### 7.3 Logic blocks

```
Branch    ::= "<branch" Attrs ">" Choice+ "</branch>"           (* requires id *)
Choice    ::= "<choice" Attrs ">" Node* "</choice>"             (* requires id, label; optional when, persist/as/value *)
Match     ::= "<match" Attrs ">" When+ Otherwise? "</match>"    (* requires on *)
When      ::= "<when" Attrs ">" Node* "</when>"                 (* requires test *)
Otherwise ::= "<otherwise>" Node* "</otherwise>"
```

Tag attributes use the §4.5 `Attrs` form (order-independent). Required attributes per tag: `branch`
→ `id`; `choice` → `id`, `label` (optional `when` as a `CelString`; optional run-promotion sugar
`persist`, `as`, `value` — §11.1); `match` → `on` (a `CelString`); `when` → `test` (a `CelString`).
Missing/unknown required attributes are a static error.

### 7.3.4 `set`

```
Set      ::= "::set{" Path WS AssignOp WS CelExpr "}"
AssignOp ::= "=" | "+=" | "-=" | "*="
```

Exactly **one** assignment per `::set`. The operator is a token (not a string value); the
right-hand side is a CEL expression. The `Path` MUST be a declared state path (§9).

### 7.4 Timeline

```
Timeline ::= "<timeline" Attrs? ">" Track+ "</timeline>"        (* optional duration *)
Track    ::= "<track" Attrs ">" Clip+ "</track>"               (* requires one of subject|channel|property *)
Clip     ::= Directive | Set
```

A `<timeline>` body MUST contain only `<track>`s (and trivia); a `<track>` body MUST contain only
staging leaves / `::set`. `:line`, prose, and logic blocks MUST NOT appear inside a `<timeline>`.
A staging clip MAY carry `at` plus the staging timing attributes (§7.5); a `::set` clip carries no
timing and fires at its sequential position.

### 7.5 Timing attributes

`duration`, `delay`, and `wait` are reserved staging timing keys. They MAY appear on staging
directives, including directives used as timeline clips. `at` is a reserved timeline-position key:
it MAY appear only on staging directives inside `<track>` and is a static error outside a timeline.

## 8. Expressions

### 8.1 CEL & definitions

All conditions and `::set` right-hand sides are CEL. `meta.defs` declares named, typed CEL values
— and is the namespace of legal `@ref`s:

```
defs:
  <name>: { type: bool|number|enum, params?: { p: <type> }, cel: "<CEL>", min?, max?, values? }
```

A `@name` / `@name(args)` reference is a **compile-time macro**: it MUST be expanded to its inline
CEL before evaluation, with `params` bound from the call arguments. A `@ref` MUST appear in a
position whose required type matches the def's declared `type`, and its `name` MUST be declared in
`defs`. `defs` MAY live inline in a scene's frontmatter or in an imported state schema (§9.2);
shared `defs` (those whose CEL reads `run`/`user`/`app`) belong in the schema document. Duplicate
`defs` names across imports are an error (§9.2).

### 8.2 The `$` subject

Inside a `<match on="S">`, the token `$` within a `<when test>` resolves to the subject expression
`S`. `$` MUST NOT appear outside a `<match>`.

### 8.3 No inline code

Conditions and values are CEL only. There is no host-language (JS, Lua, …) evaluation anywhere in
the language.

## 9. State

### 9.1 Namespaces (lifetime tiers)

Four namespaces, each named by its **reset boundary** (when it clears):

| Namespace | Reset boundary | Example |
|---|---|---|
| `scene.*` | **episode end** — one `.lute` document; **survives across its shots** (§6.3), cleared when the episode ends | on-stage state, `scene.choices.*` |
| `run.*`   | new run — one attempt (a sequence of episodes); cleared on restart | per-attempt flags, affect, cross-**episode** carry *within* an attempt |
| `user.*`  | profile/account wipe — survives runs | level, unlocks, meta-progression |
| `app.*`   | app uninstall — identity-independent | language, age rating, settings |

> **Naming note:** the `scene.*` tier's lifetime is the **episode** (the whole document). The
> `## Scene N.` / `## Shot N.` headings (§6.3) are *shots within* that document, not reset
> boundaries — `scene.*` (incl. `scene.choices.*`) persists across shots and clears only at episode
> end. Carrying a value to the *next episode* in the same attempt requires `run.*`.

The four are points on one axis (when it resets). Every path read **or written** (including by
`::set` and by branch recording, §11.1) MUST be declared (`type` + optional `default`) in the state
schema; there are **no** bare, un-namespaced state names. Each namespace maps to an engine
persistence backend; the engine **owns and fires** every reset boundary — the language never
triggers a reset.

### 9.2 Schema source & import (`uses`)

The `run`/`user`/`app` schema is **game/season-global** (one persisted value cannot have per-scene
types), so it lives in a single source-of-truth schema document, imported by each scene's
frontmatter via `uses:`. Only genuinely scene-local declarations (`scene.*`) MAY appear inline in a
scene, and a scene MUST NOT redeclare or override an imported `run`/`user`/`app` path. Imports form
a **DAG**:

- cycles are a static error (the diagnostic prints the chain);
- imported schemas are loaded/checked **before** any scene;
- duplicate `defs` names across imports are an error (no silent shadow);
- import refs are canonicalized, so two paths to the same file are **one** schema identity.

(The state schema is *game content*; it is distinct from the engine **capability manifest**, which
is *engine vocabulary* — separate owner, separate change cadence.)

### 9.3 Declaration form

```
state:
  scene.<path>: { type: number|bool|enum, default?: <literal> }
  run.<path>:   { type: …, default?: … }
  user.<path>:  { type: …, default?: … }
  app.<path>:   { type: …, default?: … }   # read-only to content (§9.5)
```

A `default` is materialized into the initial tier state at schema load **and** re-materialized
whenever the engine fires that tier's reset boundary (a cleared tier reloads its default; checker
and engine read the one schema snapshot and cannot disagree).

### 9.4 Definite assignment (path-sensitive)

Reading an undeclared path is a static error (`E-UNDECLARED`) — even when the tier exists but the
sub-path does not. For assignment:

- A `scene.*` read follows ordinary path-sensitive analysis (a read not provably assigned on the
  current path is an error unless guarded, e.g. `has()`/`isSet()`).
- A `run.*` / `user.*` / `app.*` path is **maybe-unset at scene entry** unless it carries a schema
  `default`. After entry, a dominating `::set{p = …}` write or an enclosing guard proves it; a read
  with no default, no dominating write, and no guard is `E-MAYBE-UNSET`.
- A compound assignment (`+=` / `-=` / `*=`) **reads** the old value, so it is itself subject to the
  above; only `=` (a pure write) may be the first assignment on a path.

`E-UNDECLARED` and `E-MAYBE-UNSET` are distinguishable diagnostics.

### 9.5 Write policy

`::set` MAY target `scene.*` / `run.*` / `user.*`. `app.*` is **read-only to content**:
`::set{app.*}` is a static error (the engine/settings layer owns `app.*` writes; content reads
`app.*` via `<match>`/`<when>` for gating — see §11.2).

### 9.6 Reserved namespaces

`run.choiceLog.*` is a **reserved, engine-populated** sub-namespace (an analytics/debug trace of
selected choices, §11.1.2). It is exempt from the author-declaration rule (§9.4) — the engine owns
it, as with `app.*` — and content **MUST NOT** read it in a `<match>`/`<when>`/guard (a static lint
error). Authors declare and branch on **named** `run.*` facts, never the raw log.

## 10. Reserved / collision rules

New tag and directive names MUST NOT collide with existing domain terms. In particular `cut` and
`scene` are reserved (the `::cut` directive; `## Scene` headings; scene ids), which is why the
timeline block is `<timeline>` and its tracks `<track>` (not `<cutscene>` / `<lane>`). The
attribute keys in §7.5 are reserved across all directives.

## 11. Semantics (normative)

### 11.1 Branch
`<branch id>` MUST be unique **within the episode** (the `.lute` document) — not across the run.
Presenting it offers each `<choice>` whose `when` (if any) is true. Selecting a choice records
`scene.choices.<branchId> = <choiceId>` (this path is implicitly declared with domain = the
branch's choice ids ∪ `unset`, and is **episode-scoped** per §9.1). `label` is the player-facing
text; `id` is the recorded key.

#### 11.1.1 Cross-episode carry (named run facts, not raw choice keys)
`scene.choices.*` clears at episode end, so it serves **intra-episode** reactions only (a later
shot's `<match on="scene.choices.x">`). To make a choice affect a **later episode** of the same
run, write a **named, declared** `run.*` fact — cross-episode state is a *domain outcome*, never a
remembered branch key. Two forms:

- **Explicit (primitive):** `::set{run.<path> = <value>}` in the choice arm (§7.3.4).
- **Sugar:** `<choice … persist="run" as="<run.path>" [value="<lit>"]>` desugars to a
  `::set{run.<path> = <value>}` appended to that arm. `value` defaults to `true` for a `bool`
  path; for an `enum`/`number` path it is required. The sugar is **exactly** the explicit form —
  no extra power.

**Why not record raw choice keys into `run`.** Promoting `<branch id>` into `run.choices.<id>`
would force `<branch id>` to be **globally unique across all episodes in a run** and couple a later
episode to an earlier one's *menu-arm naming* — an implementation detail, not authored consequence.
Named run facts avoid both: ids stay episode-local, and a later episode reads a meaningful fact
(`run.metHelpfully`, `run.sofaHelpOutcome`), not "which arm ep02 happened to call `couch`."

**Schema rule (no implicit declaration).** `as="<run.path>"` MUST resolve to a path **already
declared** in the imported run schema (§9.2); an undeclared/typo'd `as` is `E-UNDECLARED` (state-by-
typo MUST fail, never silently create a field). The compiler further checks: the path is writable
from this episode/choice context (§9.5); `value` is type-compatible with the declared path; and no
two writes in the same arm conflict on one path.

#### 11.1.2 `run.choiceLog` (reserved analytics trace)
The engine MAY maintain a reserved, engine-populated trace `run.choiceLog.<episode>.<branchId> =
<choiceId>` for analytics/debug. It is **not** author-declared (§9.6) and content **MUST NOT**
branch on it — a `<match>`/`<when>`/guard that reads `run.choiceLog.*` is a static (lint) error.
Cross-episode branching reads named `run.*` facts (§11.1.1), never the raw log.

### 11.2 Match
`<match>` arms are evaluated **top to bottom; first match wins**. An `<otherwise>` is **REQUIRED**
unless the subject's domain is finite and fully covered by the arms. `unset` is a member of a
nullable subject's domain — including a `run.*`/`user.*`/`app.*` subject that is maybe-unset at the
read site (§9.4); such a subject MUST be covered (an arm for `unset`, a guard that excludes it, or
`<otherwise>`). **Age gating:** an age-gated `<match on="app.rating">` MUST cover `teen` or carry
`<otherwise>` — a release build rejects a missing safe fallback (runtime gating alone is not
sufficient).

### 11.3 Timing & `wait`
`wait="true"` on a staging directive holds the script until that effect completes; an absent /
`false` `wait` is non-blocking (the next item proceeds — i.e. concurrently). The default is
**per-directive** (e.g. background and video default to `wait="true"`), not global.

### 11.4 Timeline time model
Within a `<timeline>` the local clock starts at `0`. For each track: a clip's `at` is an
**absolute** time on that clock; an **omitted** `at` starts the clip after the previous clip in
the same track (the first omitted `at` = `0.0`). `at` is never a relative nudge. Each track key
(`subject` / `channel` / `property` value) MUST be unique within the timeline (**one writer per
track**). Tracks play concurrently. The block blocks following content until `duration` (or, absent
`duration`, the maximum resolved clip end). A timeline MUST reduce to scheduled command records
(with resolved absolute offsets) plus a final barrier.

### 11.5 Reduction
A conforming document MUST reduce, at build time, to a finite ordered sequence of command records
+ CEL strings (invariant §3.2). Implicit staging that an implementation injects (stage hygiene) is
**implementation-defined** and outside this language spec; it MUST be deterministic and MUST NOT
change the meaning of explicitly authored constructs.

## Appendix A — Standard staging vocabulary (v0.0.1 baseline)

The baseline directive set. Names and existing attributes are the source of truth; new capability
is layered on, never renamed. The vocabulary is extensible by an implementation; this appendix is
the v0.0.1 baseline, not a closed set.

| Directive | Attributes |
|---|---|
| `::bg` | `location`, `time`, `assetId` |
| `::music` | `action`=`start\|change\|stop\|resume\|fade-out`, `mood`, `volume`=`silent\|down\|normal\|up\|full`, `assetId`, `track` |
| `::sfx` | `sound`, `assetId`, `name` |
| `::auto` | `character`, `anchor`=`left\|center\|right`, `action` (action-id, e.g. `fade-in-up`/`fade-out-down`/`pose-*`) |
| `::vfx` | `type`=`whiteOut\|blackOut\|rain\|snow\|leaves\|petals\|raindrop`, `label`, `transition` |
| `::cut` | `assetId` (`CUT.*`), `action`=`show\|hide`, `full` |
| `::video` | `assetId` (`VID.*`), `action`=`show\|hide`, `wait` |
| `::camera` *(new)* | `focus`, `zoom`, `move-x`, `move-y`, `shake`, `reset`, `duration`, `easing`, `delay`, `wait` |
| `::set` | `Path AssignOp CelExpr` (statement form, §7.3.4) |
| `:line[name]` | `code`, `emotion`, `variant`, `action`, `dialogMotion` |

## Appendix B — Open items

- Block comments use `/* ... */` body syntax (§4.2). A separate line-comment form is deferred.
- `property=` timeline tracks (split one subject across property-scoped tracks) — deferred.
- Persistence backends per tier (`run`/`user`/`app`) are engine-owned (§9.1); ship order
  `scene` → `run` → `user`/`app`.
- Schema-import (`uses`, §9.2): start flat (one game-global schema); additive `extends` and
  project-resolved schema ids (vs relative paths) are deferred, with the resolver designed as a DAG
  now so both land without rework.

## Appendix C — Revision note

This draft was revised after a grammar↔example consistency audit (vs the bianca S01EP02 example),
which surfaced: undefined terminals (`Meta`/`Text`/`String`/`CelString`/`Path`/`Integer`),
document-level title/comments, comments as trivia (trailing + between children), a normative line-
classification precedence, order-independent tag attributes, blank-line handling, prologue/epilogue
headings, and the rule that `::set` paths and recorded branch ids are declared state. All are
folded in above.

**State-model revision (folded into §6.1/§9/§11.2).** The state model moved from a two-namespace
`scene`/`save` pair to the four-tier lifetime ladder `scene`/`run`/`user`/`app` (named by reset
boundary, not storage); `meta` declarations moved from a `:::meta` fence to `---` frontmatter
(removing `:::` from the grammar entirely); the `run`/`user`/`app` schema became a single imported
SoT (`uses`, §9.2); and definite-assignment became path-sensitive with a scene-entry maybe-unset
seed for non-`scene` tiers (§9.4). This was reviewed by a codex DSL-critic peer over two rounds
(both blocking findings — path-sensitive assignment, normative import rules — resolved before
fold-in). Because `0.0.1` is pre-implementation and unpublished, this is a correction *within*
`0.0.1`, not a breaking version bump. Design rationale + audit trail:
[`state-model-design.md`](state-model-design.md).
