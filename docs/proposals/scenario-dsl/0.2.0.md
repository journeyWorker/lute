# Lute Scenario DSL — Document Kinds & the Quest Kind

- **Version:** 0.2.0
- **Status:** Draft (extension of 0.1.0; **`kind:` is required on every root document**)
- **Date:** 2026-07-07
- **Scope:** the **document-kind system** (making `.lute` polymorphic across authoring
  domains), one new logic construct (the `<on>` ECA trigger), one new state tier
  (`quest.*`), and the **quest kind** specified in full as the exemplar.
- **Relationship to 0.1.0:** this document is an **extension**, not a replacement.
  [`0.1.0.md`](./0.1.0.md) remains the normative source for the **scene (visual-novel)
  kind** and for the **shared kernel** — the logic layer (§7.3), Lute-CEL (§8), the state
  model (§9), totality (§3), and `lineId`/i18n (§12). This document references those
  sections rather than restating them; unqualified `§N` citations refer to 0.1.0.
- **Out of scope:** the reward / achievement / daily-quest kinds (follow-on 0.2.x specs);
  the runtime engine and SDK; any game-specific capability pack; expansion of the Lute-CEL
  profile. See §9 (Non-goals).
- **Inline examples are authoritative.** Every example here is 0.2.0 syntax.

The key words **MUST**, **MUST NOT**, **SHOULD**, **MAY** are to be interpreted as in
RFC 2119.

---

## 1. Abstract

0.1.0 is a language for authoring **visual-novel episodes**. Its foundation, however, is
domain-neutral: a **total** (not Turing-complete) declarative kernel that reduces every
document to a finite, ordered sequence of command records plus CEL expression strings
(§3), gated by a **tiered, typed state model** (§9) with **static** exhaustiveness and
definite-assignment guarantees (§11.2, §9.4).

0.2.0 exposes that kernel to **other authoring domains** — quests, and (later) rewards,
achievements, daily quests — by making a `.lute` document **kind-polymorphic**. The logic
layer is shared across kinds; only the **vocabulary** (content-line shape and
`::`-directives) differs. The kernel's static guarantees therefore apply to quest logic
for free.

The design mirrors the most extreme interwoven case, **Baldur's Gate 3**, whose story is
authored in **Osiris** — a declarative, event-driven, Event-Condition-Action layer over a
separate simulation engine. Lute occupies Osiris's role, deliberately constrained to be
**total** (statically verifiable, reduces to JSON) where Osiris is Prolog-like. The
interweaving of quests↔story↔rewards is **data** (shared state + CEL gates + ECA
triggers); the simulation (combat, dice) is **code** and remains the engine's (§8 here).

## 2. Status & versioning

0.2.0 makes the document **`kind:`** (§3.1) a **required** top-level frontmatter key on
every root document — a breaking change over 0.1.0, permitted pre-1.0 (§2 of 0.1.0). There
is no released or merged corpus to migrate: the 0.1.0 example scenes acquire an explicit
`kind: scene` when they migrate (they are not migrated yet). Scene is **not** privileged as
a default — `.lute` is a family of kinds, each named explicitly.

A document MAY declare `meta.luteVersion: "0.2.0"`; absent that, an implementation assumes
the latest it supports. This required-`kind:` change is the pre-1.0 breaking allowance
(§2 of 0.1.0) in use; the shared kernel and scene kind are otherwise unchanged from 0.1.0.

## 3. Document kinds

A `.lute` document has a **kind**. The kind selects (a) the required frontmatter keys, (b)
the **grammar admission** set (§3.3), (c) the per-kind vocabulary, and (d) the identity
rule (§7). 0.2.0 defines two kinds: **`scene`** (the 0.1.0 VN document) and **`quest`**
(§6).

### 3.1 The `kind:` discriminator

```
meta.kind ::= "scene" | "quest"          (* extensible by future revisions/kinds *)
```

`kind` is a **required** top-level frontmatter key (§6.1) on every **root/authored**
document — there is no default (scene is not privileged). A missing `kind:` is
**`E-KIND-MISSING`**; a `kind:` value that is not a defined kind is **`E-UNKNOWN-KIND`**.
It is resolved **before** the per-kind required keys (§3.1 table), since it selects which
set applies.

`Schema` documents (imported via `uses:`/`extends:`, §9.2) and `component` files (imported
via `components:`, §13) are **import-role** documents; they are orthogonal to the domain
kind and MUST NOT carry `kind:`.

Per-kind **required frontmatter keys** (a missing key is `E-META-MISSING`, §6.1):

| kind | required | notable optional |
|---|---|---|
| `scene` | `character`, `season`, `episode` | `episodeId`, `pov`, `profile`, `uses`, … (§6.1) |
| `quest` | — (no VN triad) | `uses` (the gated schema), `profile`/`plugins` (event + state-shape capability), `title`, `contentLang` |

The quest kind's identity is per-`<quest id>` (§7), so no file-level id key is required.

### 3.2 Shared kernel vs per-kind vocabulary

The 0.1.0 three-layer model (§5) is the seam between what is shared and what varies:

| Layer (§5) | Across kinds |
|---|---|
| **Logic** — `<branch>`, `<match>`/`<when>`/`<otherwise>`, `<hub>`, `::set`, CEL, `defs`, state schema, totality, `lineId`/i18n | **shared kernel** |
| **Content** — `:speaker: text` grammar shape | **per-kind vocabulary** |
| **Staging** — `::`-directives | **per-kind vocabulary** |

The `lineId`/i18n/`{{…}}` *plumbing* under content is kernel (re-skinned per kind, §7);
only the content-line *grammar shape* (speaker / `delivery` / `voiceKey`) is scene-kind
vocabulary. The static analyses (exhaustiveness §11.2, definite-assignment §9.4, totality
§3) live entirely in the kernel and are therefore **kind-independent**.

### 3.3 Grammar admission (closed grammar, per-kind subset)

The grammar is a **fixed, closed set** of constructs. Plugins extend **vocabulary**
(directive names, attributes, assetkinds, state shapes, and `events`, §4.5) but **MUST
NOT** add grammar — plugin-defined blocks would demote the static analyses to fixpoint
computations (the reason backward diverts are a non-goal, §14).

Each kind **admits a subset** of the closed grammar. A construct that appears where its
kind does not admit it is **`E-GRAMMAR-NOT-ADMITTED`** — a dedicated diagnostic, not a
generic classification fallthrough. This is the 0.1.0 per-context restriction (§7.4 already
forbids content/logic inside `<timeline>`) lifted to the document kind.

Closed grammar set (0.2.0) and admission:

| Construct | `scene` | `quest` |
|---|---|---|
| content-line `:speaker: text` (§7.1) | ✓ | ✓ (in arms) |
| `::directive` (§7.2) | ✓ | ✓ |
| `::set` (§7.3.3) | ✓ | ✓ |
| `<branch>` (§7.3) | ✓ | ✓ |
| `<match>` (§7.3) | ✓ | ✓ |
| `<hub>` (§7.3.2) | ✓ | — |
| `<timeline>` (§7.4) | ✓ | — |
| `# `/`## ` headings (§6.2–6.3) | ✓ | — |
| **`<on>`** (§4, NEW) | — | ✓ |
| **`<quest>`** (§6.3, NEW) | — | ✓ |
| **`<objective>`** (§6.4, NEW) | — | ✓ |

`<timeline>` remains an **opt-in kernel module**: a `<…>` kernel structure whose clips are
per-kind vocabulary (the same container/payload shape as `<branch>` arms), not universal.
In 0.2.0 `<on>` is admitted by the **quest** kind only; scene-kind event reactions are
deferred (a scene would need a defined placement in its shot sequence — out of scope here).

## 4. The `<on>` trigger (Event-Condition-Action)

### 4.1 Grammar

```
On ::= "<on" Attrs ">" Node* "</on>"      (* requires event; when is an optional guard *)
```

`<on>` is a nesting logic construct (§5 bracket rule). Attributes:

- **`event`** (required) — a `String` naming an **event kind**: a built-in lifecycle event
  (§6.6) or a capability-declared world event (§4.5). Every `<on>` is anchored to a
  discrete event; there is no pure state-watch trigger in 0.2.0 (§4.2).
- **`when`** (optional) — a `CelString` **guard** over declared state (the closed Lute-CEL
  profile, §8.4), evaluated at the instant `event` fires. The `$` subject (§8.2) is **not**
  in scope (that token is `<match>`-only).

An `<on>` without an `event` is **`E-ON-NO-EVENT`**. An `event` naming no declared event
kind (built-in or capability) is **`E-UNKNOWN-EVENT`**.

### 4.2 Semantics

`<on>` is the language's **Event-Condition-Action** rule. When its `event` fires and its
`when` guard (if present) holds **at that instant**, the arm's `Node*` is emitted as
command records. Because every rule is anchored to a discrete event, firing is
well-defined without any engine-specific re-evaluation schedule.

```lute
<on event="combatEnd" when="run.npc.halsin.dead">
  ::bg{location="grove" time="aftermath"}
  :narrator: Halsin had fallen. The grove went silent.
  ::set{run.groveOutcome = "halsinDead"}
</on>
```

The engine runs combat, detects the death, writes `run.npc.halsin.dead`, and fires
`combatEnd`; Lute matches and emits the authored records. The engine is an **opaque event
source** — a `.lute` document never expresses *how* combat runs (§8).

**Firing model (normative).** Unlike `<match>` (top-to-bottom, first-match-wins, §11.2),
`<on>` rules are **all-match**: for one fired event, **every** rule matching that event
whose `when` holds fires. To make this deterministic and order-insensitive at the guard
level, all guards are evaluated against the **pre-event state snapshot** (a guard does not
observe a sibling handler's writes); the matching arms then execute in **document order**,
their writes applying sequentially. Cross-document ordering is engine policy but MUST be
deterministic. Built-in lifecycle events are **quest-scoped** (§6.6): a `questComplete`
handler fires only for its own enclosing `<quest>`.

### 4.3 Totality

`<on>` preserves the §3 invariants by the same argument as `<hub>` (§3, hub totality): each
`<on>` **reduces at build time to finitely many records**; the engine's event loop merely
**consumes events each cycle**, introducing no unbounded author computation and no author
iteration. `<on>` is **forward-only** — an arm MUST NOT divert to an earlier position
(backward diverts remain a non-goal, §14). The "reduces to data" invariant (§3.2) holds.

### 4.4 Definite assignment

`<on>` arms have **no dominance relation** among one another (the same join rule as
`<match>` / `<hub>` arms, §9.4, §11.1.3): a write inside one arm is a **may-write**, never
a definite assignment. A path first written only inside `<on>` arms is maybe-unset unless
every arm writes it or it carries a schema `default`.

### 4.5 Event vocabulary (capability)

**World events** (`combatEnd`, `npcDied`, `regionEntered`, …) are **capability-provided**:
an active plugin (§6.1.1) declares them via an **`events` export** (the 0.2.0 plugin-system
erratum). Any associated payload is written by the engine into declared state **before**
the event fires, so `when` reads it as ordinary state. An `event` not in the resolved
capability snapshot (nor a built-in lifecycle event, §6.6) is `E-UNKNOWN-EVENT`. The
language defines **no** world events itself — this is the integration seam with the engine
(§8); the language stays small while a game supplies a large event vocabulary.

## 5. State: the `quest.*` instance tier

**Kernel deltas (normative).** `quest.*` extends the 0.1.0 kernel at three points:
- **Path terminal (§4.4):** `Path` gains `quest` as a fifth namespace root —
  `Path ::= ("scene" | "run" | "user" | "app" | "quest") ("." CelIdent)+`.
- **Schema declaration (§9.3):** `quest.<id>.<field>` scratch paths are declared with
  `type` + optional `default` exactly like the other tiers; the reserved sub-namespaces
  (§5.2) are implicitly declared and MUST NOT be author-declared.
- **Definite assignment (§9.4):** a `quest.*` scratch path is **maybe-unset at instance
  entry** unless it carries a schema `default` — the same rule as `run.*`/`user.*` (a
  dominating `::set` or a guard proves it, else `E-MAYBE-UNSET`).

### 5.1 Lifetime & reset boundary

0.1.0 has four tiers named by their reset boundary (§9.1). A quest is a **run-to-completion
instance** whose lifecycle none of the four can scope. 0.2.0 adds an **instance-scoped
tier `quest.*`**.

A terminal lifecycle state (`complete`/`failed`) is **absorbing and retained**: reaching
it does **not** clear the tier — `quest.<id>.state` keeps its terminal value so later
content can react to a completed/failed quest (§6.6). The engine clears `quest.<id>.*` only
when it **re-instantiates** that quest (a repeatable quest starting a fresh instance) or on
**run reset** — whichever comes first. The tier is thus **bounded above by the run** (a new
run clears all quest instances) and, for repeatable quests, per instance. As with every
tier, **the engine owns and fires the reset** (§9.1); the language never triggers it.

### 5.2 Reserved sub-namespaces

Path shape: `quest.<questId>.<field>`, each segment a `CelIdent` (no `-`, §8.4). Two
sub-namespaces are **reserved, engine-populated, and implicitly declared** (as
`scene.choices.*` is, §9.6) — content MAY read them in `<match>`/`<when>` but **MUST NOT**
write them:

- **`quest.<id>.state`** — the lifecycle enum `active | complete | failed`
  (∪ `unset` before the quest is known). Reading it is subject to `unset` coverage
  (§11.2): a `<match on="quest.q.state">` MUST cover `unset` via an arm or `<otherwise>`.
- **`quest.<id>.objectives.<oid>.done: bool`** (default `false`) — per-objective
  completion, populated when the objective's `done` predicate holds (§6.4).

Authors MAY additionally **declare** ordinary `quest.<id>.*` scratch fields in the state
schema (§9.3); these are writable via `::set` inside that quest's arms and follow ordinary
definite-assignment (§9.4).

### 5.3 Scheduled tiers (reserved)

Time-scheduled tiers **`daily.*`** / **`weekly.*`** — reset on a cron boundary, engine-owned
— are a **reserved extension point**, not defined in 0.2.0. The tier mechanism is designed
to admit them so the future daily-quest kind lands as a pure state-model extension with no
language change.

### 5.4 Write policy

`::set` MAY target declared `quest.<id>.*` scratch fields, plus `scene.*`/`run.*`/`user.*`
as in §9.5. Writing a reserved `quest.<id>.state` or `…objectives.*.done` path is
**`E-QUEST-RESERVED-WRITE`** (the engine owns them). `app.*` remains read-only to content (§9.5).

## 6. The quest kind

### 6.1 Frontmatter

A quest document declares `kind: quest` (§3.1). It carries `uses:` for the schema it gates
on, MAY carry `profile`/`plugins` (the event + state-shape capability, §6.1.1), and MAY
carry `title`, `contentLang`, `luteVersion`. It MUST NOT carry the scene triad
(`character`/`season`/`episode`) — those are scene vocabulary.

### 6.2 Document structure

```
QuestDoc  ::= Meta QuestDecl+
```

A quest document is a **pack** of one or more `<quest>` declarations (realistic: quest
files hold many quests). Trivia (blank lines, comments, §4.1–4.2) MAY appear between
declarations. Only `<quest>` is admitted at the document top level; any other construct
there is `E-GRAMMAR-NOT-ADMITTED` (§3.3).

### 6.3 `<quest>`

```
QuestDecl ::= "<quest" Attrs ">" QuestBody "</quest>"
QuestBody ::= ( Objective | On | Match | Branch | Set | Directive | Line )*
```

Attributes:

- **`id`** (required) — a stable **`CelIdent`** (no `-`, §8.4) — it is embedded in the
  `quest.<id>.*` CEL path (§5), so it MUST be CEL-safe. **Unique across the project** (the
  import graph): the global instance key for `quest.<id>.*` state (§5) and
  the identity prefix for the quest's localizable text (§7). A collision is
  **`E-QUEST-ID-DUP`**. (Global uniqueness is intended — like a named `run.*` fact,
  §11.1.1 — not an implementation leak; a quest id is a meaningful, cross-referenced entity.)
- **`title`** (optional) — a `String` (MAY interpolate, §7.6); its `lineId` is
  `{questId}.title`.
- **`start`** (optional) — a `CelString`; when it holds, the engine transitions the
  quest `unset` → `active` (declarative activation). Absent, activation is engine-driven
  (an offer/accept flow the game fires).
- **`fail`** (optional) — a `CelString`; when it holds while `active`, the engine
  transitions → `failed`. **`fail` takes precedence over completion**: if `fail` and the
  completion condition hold in the same state, the quest is `failed` (deterministic tie-break).

**Completion** is **derived**: a quest becomes `complete` when every non-`optional`
objective is `done` (§6.4). An objective's `when` (§6.4) controls only its *visibility /
tracking*, **not** the completion obligation — a required (non-`optional`) objective must
still become `done` to complete the quest, so a required objective whose `when` can be
permanently false makes completion unreachable (mark such objectives `optional`).
Objective completion is **monotonic** within an instance: once `done` holds it stays
recorded (a completed objective does not un-complete). An author does not write
`quest.<id>.state` (§5.4); the engine derives all transitions from `start`, `fail`, and
objective completion — pure predicates over state, keeping the lifecycle **total** and
read-only to content.

### 6.4 `<objective>`

```
Objective ::= "<objective" Attrs ">" Node* "</objective>"
           |  "<objective" Attrs "/>"          (* self-closing when the body is empty *)
```

Attributes:

- **`id`** (required) — a **`CelIdent`** (no `-`, §8.4) — it is the `<oid>` segment of the
  `quest.<id>.objectives.<oid>.done` CEL path (§5.2), so it MUST be CEL-safe. Unique within
  the quest (a duplicate is **`E-OBJECTIVE-ID-DUP`**); also the objective's `lineId`
  (`{questId}.{objectiveId}`, §7).
- **`done`** (required) — a `CelString` completion predicate over declared state. Missing
  `done` is **`E-OBJECTIVE-MISSING-DONE`**.
- **`when`** (optional) — a `CelString` gating the objective's *visibility / tracking* (a
  hidden or not-yet-relevant objective); it does **not** relax the completion obligation
  (§6.3).
- **`title`** (optional) — the objective's localizable description (`String`, MAY
  interpolate).
- **`optional`** (optional) — a bare boolean flag; an `optional` objective does not gate
  the quest's derived completion (§6.3).

**Objective body (execution).** An objective's body (`Node*`) is emitted as command records
**once, when the objective transitions to `done`** (its `done` predicate first holds) —
the completion transition is the discrete engine event that triggers the body. Because
completion is monotonic (§6.3), the body emits exactly once per instance. A body MAY carry
content (e.g. a log line) and `::set` (e.g. a per-objective reward); it MUST NOT contain
another `<objective>` or `<quest>`.
A body MAY be empty. Because the common objective has no completion body, an empty-body
objective SHOULD be written **self-closing** — `<objective …/>` — the author-ergonomic form;
the `<objective …></objective>` long form is equivalent. This is 0.2.0's refinement of the
§5 bracket rule: a **body-capable** construct with an empty body MAY self-close, while a
**pure leaf** (never nestable, e.g. `::set`) still uses the `::`/`:` form (so `::set` is not
`<set/>`). Constructs that require ≥1 child (`<branch>`, `<hub>`, and in practice
`<quest>`/`<on>`) never self-close.

### 6.5 Rewards

Rewards reuse existing mechanisms — no new core grammar:

- primitive: `::set{run.* / user.* = …}` in an arm (§7.3.3);
- sugar: the 0.1.0 `persist` form (`persist="run" into="run.path" [value=…]`, §11.1.1).

A richer `::grant{…}` is **capability vocabulary** (engine-provided), validated like any
`::`-directive; it is not part of the language.

### 6.6 Lifecycle & reactions

The quest kind defines three **built-in lifecycle events**, fired by the engine on the
corresponding transition and usable as `<on event>` (§4): **`questActive`**,
**`questComplete`**, **`questFailed`**. These are **quest-scoped**: a lifecycle-event
handler fires **only** for the transition of its own enclosing `<quest>` — a
`questComplete` handler in `<quest id="q1">` never fires for another quest's completion.
(World events remain capability-provided and global, §4.5.)

```lute
<quest id="rescueHalsin" title="Rescue Halsin"
       start="run.act == 1 && run.party.shadowheart"
       fail="run.npc.halsin.dead">
  <objective id="reachGrove" title="Reach the grove" done="run.region == 'grove'"/>
  <objective id="freeHalsin" title="Free Halsin" done="run.npc.halsin.freed"/>

  <on event="questComplete">
    ::set{user.xp += 300}
    ::set{run.metHalsin = true}
    :narrator: The First Druid drew a slow breath. "You have my thanks."
  </on>

  <on event="combatEnd" when="run.npc.halsin.dead">
    :narrator: The cage held only silence now.
  </on>
</quest>
```

Content also gates on lifecycle via state (§5.2):

```lute
<match on="quest.rescueHalsin.state">
  <when is="complete"> :shadowheart: You did well back there. </when>
  <when is="failed">   :shadowheart: We were too late. </when>
  <otherwise>          :shadowheart: We should keep moving. </otherwise>
</match>
```

### 6.7 Admitted grammar

Admission is **context-scoped** (`E-GRAMMAR-NOT-ADMITTED` on violation, §3.3):

- **document top level** — `<quest>` only.
- **directly in a `<quest>` body** — `<objective>`, `<on>`, `<match>`, `<branch>`, `::set`,
  content-line, `::directive`.
- **in an `<objective>` body or an `<on>` arm** — emittable nodes only: content-line,
  `::set`, `::directive`, `<match>`, `<branch>`. **No** nested `<objective>`, `<quest>`, or
  `<on>` (declarations and triggers are quest-body-level; `<match>`/`<branch>` may nest
  freely).

The quest kind **forbids** `<hub>`, `<timeline>`, and `# `/`## ` headings everywhere.

## 7. Identity & localization (per-declaration)

0.1.0 keys `lineId` on `meta.episodeId` — per **file** (§12). 0.2.0 generalizes: a
`lineId`'s prefix is the **enclosing stable declaration id**.

| kind | line `lineId` | label / text `lineId` |
|---|---|---|
| `scene` | `{character}.{episodeId}.{speaker}_{code}` | choice/hub `{character}.{episodeId}.{branchOrHubId}.{choiceId}` |
| `quest` | `{questId}.{speaker}_{code}` (in an arm) | quest title `{questId}.title`; objective `{questId}.{objectiveId}` |

This is a **generalization, not a change** to scene: the identity prefix is the enclosing
declaration's stable id — the *episode* for a scene, the *quest* for a quest. Every 0.1.0
i18n contract applies verbatim: assign-once (`code` persisted by `lute tag`, §12),
placeholder-set equality (`E-L10N-PLACEHOLDER`, §12), and the id-keyed translation sidecar.

## 8. State-contract with the engine (normative boundary)

Anything the **simulation** produces is **computed by the engine and surfaced as declared
state**; a `.lute` document **reads** it via CEL and **never computes it**. This includes:

- randomness and dice rolls;
- combat outcomes, positions, timers;
- any **aggregate over a collection** — the closed Lute-CEL profile forbids
  `map`/`filter`/`exists`/`all`/`size` (§8.4 `E-CEL-PROFILE`). An aggregate (e.g. "any
  party member is a cleric") MUST be precomputed by the engine into a scalar declared path
  (`run.party.hasCleric: bool`) and read as ordinary state.

A **`::check`-style capability directive** illustrates the contract: a directive whose
bridge performs an engine computation and writes a declared `into` path, which content then
branches on.

```lute
::check{skill="persuasion" dc="15" into="scene.shCheck"}
<match on="scene.shCheck">
  <when is="success"> :shadowheart: ...fine. I believe you. </when>
  <when is="fail">    :shadowheart: Don't change the subject. </when>
</match>
```

The roll is the engine's; the check *declaration* and the *branch on its result* are the
author's. `::check` is capability vocabulary, not core grammar. The `in` operator (§8.4)
works over **list literals** with no engine help — e.g. `when="run.act in [1, 2]"` (a
number against a literal list). Membership in a *dynamic collection* (party roster,
inventory) is **not** list-valued state (0.1.0 state is `number`/`bool`/`enum` only, §9.3):
the engine surfaces it as a scalar fact — e.g. `run.party.shadowheart: bool` — read as
`when="run.party.shadowheart"`.

The Lute-CEL profile (§8.4) is **unchanged**. Opening it (comprehensions) is a non-goal
(§9): it would trade away the static guarantees and still could not produce simulation
facts. The default lever is the engine→state contract, not CEL expansion.

## 9. Non-goals

- **Reward / achievement / daily-quest kinds** — deferred to follow-on 0.2.x specs. Each
  is a thin vocabulary pack over this framework; an achievement is nearly a `<quest>` with
  a single objective and no instance lifecycle.
- **Scheduled tiers** (`daily.*`/`weekly.*`) — extension point reserved (§5.3), not
  defined.
- **The engine, SDK, and game-specific capability packs** — downstream of the language.
- **Lute-CEL expansion** — the environment stays closed (§8.4).
- **Backward diverts** — still a permanent non-goal (§14). `<on>` is forward-only; the ECA
  event loop is the total, DAG-preserving form of reactivity.

## Appendix A — Quest vocabulary baseline (0.2.0)

The baseline additions layered on the 0.1.0 vocabulary (Appendix A of 0.1.0). Extensible by
capability; not a closed set.

| Construct | Attributes |
|---|---|
| `<quest>` | `id` (req), `title`, `start` (CEL), `fail` (CEL) |
| `<objective>` | `id` (req), `done` (CEL, req), `when` (CEL), `title`, `optional` |
| `<on>` | `event` (req), `when` (optional guard, CEL) |
| `::check` | `skill`, `dc`, `into` (capability example, §8) |
| `::grant` | reward payload (capability example, §6.5) |

Built-in lifecycle events: `questActive`, `questComplete`, `questFailed`.

## Appendix C — Migration / compatibility

0.2.0 makes `kind:` **required** (§2). Since 0.1.0 is unreleased/unmerged and its example
scenes are not yet migrated, there is no deployed corpus to break:

- every existing 0.1.0 scene gains an explicit **`kind: scene`** line when it migrates (the
  only edit; the shared kernel and scene grammar are otherwise unchanged);
- `kind:` (now required), `<on>`, `<quest>`, `<objective>`, `quest.*`, and the built-in
  lifecycle events are new surface; nothing else in 0.1.0 is removed or reinterpreted;
- the tags `on`, `quest`, `objective` become reserved (§10 of 0.1.0 extends): a scene that
  used them as plugin directive/attr names (it could not — they were not core, but a
  capability MAY have) is the only theoretical conflict, surfaced at assembly time.

## Appendix D — Worked example (a minimal quest pack)

```lute
---
kind: quest
luteVersion: "0.2.0"
uses: ../schema/act1.schema.lute
profile: bg-encounter
title: Act 1 — Emerald Grove
---

<quest id="rescueHalsin" title="Rescue the First Druid"
       start="run.act == 1"
       fail="run.npc.halsin.dead">
  <objective id="reachGrove" title="Reach the Emerald Grove"
             done="run.region == 'grove'"/>
  <objective id="freeHalsin" title="Free Halsin from the cage"
             done="run.npc.halsin.freed"/>

  <on event="questComplete">
    ::set{user.xp += 300}
    ::set{run.metHalsin = true}
    :narrator: Halsin rolled his shoulders and looked north. "Moonrise, then."
  </on>

  <on event="combatEnd" when="run.npc.halsin.dead">
    ::set{run.groveOutcome = "halsinDead"}
    :narrator: The cage held only silence now.
  </on>
</quest>

<quest id="findMissingScouts" title="The Missing Scouts"
       start="run.metHalsin">
  <objective id="findScouts" title="Find the missing scouts"
             done="run.scouts.found >= 3"/>
  <on event="questComplete">
    ::set{user.reputation.grove += 1}
  </on>
</quest>
```
