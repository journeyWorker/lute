# Lute Scenario DSL — 0.7.0: Version Unification

- **Version:** 0.7.0
- **Status:** Draft (**version-unification release** of 0.6.1)
- **Date:** 2026-07-20
- **Scope:** no grammar production, no static-semantic rule, no IR shape, no
  `decide()` (§5.1 R1–R5) change, no Closure change. Language 0.7.0 is
  **byte-for-byte 0.6.1 semantics**. The single change is the version stamp:
  `LUTE_LANG_VERSION` and `LUTE_IR_VERSION` advance from `"0.6.1"` to
  `"0.7.0"` in the same release that aligns the toolchain (Cargo workspace) and
  the four npm packages at the same number.

## 1. Motivation

Lute versions along independent axes (see [`docs/versioning.md`](../../versioning.md)):
language, IR, toolchain, capability, plugin. That independence is honest — the
grammar an author writes against, the binary they install, and the artifact
schema an engine consumes are separate contracts — but by 2026-07-20 the
visible numbers had drifted far apart: language `0.6.1`, IR `0.6.1`, toolchain
`0.2.0`, npm packages `0.2.0`. A user reading `lute version` next to a
`luteVersion: "0.6.1"` stamp next to an npm `0.2.0` install had three numbers
for one release and no way to know they describe the same shipped artifact.

0.7.0 removes that confusion by **re-aligning every axis at one number** for
this release. It is not a grammar or IR change dressed up as a bump: the
compiled artifact for any 0.6.1-clean document is byte-identical except for the
version envelope line.

## 2. What changes

- `LUTE_LANG_VERSION` → `"0.7.0"`, `LUTE_IR_VERSION` → `"0.7.0"`.
- The Cargo workspace version and all four npm packages advance to `0.7.0`.
- The IR JSON schema file is renamed `schemas/lute-ir-0.6.schema.json` →
  `schemas/lute-ir-0.7.schema.json` (its `$id` tracks the filename); the schema
  body is unchanged — same envelope, same records.

Nothing else. Every diagnostic code, grammar production, admission rule, and
`decide()` result is exactly as 0.6.1 defined it. Historical spec citations of
the form "dsl 0.6.1 §N" continue to name the proposal that defined the rule and
are **not** rewritten.

## 3. Stale-stamp remedy

Because the language version advances, a document carrying
`luteVersion: "0.6.1"` now lags the checker's `LUTE_LANG_VERSION` and fires
**`W-LUTE-VERSION-STALE`** (0.6.1 §3) — the freshness warning that was added for
exactly this situation. The stamp is still never validated against capabilities
(D13 stands); the warning is a freshness signal only. The remedy is mechanical:
restamp to `luteVersion: "0.7.0"`. Since 0.7.0 is byte-for-byte 0.6.1 semantics,
a 0.6.1-clean document restamped to 0.7.0 checks clean with no other edit.

## 4. Non-goals

- No grammar, semantic, or IR shape change — this is purely a version bump.
- No new diagnostic codes; no code removed.
- No change to the axis-independence policy: axes MAY drift apart again when an
  axis genuinely does not change in a release. This release simply re-aligns the
  visible numbers so a single release presents a single number.

## Appendix A — Diagnostic codes

None added; none removed. The only clean-document verdict change is that a
document stamped `luteVersion: "0.6.1"` now fires the pre-existing
`W-LUTE-VERSION-STALE` (0.6.1 §3) against the new `LUTE_LANG_VERSION`.

## Appendix B — Version stamp

`LUTE_LANG_VERSION` and `LUTE_IR_VERSION` advance to `"0.7.0"`; the only
permitted golden diff is the version envelope line, in the same commit as the
constant bump.
