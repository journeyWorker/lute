# Lute Scenario DSL — 0.6.0: `into=` Records; Free Shot Headings

- **Version:** 0.6.0
- **Status:** Draft (**breaking revision** of 0.5.2 — one grammar production and
  one attribute's semantics change)
- **Date:** 2026-07-15
- **Scope:** two author-surface simplifications. **F1:** the `<choice>`
  run-record sugar is driven by `into=` alone; the `persist=` attribute is
  REMOVED. **F2:** a shot heading is free text; the `Shot|Scene <int>.` /
  bookend grammar and authored shot numbers are REMOVED — shots are numbered by
  document order. No IR record-shape change; the version envelope advances and
  shot addressing becomes positional (Appendix B).

## 1. Motivation

Both changes remove ceremony the checker itself already treats as redundant:

- `persist="run"` was "pure ritual: `run` is the **only** legal value
  (`E-PERSIST-TARGET`)" (0.4.0 §2). The two-attribute idiom created a real
  authoring trap: a bare `into=` **parsed, checked clean (exit 0, warning
  only), and recorded nothing** (`W-CHOICE-INTO-NO-PERSIST`, 0.4.0 §7.3). For
  an AI-driven pipeline judging by exit code, that is a silent no-op across an
  entire season. 0.4.0 §3 (B1–B3) deferred the implied-persist reinterpretation
  "to a future major that can afford the meaning change"; this is that
  revision. The audited corpus (`docs/examples`, 23 files) contains **zero**
  bare-`into=` choices, and there are no published artifacts — the meaning
  change is deliberate and its blast radius empirically empty.
- The shot-heading grammar (`## Shot|Scene <int>.` or a bookend keyword,
  0.1 §6.3) demanded a keyword, an integer, **and a trailing period**, and
  rejected everything else (`E-SHOT-HEADING` — `## Shot 3` was an error). The
  authored number's only consumer is artifact shot addressing
  (`lute-compile`), where the durable identity spine is the per-line `code=` /
  `lineId` (§12), not the shot address: artifacts are regenerated wholesale on
  every compile. Like an ordered list in Markdown, document order IS the
  number.

## 2. F1 — `into=` records (persist removal)

### 2.1 Grammar & semantics

On a `<branch>` or `<hub>` `<choice>`:

```
<choice id="help" label="Help her up" into="run.metHelpfully">          bool: records true
<choice id="warm" label="Stay" into="run.outcome" value="warm">         enum/number/string: value= required
```

`into="run.<path>"` alone is the record sugar: selecting the choice appends
`::set{run.<path> = <value>}` to the arm — byte-identical lowering to 0.4.0's
`persist="run" into=…` pair (normalize `synth`, IR unchanged). The `value=`
policy is unchanged (0.1 §11.1.1 rule 4): OPTIONAL for a `bool` path
(defaults `true`), REQUIRED and type-compatible for every other type.

The `persist` attribute is REMOVED from the language. It is not
"deprecated-but-accepted": a `<choice>` carrying any `persist=` attr is an
error (§2.2), with a machine-applicable migration.

### 2.2 Diagnostics

| Code | Sev | Trigger | Replaces |
|---|---|---|---|
| `E-PERSIST-REMOVED` | error | any `persist=` attr on a `<choice>` (0.6.0). One `"migrate"` fixit: delete the attr. `lute fix` applies it (see §2.3). | `E-PERSIST-TARGET`, `E-PERSIST-MISSING-INTO` |
| `E-INTO-TARGET` | error | `into=` names a non-`run.*` path — cross-episode records live in the `run.*` namespace | `E-PERSIST-TARGET` (path half) |
| `E-INTO-UNDECLARED` | error | `into="run.<path>"` not declared in the run schema; an undeclared/typo'd `into` cannot create a field | old rule 3 |
| `E-INTO-VALUE` | error | `value=` policy violation (rule 4) | `E-PERSIST-VALUE` |
| `W-INTO-SET-DUP` | warn | the choice arm already `::set`s the `into` path | `E-PERSIST-CONFLICT` (severity error → warn: the dup is redundant, not wrong) |

`W-CHOICE-INTO-NO-PERSIST` is REMOVED — its trigger condition is now the
feature working as intended.

### 2.3 Migration (`lute fix`)

`lute fix` gains ONE new rule: delete a `persist="run"` attr (and one adjacent
separating space) from a `<choice>` that also carries `into=`. This is
meaning-preserving in BOTH directions — the pair recorded before, the bare
`into=` records now — so it satisfies the D16 bar for an automatic, unprompted
codemod (`kind: "migrate"`), unlike 0.4.0's two `"refactor"` remedies, which
are retired with their diagnostic. A `persist=` attr with any other value, or
without `into=`, was already an error in 0.4.0 and stays manual
(`E-PERSIST-REMOVED`'s fixit still offers the deletion; `lute fix` applies it
only in the `persist="run"` + `into=` shape it can prove).

### 2.4 The accepted meaning change

A pre-0.6.0 document carrying a bare `into=` (no `persist=`) was a silent
no-op; under 0.6.0 it records. This is the ONE reinterpretation B1–B3
guarded against, accepted here because the guarded population is empty
(§1). Post-0.6.0, the silent-no-op state is unrepresentable: `into=` records,
and there is no second attribute to forget.

## 3. F2 — free shot headings

### 3.1 Grammar

```
ShotHeading ::= "## " Text        (Text non-empty after trimming)
```

The heading is an opaque title. `Shot 1.`, `Scene 2. Title`, `Prologue`,
`프롤로그`, `골목길에서`, `The Interrogation` are all equally valid and
equally meaningless to the toolchain beyond display. The keyword/integer/
period/bookend grammar of 0.1 §6.3, the `HeadingKind` classification, and
`E-SHOT-HEADING` are REMOVED.

### 3.2 Numbering

A shot's number is its 1-based document position. The AST `Shot.number`
field is removed; `lute-compile` addresses shots positionally (this was
already the fallback for bookend headings). Inserting a shot mid-document
shifts later shots' artifact addresses — accepted: the artifact is
regenerated and consumed as one unit, and stable line identity/voice keys
ride `code=`/`lineId` (§12), which are untouched.

### 3.3 Diagnostics

- `E-SHOT-HEADING` REMOVED (no invalid heading exists).
- `E-CONTENT-OUTSIDE-SHOT` message now names the free form: "content lives
  inside a shot; add a `## <title>` heading above it".
- `E-TITLE-PLACEMENT` (§6.2, the `# ` H1 rules) is unchanged.

### 3.4 Migration (`lute fix`)

Pre-0.6.0, `Shot N.` in `## Shot N. Title` was grammar, not title; under the
free-text rule it would leak into the display title. `lute fix` gains a
second 0.6.0 rule: strip the structural prefix from a heading that carries a
trailing title — `## Shot N. Title` / `## Scene N. Title` → `## Title`
(byte-exact, idempotent). A BARE `## Shot N.` / `## Scene N.` is left
untouched: stripping would leave an empty heading, and the bare form is a
valid (if redundant) title an author may keep or rename.

## 4. Non-goals

- **No IR record-shape change.** The synthesized `::set` lowering, command
  record fields, and `lineId`/`voiceKey` derivation (§12) are byte-identical.
- **No new persistence tiers.** `into=` writes `run.*` only; `user.*`/`app.*`
  writes remain the engine's (0.1 §9).
- **No heading semantics.** A heading that happens to read `Shot 7.` in a
  5-shot document carries no meaning and raises no diagnostic.

## Appendix A — Diagnostic codes

New: `E-PERSIST-REMOVED`, `E-INTO-TARGET`, `E-INTO-UNDECLARED`,
`E-INTO-VALUE`, `W-INTO-SET-DUP` (§2.2 — the last four are renames of the
`E-PERSIST-*`/`W-PERSIST-*` family, same triggers minus the `persist` attr).
Removed: `E-PERSIST-TARGET`, `E-PERSIST-MISSING-INTO`, `E-PERSIST-VALUE`,
`E-PERSIST-CONFLICT`, `W-CHOICE-INTO-NO-PERSIST`, `E-SHOT-HEADING`.

## Appendix B — Version stamp

`LUTE_LANG_VERSION` and `LUTE_IR_VERSION` advance to `"0.6.0"`
(`lute-compile/src/lib.rs`). Permitted golden diffs: the `lute`/`irVersion`
envelope line, plus (F1) goldens whose sources carried `persist="run"`
(attribute deleted, records unchanged) and (F2) any golden whose shot
addresses change under positional numbering — each re-accepted in the same
commit as its feature.
