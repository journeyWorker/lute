# Lute Scenario DSL — Language Proposal

- **Version:** 0.1.0
- **Status:** Draft (supersedes 0.0.1)
- **Date:** 2026-07-06
- **Scope:** the authoring **language** only — lexical structure, grammar, and normative
  semantics of a `.lute` scenario document.
- **Out of scope:** implementation architecture (AST shapes, compiler passes, auto-injection,
  validation core / LSP, runtime engine) — see [`../../architecture.md`](../../architecture.md);
  and the capability/plugin system (manifest schemas, profiles, providers, bridge) — see
  [`../plugin-system/0.0.1.md`](../plugin-system/0.0.1.md).
- **Inline examples are authoritative.** Every example in this document is 0.1.0 syntax. The
  checked-in `docs/examples/*.lute` remain on the frozen 0.0.1 form until the implementation pass
  migrates them; they are **not** normative for 0.1.0.

The key words **MUST**, **MUST NOT**, **SHOULD**, **MAY** are to be interpreted as in RFC 2119.

---

## 1. Abstract

The Lute Scenario DSL is a line-oriented language for authoring visual-novel episodes. A document
interleaves three syntactic layers — **content** (dialogue + narration), **staging** (single-line
directives), and **logic** (nesting blocks: branching, matching, revisit hubs, and a multi-track
timeline) — and embeds [CEL](https://cel.dev) expressions for conditions. The language is
**total**: every construct MUST reduce to a finite, ordered sequence of command records plus CEL
expression strings. It is deliberately **not** Turing-complete.

## 2. Status & versioning

This is `0.1.0`, the first **breaking** revision of the language, spec-first (the documents are
revised before any implementation). Versions are `MAJOR.MINOR.PATCH`. Pre-`1.0.0` minor versions
MAY make breaking changes; `0.1.0` exercises that allowance relative to `0.0.1`.

`0.0.1` (the pre-implementation draft) is **frozen**: its file is retained unchanged as the prior
record. `0.1.0` replaces it wholesale as the language of record — this document is self-contained
and does not require reading `0.0.1`. The breaking deltas are catalogued in Appendix D (Migration).

A document MAY declare the language version it targets via `meta.luteVersion` (§6.1); absent that,
an implementation assumes the latest it supports.

## 3. Design invariants (normative)

1. **Total, not Turing-complete.** No construct introduces unbounded iteration, recursion, or
   user-defined functions with bodies. All conditions are CEL, which is terminating and
   side-effect-free.
2. **Reduces to data.** Every document MUST be reducible at build time to a flat, ordered
   command-record sequence + CEL strings. A construct that cannot be so reduced is **not** part of
   this language.
3. **Layer legibility.** The three layers are distinguished by leading syntax so a reader can
   classify any line at a glance (§4.3).

**Hubs and totality.** The revisit hub (§7.3, §11.1.3) re-presents its choices at runtime, which
might appear to introduce a loop. It does not violate the invariants: a hub **reduces at build
time to finitely many records** (one option table plus its arms), and its runtime re-presentation
merely **consumes player input each cycle** — it introduces no unbounded *computation*, no
author-written iteration, and no divert back to an earlier position. The "reduces to data"
invariant (§3.2) is therefore preserved; backward diverts, which would break it, remain a
non-goal (§14).

## 4. Lexical structure

### 4.1 Lines, whitespace, blank lines

A document is a sequence of lines separated by U+000A. Most productions are **line-anchored**:
they match from the first non-whitespace character of a line to its end. Leading whitespace on a
line is **insignificant** (the language is not indentation-scoped). A **blank line** (empty or
whitespace-only) is **trivia**: it MUST be ignored.

### 4.2 Comments are trivia

Two comment forms exist. Both are trivia — never nodes — recognized during line scanning under
the ordering rules at the end of this section.

- **Block comment** `/* … */`: starts at `/*` and ends at the next `*/`. It MAY appear standalone,
  inline, trailing, or span multiple lines. Block comments do **not** nest; an unterminated block
  comment is a parse error. A `/*` opened outside content `Text` continues to the next `*/`
  **even across line boundaries**.
- **Line comment** `//`: recognized only at the **start** of a body line, after any leading
  whitespace, and runs to end of line. A `//` that is not line-leading (i.e. appears mid-line) is
  **not** a comment; it is ordinary content. Line comments never span lines.

Comment recognition is subject to two exclusions:

1. Comments are not recognized inside quoted `String` / `CelString` values (§4.4): a `/*`, `*/`,
   or `//` inside quotes is content.
2. Comments are **never** recognized inside a content line's `Text` — the region after the second
   colon is opaque to end of line (§4.4, §7.1). A `//` or `/*` there is literal text (the `Text`
   is truly opaque), not a comment.

**Ordering (normative).** While a block comment is open (an unmatched `/*` is in effect), no line
classification, structural scanning, or `//` recognition occurs until the matching `*/`; this is
why a `/*` opened outside `Text` swallows subsequent lines. Otherwise, on each body line, a
line-leading `//` (after whitespace) makes the whole line trivia, and a `/*` begins a block
comment as above. The content-line `Text` exclusion binds tighter than either form: once the
classifier has consumed a content line's `:` `Speaker` `Attrs?` `:` prefix, the remaining `Text`
is taken verbatim and no comment is recognized in it. Frontmatter (§6.1) is peeled first and uses
YAML's own comment rules, not this body-comment syntax.

### 4.3 Line classification (normative precedence)

The leading `---` **frontmatter** envelope (§6.1) is recognized first and is not a body line; it is
peeled (delimiter-to-delimiter) before body comment stripping and classification begin. After
comments and leading whitespace are stripped, a non-blank **body** line is classified by the
**first matching** rule in this order (most specific first; an **unmatched body line is an error**):

1. `## ` — shot heading (§6.3)
2. `# ` — document title (§6.2)
3. `::set{` — assignment directive (§7.3.3)
4. `::` *ident* — staging directive (§7.2)
5. `:` *ident* — content line: dialogue / narration / player by speaker (§7.1)
6. `<` *ident* / `</` — logic / timeline block open or close (§7.3, §7.4)
7. otherwise — an **unrecognized line is a static error** (there is no prose fallback)

This order is normative: e.g. `::set{` MUST be tried before the generic `::`-directive rule, and
the `::`-directive rule MUST be tried before the `:`-content rule. (There is **no** `:::` construct
in the body — `meta` is frontmatter (§6.1) and the former `:::route` is the `<branch>`/`<match>`
nesting (§7.3); `:::` is not part of the grammar. The former `:line[…]` content form is removed —
see Appendix D.)

### 4.4 Terminals

```
Ident     ::= [A-Za-z] [A-Za-z0-9_-]*          (* directive/attr/speaker/choice/branch/hub/asset ids — '-' permitted *)
CelIdent  ::= [A-Za-z] [A-Za-z0-9_]*           (* CEL-facing: state-path segment, def name, def param — '-' FORBIDDEN (E-PATH-IDENT) *)
Integer   ::= [0-9]+
Number    ::= "-"? [0-9]+ ( "." [0-9]+ )?
Bool      ::= "true" | "false"
Speaker   ::= Ident                            (* a character id; incl. reserved narration / player speakers, §7.1 *)
String     ::= '"' ( <any char except '"' or '\'> | Escape )* '"'
Escape    ::= '\"' | "\\" | "\n" | "\t"        (* any other backslash escape is E-STRING-ESCAPE *)
Path      ::= ("scene" | "run" | "user" | "app") ("." CelIdent)+   (* a declared state path, §9 *)
Text      ::= <the rest of the line, verbatim, to EOL, except {{…}} interpolations (§7.6)>
Ref       ::= "@" CelIdent ( "(" CelArgs ")" )?
CelString ::= '"' <a CEL expression> '"'
Interp    ::= "{{" WS? ( Path | Ref | ReservedToken ) WS? "}}"     (* §7.6 *)
ReservedToken ::= "userName"
```

- **`String`** is double-quoted with backslash escaping; it MUST NOT contain a raw newline. The
  **only** defined escapes are `\"`, `\\`, `\n`, `\t`; any other backslash escape (e.g. `\d`) is a
  static error **`E-STRING-ESCAPE`** (S7).
- **`CelString`** is a double-quoted **CEL expression** (interpreted per the Lute-CEL profile, §8.4).
  CEL's own string literals inside it use single quotes (`'blunt'`) or escaped double quotes; thus
  `test="$ == 'blunt'"` is well-formed. A `Ref` (`@name` / `@name(args)`) MAY appear inside a
  `CelString` and is expanded to inline CEL before evaluation (§8.1, §8.4). The distinguished token
  `$` MAY appear only inside a `<match>` (§8.2).
- **`Text`** is opaque to end of line: once a content-line body begins (after the second colon),
  the remainder of the line is literal (so `(`, `?`, `<`, `:`, quotes, `//`, `/*` are not special
  there). The **sole** exception is an interpolation `{{…}}` and its escape `\{{` (§7.6). Empty
  `Text` is permitted (e.g. `@fixer{code="0010"}: `).
- **CEL-facing identifiers forbid `-` (S2).** State-path segments (`CelIdent` in `Path`), `defs`
  names, and def parameter names MUST NOT contain `-` — a `-` there is **`E-PATH-IDENT`** (§8.4).
  All other identifier positions — directive names, attribute keys, speaker ids, choice/branch/hub
  ids, and asset ids — keep permitting `-` via `Ident`.
- **Quoted-string boundaries are respected before structural scanning.** A lexer MUST tokenize
  `String` / `CelString` quotes before classifying lines or scanning tag/attr structure, so
  structural-looking characters inside a quoted value (`<`, `{`, `}`, `:` — e.g. `test="$ < 3"` or
  a CEL map literal) are content, not delimiters.

### 4.5 Attributes

```
Attrs ::= "{" ( Attr ( WS Attr )* )? "}"
Attr  ::= Ident "=" String | Ident "=" Ref | Ident      (* bare Ident ⇒ boolean true *)
```

Attribute order is **not significant**. All attribute **values** are a `String` or a `Ref`; there
is **no** `{ expression }` interpolation of attribute values (preserving invariant §3.1).

**Coercion grammar (S7).** An implementation coerces a `String` value to the attribute's declared
type:

- declared type **number** — the (whitespace-trimmed) `String` content MUST match `Number`
  (`-?[0-9]+("."[0-9]+)?`); e.g. `zoom="1.5"`, `move-x="-40"`.
- declared type **bool** — the content MUST be `Bool` (`true` | `false`); a **bare** `Ident`
  attribute with no `=` is boolean `true`.
- declared type **string / enum** — the content is used verbatim (after escape resolution, §4.4).

A value that does not match its declared type's coercion grammar is a static coercion error.

## 5. Layers (overview)

| Layer | Syntax | Members |
|---|---|---|
| **Content** | `@speaker{…}: text` — dialogue / narration / player by speaker (delivery selects role, §7.1, §12) | §7.1 |
| **Staging** (leaf) | `::name{…}` | §7.2, Appendix A |
| **Logic / timeline** (nesting) | `<tag>…</tag>` — branch, match, hub, timeline | §7.3, §7.4 |

**Bracket rule (normative):** a construct with children MUST use the `<tag>…</tag>` form (and MUST
be explicitly closed); a single-line construct MUST use a `::`-directive or `@`-content form.

## 6. Document structure

```
Document  ::= Meta Title? ShotBlock*
```

Trivia (blank lines, comments) MAY appear between any of these.

### 6.1 `meta` (frontmatter)

```
Meta      ::= "---" NEWLINE YamlBody "---" NEWLINE
```

`Meta` is **YAML frontmatter** and MUST be the document's first construct (it precedes all body
trivia and nodes). `YamlBody` is an **opaque YAML block** (the implementation parses it as YAML).
This is the repo-wide `---` frontmatter convention; there is no `:::`-fenced meta form. `meta` MUST
define `character`, `season`, `episode`; it MAY define `episodeId`, `mode`, `pov`, `title`,
`luteVersion`, `contentLang`, `profile`, `plugins`, `uses`, and inline `state` (§9) and `defs`
(§8.1) blocks.

- `episodeId` is a **stable opaque episode id** and the episode-prefix input to every `lineId`
  derivation (§12). If omitted, it defaults to `s{season:02}ep{episode:02}` (each of `season`,
  `episode` zero-padded to two digits — e.g. `season: 1, episode: 2` ⇒ `s01ep02`). Pinning
  `episodeId` explicitly lets an author **renumber** `season`/`episode` without breaking existing
  translation / VO keys (S3): the derived `lineId`s stay stable because they key on `episodeId`,
  not on the raw numbers.
- `luteVersion` is the **DSL language-version** pin (§2) — distinct from the `app.lang` UI-language
  *state* (§9). They coexist; one is a version target, the other is game state.
- `profile` selects the root-level capability profile used to resolve active plugins (§6.1.1).
- `plugins` MAY add scene-local plugin activations/options on top of the selected profile (§6.1.1).
- `uses` imports the shared state schema (§9.2). Most scenes carry only `character`/`season`/
  `episode` + `profile` + `uses` (and at most a few `scene.*` declarations); the `run`/`user`/`app`
  schema and shared `defs` live in the imported schema document, not per scene.
- `pov` names the **player/protagonist speaker id**: the content-line speaker whose id equals
  `meta.pov` is the reserved **player** kind (renders the runtime `{{userName}}`, no sprite; §7.1).
  It is project-defined (Idola uses `fixer`).
- `mode` is the authoring mode; `inline` is the only form defined in 0.1.0.
- `contentLang` is the document's **source authoring language** — a language code from the project's
  declared language set (BCP 47, e.g. `en-US` / `ko-KR`). It declares which source language the
  authored text is in (the translation pass reads it, §12). Distinct from `luteVersion` (the
  DSL-version pin) and `app.lang` (UI-language *state*).
- **Plugin-contributed keys:** an active plugin (§6.1.1) MAY declare additional top-level meta keys
  via its plugin-system `frontmatter` export, validated against that plugin's schema. A top-level
  meta key that is neither a core key above nor owned by an active plugin is a static error.
  (Example: `cast:` is owned by the character/cast capability.)

#### 6.1.1 Capability profile activation

`profile` is a root-level scene capability selector, not a plugin-local field. If absent, the
project's `defaultProfile` applies. The reserved profile name `global` is inherited by every other
profile before that profile's own settings are applied.

`plugins` is a map from plugin id to a typed option object. Presence of a plugin id activates that
plugin for this scene; an empty option object means active with default options. The shorthand
`true` MAY be accepted and normalized to an empty option object. A `plugins` entry whose value is
neither a map nor `true` (including `false`) is a static error (`E-PROFILE-PLUGIN-VALUE`), never a
silent activation. There is no `plugins.use` list in 0.1.0, and 0.1.0 does not define scene-local
plugin disabling.

```yaml
profile: date-minigame
plugins:
  idola.minigame:
    resultScope: scene
    allowedKinds: [rhythm, timing]
```

Effective capabilities resolve in the deterministic order specified normatively in plugin-system
§11.1 (`lute.core` → `profiles.global` → the selected profile's `extends` chain, parent first →
selected profile → scene-local `plugins` → dependency closure). Plugin activation is additive. The
checker, LSP, and compiler MUST validate the document against the same resolved capability snapshot.

### 6.2 Title

```
Title ::= "# " Text
```

An optional document-level H1 title MAY appear after `Meta` and **before the first shot**; at most
one is allowed. A `# ` line after the first shot, or a second `# ` title, is **`E-TITLE-PLACEMENT`**
— the §4.3 classifier still recognizes the `# ` shape, but placement/cardinality is enforced here,
not by falling through to `E-UNCLASSIFIED`.

### 6.3 Shots

```
ShotBlock   ::= ShotHeading Node*
ShotHeading ::= "## " ( "Shot" | "Scene" ) WS Integer "." ( WS Text )?
              | "## " ( "Prologue" | "Epilogue" | "프롤로그" | "에필로그" ) ( WS Text )?
```

A shot heading opens a shot. Shots are flat (not nested), ordered, and are the document's
sequence boundaries. Shot integers SHOULD be ascending; gaps are permitted. `Shot` and `Scene` are
equivalent numbered-heading keywords; `Prologue`/`Epilogue` (and their `프롤로그`/`에필로그`
equivalents) are the unnumbered book-end keywords.

The `ShotHeading` production is **ENFORCED** (S9): a `## ` line that does not match either
alternative — a missing integer, a missing `.` after the integer, or an unrecognized keyword — is
a static error **`E-SHOT-HEADING`** (it is not silently treated as prose or as an H2 title).

## 7. Grammar (nodes)

```
Node ::= Line | Directive | Set | Branch | Match | Hub | Timeline
```

(Comments and blank lines are trivia, §4.1–4.2, not nodes.)

### 7.1 Content

```
Line ::= "@" Speaker Attrs? ":" WS Text
```

A content line is `@speaker{…}: text` — e.g. `@bianca{code="0010" emotion="delighted"}: Mr. Fixer!`
or `@narrator: Venny's again.`. The **speaker** selects the kind: a registered character →
dialogue; the reserved `narrator` → narration (speakerless); the speaker whose id equals `meta.pov`
→ the reserved **player** (protagonist). There is **no** separate monologue or prose node — role is
derived from `speaker` + `delivery` per the table in §12.

`Line` attributes are content metadata (e.g. `code`, `emotion`, `variant`, `action`, `dialogMotion`,
`delivery` = `spoken`\|`thought`\|`voiceover`, and `as` — a one-off speaker-label override);
domains are vocabulary, not grammar (Appendix A). When `as` is absent the shown label is resolved
by the active display-name capability (e.g. character/cast). `delivery="thought"` is legal on any
character (that character's inner voice — a monologue), not only the player; `delivery` on
`narrator` is a static error (§12). The `Text` (and a `<choice label>` string, §7.3) MAY contain
`{{…}}` interpolations (§7.6).

```lute
@bianca{code="0010" emotion="delighted"}: Mr. Fixer!
@narrator: Venny's again — the coffee shop on the corner.
@fixer{delivery="thought"}: I should tread carefully here.
@bianca{as="???"}: ...who's there?
```

Absent any active display-name / registry capability, a content line still renders its speaker id
as the label (dialogue); `narrator` and the `pov` player remain reserved. The character/cast
capability adds registry validation, costume, and name-reveal.

**Migration fix-it (non-normative).** An implementation encountering a leading `:line[` SHOULD
detect the removed 0.0.1 form and emit a fix-it rewriting `:line[speaker]{…}: text` to
`@speaker{…}: text` — the current sigil (0.2.2+, Appendix D).

### 7.2 Staging directive

```
Directive ::= "::" Ident Attrs?
```

A single-line leaf. The `Ident` and attribute meanings are **vocabulary** (Appendix A), extensible
without grammar change. Staging timing attributes (§7.5) MAY appear on staging directives; `at` is
valid only for timeline clips (§7.5, §11.4).

### 7.3 Logic blocks

```
Branch    ::= "<branch" Attrs ">" Choice+ "</branch>"           (* requires id; ≥1 unguarded choice *)
Choice    ::= "<choice" Attrs ">" Node* "</choice>"             (* requires id, label; optional when, persist/into/value *)
Match     ::= "<match" Attrs ">" When+ Otherwise? "</match>"    (* requires on *)
When      ::= "<when" Attrs ">" Node* "</when>"                 (* requires is and/or test *)
Otherwise ::= "<otherwise>" Node* "</otherwise>"                (* NO attrs *)
Hub       ::= "<hub" Attrs ">" HubChoice+ "</hub>"              (* requires id *)
HubChoice ::= "<choice" Attrs ">" Node* "</choice>"             (* requires id, label; flags once, exit; optional when, persist/into/value *)
```

Tag attributes use the §4.5 `Attrs` form (order-independent). Required / permitted attributes per
tag:

- `branch` → `id`. A `<branch>` MUST contain **at least one unguarded** (`when`-less) `<choice>`;
  a branch whose every choice carries a `when` is **`E-BRANCH-ALL-GUARDED`** (S5) — a branch could
  otherwise present an empty menu.
- `choice` (inside `<branch>`) → `id`, `label` (a `String`, MAY interpolate §7.6); optional `when`
  (a `CelString`); optional run-promotion sugar `persist`, `into`, `value` (§11.1). The persist
  target attribute is `into` (**renamed from 0.0.1 `as`**); `as` on a `<choice>` is no longer
  accepted (it survives only on content lines as the display-label override, §7.1).
- `match` → `on` (a `CelString`).
- `when` → `is` and/or `test` (see below); **neither present is `E-WHEN-PATTERN`** (D-D).
- `otherwise` → **no attributes**; any attribute on `<otherwise>` is a parse error (S10).
- `hub` → `id`. `hub` choices are `HubChoice` (below).

Missing/unknown required attributes are a static error.

#### 7.3.1 `<when>` literal patterns (`is`)

A `<when>` arm matches on a literal pattern (`is`), a CEL guard (`test`), or both.

```
WhenPattern ::= Literal ( WS? "|" WS? Literal )*
Literal     ::= EnumMember | "true" | "false" | Number | "unset"
EnumMember  ::= Ident                     (* enum member id; not CEL-facing, MAY contain '-' (§4.4) *)
```

- **`is`** value is a `String` whose content is a `WhenPattern`: one literal, or a `|`-alternation
  of literals. Legal literals are enum member idents, `true`/`false`, decimal `Number`s, and the
  keyword `unset`. Matching is **equality on the match subject** (§8.2); `unset` matches the unset
  state of a nullable subject (§9.4). Example: `<when is="delighted">`, `<when is="blunt | curt">`,
  `<when is="unset">`, `<when is="1 | 2 | 3">`.
- **`test`** is a `CelString` guard (the `$` subject, §8.2, is in scope).
- **`is` + `test` together** = pattern **AND** guard: the arm fires only when the subject matches
  `is` *and* the guard holds (the ts-pattern `P.when` composition). Example:
  `<when is="delighted" test="$ != 'staged'">`.
- A `<when>` with **neither** `is` nor `test` is **`E-WHEN-PATTERN`** (D-D).

Exhaustiveness is computed from `is` arms (§11.2).

#### 7.3.2 `<hub>` revisit conversations

A `<hub id>` is a revisit conversation: on entry, and after each non-`exit` arm completes, it
**re-presents** all currently eligible choices, letting the player weave through them in any order.

- `HubChoice` requires `id`, `label`; MAY carry the boolean flags `once` and `exit`, an optional
  `when` guard, and the persist sugar `persist`/`into`/`value` (§11.1).
- **Eligibility.** A choice is eligible at a presentation point iff its `when` (if any) is true
  **and** (if flagged `once`) it has not yet been taken **in this scene** (i.e.
  `scene.visited.<hub>.<choice>` is still `false`; that flag persists scene-wide, §11.1.3, so a
  `once` topic stays taken across hub re-entries within the scene and resets only at the
  engine-owned scene boundary).
- **`once` governs re-selectability only, not "seen" marking.** *Every* selection records
  `scene.visited.<hub>.<choice> = true` (§11.1.3) regardless of `once`, so the engine MAY grey or
  check an already-seen choice from that flag alone. A choice **without** `once` stays eligible and
  MAY be re-taken (e.g. a player replaying a line they clicked past); `once` merely *additionally*
  removes the choice from the eligible set after its first take (and, when all choices are `once`,
  guarantees termination — below).
- **Re-presentation & auto-exit.** After a non-`exit` arm runs, the hub re-evaluates eligibility
  and presents the eligible set again. Taking an **`exit`**-flagged choice runs its arm and then
  **leaves** the hub. If **no** choice is eligible at a presentation point, the hub ends
  (**auto-exit**). An `exit` choice MAY also be `once` (legal; `once` is simply irrelevant once the
  arm exits).
- **Static guarantee `E-HUB-NO-EXIT`** (D-C): a hub MUST have at least one **unguarded** (no
  `when`) `exit` choice, **OR** all of its choices MUST be `once` (so the eligible set provably
  empties and auto-exit fires). A hub that satisfies neither is `E-HUB-NO-EXIT`.

```lute
<hub id="chatWithBianca">
  <choice id="askCoffee" label="Ask about the coffee" once>
    @bianca: House blend. Bold, like the clientele.
  </choice>
  <choice id="askVenny" label="Ask about Venny" when="scene.metVenny" once>
    @bianca: He's around here somewhere.
  </choice>
  <choice id="leave" label="Head out" exit>
    @fixer: I'd better get moving.
  </choice>
</hub>
```

The hub's recording model, id-uniqueness domain, and definite-assignment rules are in §11.1.3.

#### 7.3.3 `set`

```
Set      ::= "::set{" Path WS AssignOp WS CelExpr "}"
AssignOp ::= "=" | "+=" | "-=" | "*="
```

Exactly **one** assignment per `::set`. The operator is a token (not a string value); the
right-hand side is a CEL expression (Lute-CEL profile, §8.4). The `Path` MUST be a declared state
path (§9) and MUST use `CelIdent` segments (no `-`, §8.4).

### 7.4 Timeline

```
Timeline ::= "<timeline" Attrs? ">" Track+ "</timeline>"        (* optional duration *)
Track    ::= "<track" Attrs ">" Clip+ "</track>"               (* key: channel | subject | subject+property, §11.4 *)
Clip     ::= Directive | Set
```

A `<timeline>` body MUST contain only `<track>`s (and trivia); a `<track>` body MUST contain only
staging leaves / `::set`. Content lines and logic blocks MUST NOT appear inside a `<timeline>`.
A staging clip MAY carry `at` plus the staging timing attributes (§7.5); a `::set` clip carries no
timing and fires at the resolved end of the previous clip in its track (§11.4). `at` and `delay`
MUST NOT both appear on one clip (**`E-CLIP-TIMING`**, §7.5, §11.4).

### 7.5 Timing attributes

`duration`, `delay`, and `wait` are reserved **staging** timing keys. They MAY appear on staging
directives, including directives used as timeline clips. `at` is a reserved **timeline-position**
key: it MAY appear only on staging directives inside a `<track>`; `at` anywhere else — on a
content-context staging directive, or on any node outside a `<track>` clip — is the dedicated
diagnostic **`E-AT-CONTEXT`** (S8), not an `E-UNKNOWN-ATTR` fallthrough.

`at` and `delay` are mutually exclusive on one clip: an absolute position and a relative nudge on
the same clip is **`E-CLIP-TIMING`** (S6).

These four keys (`at`, `duration`, `delay`, `wait`) are **cross-cutting reserved** across all
directives and profiles. A plugin manifest MUST NOT declare any of these as one of its attribute
names — doing so is an assembly-time error (see plugin-system Errata).

### 7.6 Interpolation `{{…}}`

Content `Text` and `<choice label>` string values MAY embed **interpolations** that read game
state at render time.

**Grammar.**

```
Interp        ::= "{{" WS? ( Path | Ref | ReservedToken ) WS? "}}"
ReservedToken ::= "userName"
```

- **`Path`** (§4.4) — a declared state path read; e.g. `{{run.coins}}`, `{{scene.trust}}`.
- **`Ref`** (§8.1) — `@def` / `@fn(args)`; **defs are allowed** in interpolation. An undeclared ref
  is `E-UNDECLARED-REF`; an undeclared path is `E-UNDECLARED`. A `@fn(args)` argument is CEL — as
  wherever a `@ref` call appears (§8.1) — and is the *only* CEL admitted inside `{{…}}`. An
  interpolated `Ref` MUST resolve to a renderable type (number / bool / enum, per the rendering rule
  below); a `@def` of any other type is a static error.
- **`ReservedToken`** — `userName` renders the runtime player name (the `pov` player, §7.1).

**Reading rules.** An interpolation is a **read** for definite-assignment (§9.4): interpolating a
maybe-unset path without a guard is `E-MAYBE-UNSET`. Only the three forms above are legal inside
`{{…}}` — a bare CEL expression is not; name a computed value with a `@def` (whose body is CEL, and
whose `@fn(args)` call MAY pass CEL arguments as in §8.1).

**Rendering (normative).** The read value renders as: a **number** → its shortest decimal form; a
**bool** → `true` / `false`; an **enum** → the member text verbatim.

**Escapes.** A literal `{{` is written **`\{{`** (rendering a single `{{`). An unterminated `{{`
(no closing `}}` before EOL) is **`E-INTERP-UNTERMINATED`**.

**i18n contract.** Placeholders are part of the line's translatable text keyed by `lineId`; the
source↔translation **placeholder-set equality** check is `E-L10N-PLACEHOLDER` (compile-stage,
§12).

**ICU reserved.** ICU MessageFormat plural/select is an explicitly **reserved** section — a
non-goal for 0.1 (§14). The placeholder-set contract above ships now; richer message formatting
does not.

```lute
@bianca: Nice to finally meet you, {{userName}}!
@narrator: You have {{run.coins}} credits left.
@fixer: Trust stands at {{scene.trust}}.
```

## 8. Expressions

### 8.1 CEL & definitions

All conditions and `::set` right-hand sides are CEL, interpreted under the Lute-CEL profile (§8.4).
`meta.defs` declares named, typed CEL values — and is the namespace of legal `@ref`s:

```yaml
defs:
  <name>: { type: bool|number|enum, params?: { p: <type> }, cel: "<CEL>" }
```

Def `<name>` and each param name are **CEL-facing identifiers**: they MUST use `CelIdent` (no `-`,
`E-PATH-IDENT`, §8.4). A `@name` / `@name(args)` reference is a **compile-time macro**: it MUST be
expanded to its inline CEL before evaluation, with `params` bound from the call arguments
(parenthesized per §8.4). A `@ref` MUST appear in a position whose required type matches the def's
declared `type`, and its `name` MUST be declared in `defs`. `defs` MAY live inline in a scene's
frontmatter or in an imported state schema (§9.2); shared `defs` (those whose CEL reads
`run`/`user`/`app`) belong in the schema document. Duplicate `defs` names across imports are an
error (§9.2).

The `min`/`max`/`values` def fields of 0.0.1 are **removed** (they were dead — never consulted by
the checker or engine; S11); a def is fully described by `type`, optional `params`, and `cel`.

> **NOTE (checker surface).** The static checker owns four `@ref` / `@name(args)`
> checks: the name is declared in `defs` (`E-UNDECLARED-REF`), the call arity
> matches the declared `params` (`E-REF-ARITY`), each statically-resolvable
> positional argument matches its param's declared type (`E-REF-ARG-TYPE`), and a
> whole-slot `@ref` produces the position's required type (`E-REF-TYPE`). Each is
> conservative — only provably-wrong cases flag. Macro **expansion** itself (§8.1
> above + §8.4 hygiene) is engine-owned, not a checker concern.

### 8.2 The `$` subject

Inside a `<match on="S">`, the token `$` within a `<when test>` resolves to the subject expression
`S`. `$` MUST NOT appear outside a `<match>`. When `$` is substituted, a compound subject is
substituted **parenthesized** (§8.4).

### 8.3 No inline code

Conditions and values are CEL only. There is no host-language (JS, Lua, …) evaluation anywhere in
the language.

### 8.4 Lute-CEL profile (normative)

Lute embeds a **restricted CEL profile**. This section is the single source of truth for what CEL
means in a `.lute` document.

**Numeric model.** A state `type: number` is an IEEE-754 **double** (the compile target is JSON,
which has one numeric type). In every Lute CEL slot each numeric literal denotes a double (integer
literals promote to double). Arithmetic is `+`, `-`, `*`, `/` only. The modulo operator `%` is
**not** available in 0.1 (there is no integer domain to define it over); neither is `/=` an
assignment operator (§7.3.3).

**Environment (closed).** The Lute-CEL environment provides exactly: CEL operators, literals,
list literals, and the ternary conditional; the `in` membership operator; the `has()` macro; and
the single Lute extension `isSet(<path>)` (true iff the state path is assigned). **No other
functions** are available in 0.1 — the environment is closed, and any other function or macro call
(a comprehension `map`/`filter`/`exists`/`all`, `size`, `matches`, …) is a static error
(**`E-CEL-PROFILE`**). A `@name(args)` def reference is a compile-time macro (§8.1), not a runtime
call, and is exempt.

**Hygiene (normative).** `@ref` expansion and `$` substitution are **parenthesized, AST-safe**
substitutions:

- an expanded def body MUST be wrapped in `(…)` at its insertion site;
- every bound `@name(args)` argument MUST be wrapped in `(…)`;
- a **compound** `$` subject MUST be substituted parenthesized; a bare dotted path (e.g.
  `scene.trust`) MAY be substituted verbatim.

This guarantees operator-precedence safety after expansion and codifies the shipped
`lute-compile/src/expand.rs` behavior.

**Identifier alignment (S2).** `-` is **FORBIDDEN** in CEL-facing names — state-path segments,
`defs` names, and def parameter names — because CEL parses `-` as subtraction; a `-` in any of
these is **`E-PATH-IDENT`**. Directive names, attribute keys, speaker ids, choice/branch/hub ids,
and asset ids are **not** CEL-facing and keep permitting `-` (§4.4).

## 9. State

### 9.1 Namespaces (lifetime tiers)

Four namespaces, each named by its **reset boundary** (when it clears):

| Namespace | Reset boundary | Example |
|---|---|---|
| `scene.*` | **episode end** — one `.lute` document; **survives across its shots** (§6.3), cleared when the episode ends | on-stage state, `scene.choices.*`, `scene.visited.*` |
| `run.*`   | new run — one attempt (a sequence of episodes); cleared on restart | per-attempt flags, affect, cross-**episode** carry *within* an attempt |
| `user.*`  | profile/account wipe — survives runs | level, unlocks, meta-progression |
| `app.*`   | app uninstall — identity-independent | language, age rating, settings |

> **Naming note:** the `scene.*` tier's lifetime is the **episode** (the whole document). The
> `## Scene N.` / `## Shot N.` headings (§6.3) are *shots within* that document, not reset
> boundaries — `scene.*` (incl. `scene.choices.*`, `scene.visited.*`) persists across shots and
> clears only at episode end. Carrying a value to the *next episode* in the same attempt requires
> `run.*`.

The four are points on one axis (when it resets). Every path read **or written** (including by
`::set` and by branch/hub recording, §11.1) MUST be declared (`type` + optional `default`) in the
state schema; there are **no** bare, un-namespaced state names. Each namespace maps to an engine
persistence backend; the engine **owns and fires** every reset boundary — the language never
triggers a reset.

### 9.2 Schema source & composition (`uses` / `extends`)

The `run`/`user`/`app` schema is **game/season-global** (one persisted value cannot have per-scene
types), so it lives in a single source-of-truth schema document, imported by each scene's
frontmatter via `uses:`. Only genuinely scene-local declarations (`scene.*`) MAY appear inline in a
scene, and a scene MUST NOT redeclare or override an imported `run`/`user`/`app` path. Imports form
a **DAG**:

- cycles are a static error (the diagnostic prints the chain);
- imported schemas are loaded/checked **before** any scene;
- duplicate `defs` names across imports are an error (no silent shadow);
- import refs are canonicalized, so two paths to the same file are **one** schema identity.

(The state schema is *game content*; it is distinct from the engine **capability manifest**, which
is *engine vocabulary* — separate owner, separate change cadence.)

**`extends:` — base composition with override.** Where `uses:` unions PEER schemas (a name
declared by two peers is an error), `extends:` names one or more BASE schemas that this document
*refines*. A base is a **lower-precedence layer**: if the extending document (or its `uses:` peers)
declares a name also declared by a base, the extending declaration **overrides** the base's — no
duplicate error. Precedence, low → high: a document's `extends` bases (recursively; a base's own
`extends` are even lower) < its `uses` peers < its own inline `state:`/`defs:`. Among multiple
bases, one declared closer to the extending document overrides one further away; two *unrelated*
bases declaring the same name is a cross-base conflict — the same error as a peer duplicate.

Override rules on a base collision: a `defs` entry is replaced wholesale (defs are pure CEL
macros). A `state` entry is likewise overridden, but because persisted state must keep a stable
type, an override that changes the declared **type** is a static error (`E-EXTENDS-STATE-TYPE`); a
`default`-only refinement (same type) is allowed silently, and the closer declaration wins in the
merged schema either way. `extends` edges share the DAG discipline of `uses:` — cycles, missing
files, and parse errors on a base reuse the `E-USES-{CYCLE,NOT-FOUND,PARSE}` diagnostics, and a
file reached by two paths is one schema identity.

### 9.3 Declaration form

```yaml
state:
  scene.<path>: { type: number|bool|enum, default?: <literal> }
  run.<path>:   { type: …, default?: … }
  user.<path>:  { type: …, default?: … }
  app.<path>:   { type: …, default?: … }   # read-only to content (§9.5)
```

Each `<path>` segment is a `CelIdent` (no `-`, §8.4). A `default` is materialized into the initial
tier state at schema load **and** re-materialized whenever the engine fires that tier's reset
boundary (a cleared tier reloads its default; checker and engine read the one schema snapshot and
cannot disagree).

### 9.4 Definite assignment (path-sensitive)

Reading an undeclared path is a static error (`E-UNDECLARED`) — even when the tier exists but the
sub-path does not. For assignment:

- A `scene.*` read follows ordinary path-sensitive analysis (a read not provably assigned on the
  current path is an error unless guarded, e.g. `has()`/`isSet()`).
- A `run.*` / `user.*` / `app.*` path is **maybe-unset at scene entry** unless it carries a schema
  `default`. After entry, a dominating `::set{p = …}` write or an enclosing guard proves it; a read
  with no default, no dominating write, and no guard is `E-MAYBE-UNSET`.
- A compound assignment (`+=` / `-=` / `*=`) **reads** the old value, so it is itself subject to the
  above; only `=` (a pure write) may be the first assignment on a path.
- An interpolation read (`{{path}}`, §7.6) is a **read** for this analysis — a maybe-unset path
  interpolated without a guard is `E-MAYBE-UNSET`.

`E-UNDECLARED` and `E-MAYBE-UNSET` are distinguishable diagnostics.

### 9.5 Write policy

`::set` MAY target `scene.*` / `run.*` / `user.*`. `app.*` is **read-only to content**:
`::set{app.*}` is a static error (the engine/settings layer owns `app.*` writes; content reads
`app.*` via `<match>`/`<when>` for gating — see §11.2).

### 9.6 Reserved namespaces

Two `scene.*` sub-namespaces are **reserved, engine-populated, and implicitly declared** — content
never declares them and MUST NOT write them directly (recording happens through branch/hub
selection, §11.1):

- `scene.choices.<branchOrHubId>` — the last selected choice id for a `<branch>` or `<hub>`; its
  domain is that construct's choice ids ∪ `unset`. Content MAY read it in a `<match>`/`<when>`.
- `scene.visited.<hubId>.<choiceId>: bool` (default `false`) — a per-hub-choice "taken" flag,
  populated on selection (§11.1.3). This is a **NEW** reserved implicit namespace, kept **separate
  from `scene.choices.*`** so a hub id can be both a leaf (`scene.choices.<hubId>`) and a parent
  (`scene.visited.<hubId>.*`) without a path parent/leaf collision. Content MAY read it in a
  `<match>`/`<when>` for gating.

`run.choiceLog.*` is a further **reserved, engine-populated** sub-namespace (an analytics/debug
trace of selected choices, §11.1.2). Unlike the two above, content **MUST NOT** read it in a
`<match>`/`<when>`/guard (a build-failing static **error**, `E-CHOICELOG-READ` — not a warning). All three are exempt from the author-declaration
rule (§9.4) — the engine owns them, as with `app.*`. Authors declare and branch on **named** `run.*`
facts, never the raw log.

## 10. Reserved / collision rules

New tag and directive names MUST NOT collide with existing domain terms. In particular `cut` and
`scene` are reserved (the `::cut` directive; `## Scene` headings; scene ids), which is why the
timeline block is `<timeline>` and its tracks `<track>` (not `<cutscene>` / `<lane>`). The reserved
block tags are `branch`, `choice`, `match`, `when`, `otherwise`, **`hub`**, `timeline`, `track`;
the reserved built-in directives include `set` and `use` (§13.2). The timing attribute keys in §7.5
(`at`, `duration`, `delay`, `wait`) are reserved across all directives and profiles.

## 11. Semantics (normative)

### 11.1 Branch
`<branch id>` MUST be unique **within the episode** (the `.lute` document) — not across the run.
Presenting it offers each `<choice>` whose `when` (if any) is true; a branch MUST have at least one
unguarded choice so the menu is never provably empty (`E-BRANCH-ALL-GUARDED`, §7.3). Selecting a
choice records `scene.choices.<branchId> = <choiceId>` (implicitly declared with domain = the
branch's choice ids ∪ `unset`, and **episode-scoped** per §9.1). `label` is the player-facing text
(MAY interpolate, §7.6); `id` is the recorded key.

Within a `<branch>`, each `<choice id>` MUST be **unique** — duplicate choice ids are a static
error (`E-CHOICE-DUP`). Both the recorded value's domain (`scene.choices.<branchId>`) and the
choice-label `lineId` (§12) key on it.

#### 11.1.1 Cross-episode carry (named run facts, not raw choice keys)
`scene.choices.*` clears at episode end, so it serves **intra-episode** reactions only (a later
shot's `<match on="scene.choices.x">`). To make a choice affect a **later episode** of the same
run, write a **named, declared** `run.*` fact — cross-episode state is a *domain outcome*, never a
remembered branch key. Two forms:

- **Explicit (primitive):** `::set{run.<path> = <value>}` in the choice arm (§7.3.3).
- **Sugar:** `<choice … persist="run" into="<run.path>" [value="<lit>"]>` desugars to a
  `::set{run.<path> = <value>}` appended to that arm (the persist target attribute is **`into`**,
  renamed from 0.0.1 `as`). `value` defaults to `true` for a `bool` path; for an `enum`/`number`
  path it is required. The sugar is **exactly** the explicit form — no extra power.

```lute
<choice id="couch" label="Offer the couch" persist="run" into="run.metHelpfully" value="warm">
  @fixer: Take the couch — you look exhausted.
</choice>
```

**Why not record raw choice keys into `run`.** Promoting `<branch id>` into `run.choices.<id>`
would force `<branch id>` to be **globally unique across all episodes in a run** and couple a later
episode to an earlier one's *menu-arm naming* — an implementation detail, not authored consequence.
Named run facts avoid both: ids stay episode-local, and a later episode reads a meaningful fact
(`run.metHelpfully`, `run.sofaHelpOutcome`), not "which arm ep02 happened to call `couch`."

**Schema rule (no implicit declaration).** `into="<run.path>"` MUST resolve to a path **already
declared** in the imported run schema (§9.2); an undeclared/typo'd `into` is `E-UNDECLARED`
(state-by-typo MUST fail, never silently create a field). The compiler further checks: the path is
writable from this episode/choice context (§9.5); `value` is type-compatible with the declared
path; and no two writes in the same arm conflict on one path.

**Implemented (checker).** `lute check` validates the sugar's well-formedness (the engine
materializes the `::set`): `persist` must be `"run"` (`E-PERSIST-TARGET`); `into` is required
(`E-PERSIST-MISSING-INTO`) and must name a `run.*` path (`E-PERSIST-TARGET`) that is already
declared (`E-UNDECLARED`); `value` is optional for a `bool` path (defaults to `true`) but required
and type-compatible for a `number`/`enum` path (`E-PERSIST-VALUE`); and the arm must not already
`::set` the same path (`E-PERSIST-CONFLICT`).

#### 11.1.2 `run.choiceLog` (reserved analytics trace)
The engine MAY maintain a reserved, engine-populated trace `run.choiceLog.<episode>.<branchId> =
<choiceId>` for analytics/debug. It is **not** author-declared (§9.6) and content **MUST NOT**
branch on it — a `<match>`/`<when>`/guard that reads `run.choiceLog.*` is a build-failing static **error** (`E-CHOICELOG-READ`).
Cross-episode branching reads named `run.*` facts (§11.1.1), never the raw log.

#### 11.1.3 Hub
A `<hub id>` (§7.3.2) presents its eligible choices repeatedly until an `exit` arm is taken or no
choice is eligible (auto-exit). Its **runtime** re-presentation adds no unbounded computation
(§3, hub totality); at build time a hub reduces to one finite option table plus its arms.

**Id-uniqueness domain.** Hub ids and branch ids **share one per-episode uniqueness domain** — both
record under `scene.choices.*`, so a hub id may not collide with a branch id (or another hub id) in
the same episode. Choice ids are unique **within** a hub (`E-CHOICE-DUP`, reused).

**Recording model (implicit, episode-scoped, §9.6).** Selecting a hub choice records:

- `scene.choices.<hubId> = <choiceId>` — the *last* selected choice id (domain = the hub's choice
  ids ∪ `unset`);
- `scene.visited.<hubId>.<choiceId> = true` — the per-choice "taken" flag (default `false`),
  recorded on **every** selection (independent of `once`). It is both the "already-seen" signal the
  engine MAY surface (grey/check) and the driver of `once`-eligibility; content MAY also read it in a
  `<match>`/`<when>` for gating. It persists scene-wide (resets only at the engine-owned scene
  boundary), so `once` means "not re-takable **within the scene**".

**Definite assignment.** Hub arms have **no dominance relation** among one another (the same join
rule as `<match>` arms): a write inside one arm is a **may-write** at hub exit, not a definite
assignment. A path first written only inside hub arms is maybe-unset after the hub unless every
arm writes it or it carries a schema `default` (§9.4).

**Coverage.** Hubs are covered **dynamically** by auto-exit — there is no exhaustiveness obligation
on a hub (unlike `<branch>`, which needs an unguarded choice, and `<match>`, which needs full `is`
coverage or `<otherwise>`).

### 11.2 Match
`<match>` arms are evaluated **top to bottom; first match wins**. Exhaustiveness is computed from
the arms' **`is` patterns** (§7.3.1) plus `<otherwise>` (D-D): the union of `is` literals (and any
`unset` arm) must cover the subject's domain, or an `<otherwise>` must be present. Recognizing that
a provable CEL `test` equality also narrows the domain is **downgraded to MAY** — a conforming
checker is not required to reason about `test`-only coverage; authors express coverage through `is`
arms. Omitting `<otherwise>` remains the **exhaustive assertion** (≡ the ts-pattern
`.exhaustive()`): the checker MUST prove the `is` arms cover the domain, else error.

> **ts-pattern correspondence (non-normative).** `<when is="…">` ≈ a literal `P` pattern;
> `<when is="…" test="…">` ≈ `P.when(guard)`; `<otherwise>` ≈ `.otherwise(…)`; omitting
> `<otherwise>` ≈ `.exhaustive()`.

`unset` is a member of a nullable subject's domain — including a `run.*`/`user.*`/`app.*` subject
that is maybe-unset at the read site (§9.4); such a subject MUST be covered by an **`is="unset"`
arm** or an **`<otherwise>`**. A `test` guard that merely happens to exclude `unset` does **not**
satisfy coverage — consistent with the `test`-coverage MAY-downgrade above, a conforming checker is
not required to reason about guard narrowing, so `unset` coverage MUST be explicit
(`E-UNSET-UNCOVERED`). **Rating gating:** an age/rating-gated `<match on="app.rating">` MUST
explicitly handle the safe/minimum rating tier (the least-restrictive member of the `rating`
domain) **or** carry `<otherwise>` — a release build rejects a missing safe fallback (runtime
gating alone is not sufficient).

### 11.3 Timing & `wait`
`wait="true"` on a staging directive holds the script until that effect completes; an absent /
`false` `wait` is non-blocking (the next item proceeds — i.e. concurrently). The default is
**per-directive** (e.g. background and video default to `wait="true"`), not global.

### 11.4 Timeline time model
Within a `<timeline>` the local clock starts at `0`. For each track:

- a staging clip's `at` is an **absolute** time on that clock; an **omitted** `at` starts the clip
  after the previous clip in the same track (the first omitted `at` = `0.0`). `at` is never a
  relative nudge. `at` and `delay` on the same clip are mutually exclusive (**`E-CLIP-TIMING`**,
  S6).
- a clip's **resolved end** = its start + `duration` (or its start, if it carries no `duration`).
- a **`::set` clip** carries no timing and fires at the **resolved end of the previous clip** in
  its track (`0.0` if it is the track's first clip) — an instantaneous write at that instant.
- clips within one track MUST NOT overlap: a clip whose start precedes the previous clip's resolved
  end is **`E-CLIP-OVERLAP`**.

Each track key (`subject` / `channel` / `property` value) MUST be unique within the timeline
(**one writer per track**). Tracks play concurrently; two clips (across tracks) that resolve to a
write on the **same** underlying subject/property at the **same** instant are a determinism
violation, **`E-WRITE-CONFLICT`**. (`E-CLIP-OVERLAP` and `E-WRITE-CONFLICT` are normative spec text,
promoted here from the shipped checker.)

The block blocks following content until `duration` (or, absent `duration`, the maximum resolved
clip end across all tracks). An explicit `duration` that is **less than** the maximum resolved clip
end is **`E-TIMELINE-DURATION`** (S6) — the timeline may not be truncated below its own content. A
timeline MUST reduce to scheduled command records (with resolved absolute offsets) plus a final
barrier.

A `property` track additionally requires a `subject`; its track key is `<subject>.<property>`,
so several property tracks MAY drive the same subject concurrently as long as their properties
differ. A `<track>` with no usable key (none of `subject`, `channel`, or `subject`+`property`)
is rejected (`E-TRACK-KEY`).

### 11.5 Reduction
A conforming document MUST reduce, at build time, to a finite ordered sequence of command records
+ CEL strings (invariant §3.2). Implicit staging that an implementation injects (stage hygiene) is
**implementation-defined** and outside this language spec; it MUST be deterministic and MUST NOT
change the meaning of explicitly authored constructs.

## 12. Localization (i18n) & line identity

Content text is authored **monolingually and in flow** (§4.4 opaque `Text`); there is **no
text-key extraction** — authors never write against string keys. Every content line and every
`<choice>` / `<hub>` choice label carries one stable **`lineId`** — the localization string-table
key **and** the stable anchor a voice asset aligns to (its actual lookup key is `voiceKey`, below);
the Yarn-Spinner `#line:` model. Derivations key on `meta.episodeId` (§6.1, S3):

- a content line derives its `lineId` from its `code`, composed
  `{character}.{episodeId}.{speaker}_{code}`;
- a `<choice>` / hub-choice label derives its `lineId` from the choice's structural id, composed
  `{character}.{episodeId}.{branchOrHubId}.{choiceId}` (episode-prefixed);
- a content line with no author `code` is assigned one by a `lute tag` pass, generated **once and
  persisted into `code`** so `lineId` stays stable across edits/insertions.

Because the derivation keys on `episodeId` (not the raw `season`/`episode` numbers), **renumbering
an episode does not break translation/VO keys** when `episodeId` is pinned (S3). The derivation
still includes the speaker, so changing a line's *speaker* changes its `lineId` (a recast line is a
new unit); an opaque id would be a `lute tag` derivation change, not a language one (§14).

Translations are an **id-keyed sidecar / string table** (`lineId → { lang: text }`). Because
`app.lang` is runtime *state* (§9.1 — the language may switch mid-game), translated text is **not
baked into per-language artifact copies**: the compiled artifact keeps the source `text` (plus its
placeholder list), and each locale's table is loaded by the engine at runtime. Build time
**validates** — placeholder-set equality (below) and key coverage — rather than fills. The
translation **process** is external tooling, outside this language. A `lineId` is **assigned once
and never recomputed** (not positional, not a content hash), so editing or inserting text does not
break existing translations.

**Interpolation placeholder contract.** Interpolations (§7.6) are part of the line's translatable
text keyed by `lineId`. At build time the compiler MUST check **placeholder-set equality** between
source and each translation for a `lineId`: the set of placeholders (paths / refs / reserved
tokens, by identity) in the translation MUST equal the source's set. A mismatch (a translation that
drops, adds, or renames a placeholder) is **`E-L10N-PLACEHOLDER`** (a compile-stage diagnostic).

**Voiced lines MAY interpolate.** A voiced line (dialogue / voiceover) carrying a `{{…}}`
placeholder keeps its `voiceKey` — interpolation does **not** suppress voice, and there is **no**
`E-VOICE-INTERP`. By production convention the VO team records against a substitute word for each
placeholder; the line's `placeholders` list (IR addendum A3) doubles as that substitution manifest
(the slots a recording must leave room for). This is an authoring/recording concern, not a static
error.

**`voiceKey`.** The separate `voiceKey` (voiced lines only) is the voice-asset key, aligned to
`lineId`: core `{characterId}-{code}`, or `{voiceBank}-{code}` when a static costume voice bank
applies — see the character-cast contract (`character-cast/0.0.1.md` §7.3).

### 12.1 `delivery` → role derivation (single source; X1)

Role is derived from `speaker` + the `delivery` attribute (§7.1). This table is the single source
of truth (the compile-IR addendum references it, not a local copy):

| Speaker | `delivery` | Derived role |
|---|---|---|
| `narrator` | *any present* | **static error** — narration takes no `delivery` |
| `narrator` | absent | `narration` (speakerless) |
| any other | `thought` | `monologue` (inner voice) |
| any other | `voiceover` | `voiceover` |
| any other | `spoken` / absent | `dialogue` |

A `delivery` attribute on `narrator` is a static error. `delivery="thought"` is **legal on any
character**, not only the `pov` player — a non-player `thought` line is that character's inner voice
(monologue).

## 13. Reusable content components (`::use`)

`defs` (§8.1) reuse typed CEL *values*; `uses`/`extends` (§9.2) reuse *schema*. A **component**
reuses *content* — a named, parameterized block of lines + staging that is expanded inline wherever
it is invoked.

### 13.1 Definition
A component is a **component file**: a `.lute` document whose frontmatter declares `component:
<name>` and (optionally) `params: { <p>: <type>, … }` (typed exactly as `defs` params, §8.1). Its
body is the template. Within the body, a parameter is referenced as `@<p>` (§8.1) in any ref/attr
position; a `@<p>` that is not a declared param is `E-UNDECLARED-REF`. A component file declares no
scene keys (checked in *component mode*, like a schema doc §9.2); a malformed `params:` mapping is
`E-COMPONENT-PARSE`.

### 13.2 Import & invocation
A scene imports components via a `components: [<path>, …]` frontmatter key — canonicalized,
cycle-checked, diamond-deduped like `uses:` (§9.2). Invocation is the reserved built-in directive
`::use{ component="<name>" <arg>=<value> … }`, which expands the named component's body inline with
each `@<p>` bound to the matching `<arg>` (a compile-time macro over content; expansion is
engine-side, §11.5). `use` is reserved (§10).

### 13.3 Static checks
- `component` MUST name a component from an imported component file — else `E-COMPONENT-UNDECLARED`.
- named args MUST match the component's `params` in count and type (as `@name(args)`, §8.1) — else
  `E-COMPONENT-ARG`.
- component expansion MUST be acyclic — a cycle is `E-COMPONENT-CYCLE`; a duplicate component name
  across imports is `E-COMPONENT-DUP`.
- each imported component file is validated (component mode); malformed → `E-COMPONENT-PARSE`.

### 13.4 0.1.0 scope
One component per file; a component body is **presentational** (lines + staging + `@param` refs; no
scene/run state read/write and no logic blocks — pass values via params); params are usable in
ref/attr positions **and, as of 0.1.0, in content `Text` via `{{@param}}` interpolation** (§7.6) —
the 0.0.1 restriction that deferred text interpolation is lifted. A body MAY `::use` other
components (acyclic) but MAY NOT define one. Invocation is directive-form (`::use`) to reuse the
existing directive + frontmatter + import-DAG surface with no grammar change; a
`<component>`/`<use>` element syntax is possible future work.

## 14. Non-goals

- **Opaque `lineId` mode** — **deferred, not rejected.** With `episodeId` pinning (§12), the
  remaining breakage (recast, cross-episode line moves) is either semantically a new unit or rare;
  opaque ids would cost pipeline readability (VO file naming, string-table review). Reserved as a
  future `lute tag` derivation option (`lineIdMode: opaque`) — a **tooling** change, not a language
  change.
- **Backward diverts** — **permanent non-goal.** Cycles would demote every static analysis
  (definite assignment, exhaustiveness, stage join, path enumeration) to fixpoint computations and
  make termination author-owned. `<hub>` (§7.3.2, §11.1.3) covers the legitimate revisit patterns
  structurally. **Reserved constraint:** any future divert MUST be forward-only (DAG-preserving),
  e.g. a `::goto{shot=N}` with `N` strictly ahead of the current shot.
- **ICU MessageFormat** — **reserved section only** (§7.6). The placeholder-set contract (§12)
  ships now; ICU plural/select does not.
- **Kept as-is** (complexity judged already minimal): the 4-tier state model (§9.1), the timeline
  `<track>` syntax (§7.4), YAML frontmatter (§6.1), the `::set` statement form (§7.3.3), and the
  absence of a `/=` operator (§8.4).

## Appendix C — Revision note (0.1.0)

`0.1.0` is the first **breaking** revision, produced spec-first from an approved multi-domain
language evaluation (vs Yarn Spinner / Ren'Py / Ink / LaTeX / SMIL / k8s manifests) plus three
read-only audits (plugin spec↔impl, cast↔IR cross-check, checker/grammar reality check). The
breaking changes over the frozen `0.0.1`:

- **Content line** `:line[speaker]: text` → `:speaker: text` (hot-path cost; §7.1, §4.3, Appendix D).
- **`{{…}}` interpolation** — new (§7.6), including the `{{userName}}` reserved token and the
  translation placeholder-set contract (§12).
- **`<hub>`** — new revisit conversation (§7.3.2, §11.1.3), Ink-parity without backward diverts.
- **`<when is>`** literal patterns and `is`-based exhaustiveness (§7.3.1, §11.2).
- **Lute-CEL profile** — a normative section fixing the numeric model, closed environment,
  expansion hygiene, and CEL-facing identifier rules (§8.4).
- **`meta.episodeId`** and episode-prefixed `lineId` derivation (§6.1, §12).
- Defect fixes S4–S11, X1: line/block comments and `Text` opacity (§4.2), `E-BRANCH-ALL-GUARDED`
  (§7.3), timeline timing diagnostics (§7.4–§7.5, §11.4), attr coercion + `E-STRING-ESCAPE` (§4.5,
  §4.4), `E-AT-CONTEXT` (§7.5), `E-SHOT-HEADING` + Prologue/Epilogue (§6.3), `<otherwise>` no-attrs
  (§7.3), `defs` `min`/`max`/`values` removed (§8.1), and the delivery→role table (§12.1).

`0.0.1` was itself revised (pre-implementation) after a grammar↔example consistency audit and a
state-model redesign to the four-tier lifetime ladder; that history is retained in the frozen
`0.0.1.md` and its `state-model-design.md`.
