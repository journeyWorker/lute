## 1. Abstract

Through 0.2.0, Lute state is **scalar**: typed paths (`number|bool|enum`) in tiered
namespaces (§9). Scalars cannot express *relationships between entities* — "Shadowheart is
in the party," "the player told Halsin about the grove," "faction A is hostile to faction
B" — without the engine pre-flattening each into a bespoke boolean, the pattern the 0.2.0
state-contract (`0.2 §8`) prescribes.

0.3.0 adds a **relational fact database** as a first-class state kernel: a set of **tuples**
over a **declared, closed** vocabulary of **relations** and **entity kinds**, each tuple
carrying a **valid-time interval**. Content **asserts** and **retracts** facts as
deltas; the engine maintains the cumulative view. A **total Datalog derivation** layer
computes new relations from base facts (joins, projection, transitive closure) with
**guaranteed termination** and **static verifiability**. Conditions (CEL, `<match>`,
`<when>`, `<on>`) query the fact set through a **bounded** predicate surface.

The relational layer is **Datalog, not Prolog**: function-free, so the fact universe is the
finite set of declared entities/values (a finite Herbrand base), so every derivation
terminates and every static guarantee Lute already makes (exhaustiveness §11.2,
definite-assignment §9.4, reduction §3.2) extends to it. It is **not** a runtime logic
interpreter authored ad hoc — it reduces, like everything in Lute, to data the engine
evaluates deterministically.

0.3.0 is written **full-feature**: the entire relational vision (schema, assert/retract,

nothing existing is removed or reinterpreted). A document opts in simply by declaring
relations (§4); a document with no relation declarations behaves exactly as 0.2.0.

## 3. The relational fact model

### 3.1 Entities, relations, facts

```
Entity   ::= a value of a declared entity kind (an id), or an enum member, or a Bool
Relation ::= a declared, named predicate of fixed arity over declared arg domains
Fact     ::= an instance of a relation applied to entities:  rel(a1, …, an)
```

- An **entity kind** is a declared, closed domain of ids (e.g. `character`, `faction`,
  `location`, `item`), declared as **exactly one** of:
  - `{ members: [<id>…] }` — a finite, author-enumerated id list, closed at compile time;
  - `{ open: engine }` — an **engine-populated** kind: ids are minted by the engine (the
    engine→state contract, `0.2 §8`) and are **READ-ONLY to content** — content queries
    them; it never invents or asserts one into existence. An id simply becomes a valid
    `K(X)` member once the engine mints it.

  A kind declaring **neither**, or **both**, is **`E-ENTITY-KIND-SHAPE`**.
- Entity ids are `CelIdent` (no `-`, §8.4) — they are CEL/path-facing. An entity kind `K`
  doubles as a **unary domain predicate** `K(X)` usable in rule bodies ("X is a declared
  `K`"): for `members:` this ranges over the declared list; for `open: engine` it ranges
  over the **currently-engine-minted** id set. Either way the domain is finite **at every
  evaluation instant** — an engine mints finitely many ids over any finite play session,
  even though an `open: engine` kind's exact membership is not enumerable at *compile*
  time — the finite domain both §7.2 safety and §10's finite-Herbrand-base argument rely
  on.
- **One global id namespace, partitioned by kind.** An id belongs to exactly one entity
  kind. An id declared a member of two different kinds is **`E-ENTITY-KIND-CLASH`**. This,
  plus the domain check below, is the closure that prevents nekg-style open-vocabulary
  drift the spec cites as its control: an `::assert` can no longer name an id that was
  never declared, or that belongs to the wrong kind.
- **Reset semantics.** A tier reset (§9.1, §4) clears that tier's **facts**, never a kind's
  **membership** — `members:`/`open:` is schema, resolved once at load, not per-tier state.
- A **relation** is a declared predicate with a fixed **arity** and a **typed argument
  signature**; each argument ranges over a declared entity kind, a declared `enums:` name
  (§4), or `bool`. The common binary case is **SVO** (`subject`, `object`); higher arity and
  reified events (§9.4) cover the rest.
- A **fact** is a relation applied to entities of the matching signature. A fact is
  **symbolic** — it either holds or does not. It carries **no numeric value slot**;
  magnitudes live in the scalar tiers, discretized into enum-valued relation args when they
  must participate in facts (facts stay symbolic — §11 has the scalar-magnitude
  round-trip pattern).

The relation vocabulary and every entity kind are **closed**: declared up front, finite (or,
for an `open: engine` kind, finite at every instant), and statically domain-checkable. This
is the doubly-closed property (§10) — Datalog's function-free closure *and* the game's
declared-schema closure — that makes the layer total and checkable.

### 3.2 Valid-time interval

Every fact carries a validity interval keyed to **narrative time** (the engine-stamped
current position, e.g. an episode/shot token or a monotonic tick — §6):

```
established : <narrative-time>          (* when the fact became true *)
invalidated : <narrative-time> | unset  (* when it ceased to be true; unset = still valid *)
```

The interval is **half-open**: `[established, invalidated)`. A fact is **valid at** query
time `T` iff `established ≤ T ∧ (invalidated = unset ∨ T < invalidated)`. **Retraction is a
tombstone**, never a deletion: `::retract` sets `invalidated`; the fact and its history
remain (auditability, correct replay). Default queries (§8) read the **currently-valid**
set. This models **validity time only** — the engine does not additionally track when a
fact was *recorded* versus when it was *true* (a bitemporal system's second axis); one
timeline, keyed to narrative time, is enough for a scenario's needs.

## 4. Fact schema declaration

Entity kinds, named enum types, relations, and seed facts are declared in the state schema
(§9.2 schema documents / frontmatter `state:`-adjacent), in new `entities:` / `enums:` /
`relations:` / `facts:` blocks:

```yaml
entities:
  character: { members: [shadowheart, halsin, player] }   # closed: exactly these ids
  location:  { members: [grove, moonrise] }
  faction:   { members: [harpers, absolute] }
  npc:       { open: engine }                              # engine-minted ids only

enums:
  trustLevel: [stranger, ally]

relations:
  inParty:    { args: [character], tier: run }                            # unary
  atLocation: { args: [character, location], tier: run, key: [0] }        # binary (SVO), functional on subject
  hostile:    { args: [faction, faction], tier: run }                     # binary, directional base fact
  trustTier:  { args: [character, character, trustLevel], tier: run, reserved: true } # engine-asserted only (§7.3)
  knowsLoc:   { args: [character, character, location], tier: run }       # ternary (holder-first, §9)

facts:
  - "hostile(harpers, absolute)"
```

- **`entities`** — see §3.1 for the `members:` / `open: engine` shapes and their
  diagnostics.
- **`enums`** — a named enum type: an ordered, closed list of members (`EnumMember`, §4.4).
  A relation `args:` entry, or a scalar `state:` path's `type:` (§9.3), MAY name a declared
  `enums:` entry instead of the bare `enum` keyword — sharing one closed vocabulary across
  every use site instead of a per-path inline list. Anonymous inline `type: enum`
  declarations remain legal for a single scalar path that needs no shared vocabulary.
- **`relations`**:
  - **`args`** — the ordered argument signature; each entry names a declared entity kind, a
    declared `enums:` name, or `bool`. Arity = `len(args)`. An arg naming no declared domain
    is **`E-RELATION-DOMAIN`**; a relation declared with **no** `args` at all is
    **`E-RELATION-EMPTY`** (a *use* of a relation at the wrong arity — assert, retract,
    rule, or query — is the separate **`E-RELATION-ARITY`**, §5/§7/§8: a schema typo and a
    call-site typo get distinct diagnostics).
  - **`tier`** — the reset boundary that owns this relation's live facts AND their interval
    history (§3.2) (`scene`/`run`/`user`/`app`/`quest`, §9.1, `0.2 §5`); a reset clears
    both. Default `run` if omitted. **`tier:` MUST NOT appear on a `derive: true`
    relation** — a derived relation's lifetime is induced by the relations its rules read,
    not declared independently; `tier:` on a `derive: true` relation is **`E-DERIVE-TIER`**
    (§7.1).
  - **`derive`** — `true` marks a relation as computed by `rules:` (§7); its facts are
    engine-computed and read-only to content (`E-DERIVED-WRITE`, §5). A `derive: true`
    relation with zero matching `rules:` entries is **`W-DERIVE-NO-RULES`** (a warning —
    almost always a typo'd head name; the relation is legal but permanently empty).
  - **`reserved`** — `true` marks a relation as **engine-populated by non-Datalog means**
    (e.g. a threshold-crossing sensor, §7.3): content MAY query it (`holds`/`count`/
    `validAt`, §8) but MUST NOT `::assert`/`::retract` it (**`E-RELATION-RESERVED-WRITE`**,
    mirrors `E-QUEST-RESERVED-WRITE`, `0.2 §5.4`). `reserved` and `derive` are independent
    write-owner axes; a relation MUST NOT declare **both** (two conflicting write-owners) —
    also **`E-RELATION-RESERVED-WRITE`**.
  - **`key`** — an optional list of `args` indices (0-based) forming a **functional key**:
    at most one currently-valid fact may share a given key-tuple. A new `::assert` on an
    already-valid key **auto-invalidates** the previous fact on that key, engine-side, at a
    single narrative-time token (§6, §5) — the fix for a single-valued SVO relation (e.g.
    `atLocation`) accumulating contradictory facts. Indices MUST be valid, distinct
    positions within `args` (`E-RELATION-DOMAIN`). **Recommended** for single-valued SVO
    state relations.
- **`facts`** — a schema-level list of **ground seed facts**, checked against the declared
  schema (`E-FACT-DOMAIN`/`E-RELATION-ARITY` as for `::assert`) and **materialized at load
  AND at the owning relation's tier reset** — the exact fact-DB analogue of scalar
  `default:` (§9.3). This is the only way to populate an `app`-tier (or `reserved`)
  relation, since both are otherwise unwritable by content (§5) — e.g. static map topology
  (`connected`, Appendix B). Duplicate seed facts (within one document or across `uses`
  peers) are harmless: facts are a set, and re-asserting a currently-valid fact is a no-op
  (§5).
- Relation names, entity-kind names, and `enums:` names are `CelIdent` (no `-`, §8.4). A
  duplicate relation name is **`E-RELATION-DUP`**. An entity-kind name reused for a second
  kind, or colliding with a relation name — kinds and relations share the rule-body
  predicate namespace (§3.1's `K(X)`, §7.1's `Relation(...)`) — is **`E-KIND-NAME-CLASH`**.
- The vocabulary is **closed**: asserting/deriving/querying a relation not declared here is
  **`E-RELATION-UNKNOWN`**; applying a relation at the wrong arity is **`E-RELATION-ARITY`**.
  **`E-FACT-DOMAIN`** checks EVERY argument against its declared domain — entity-kind
  membership (closed `members:`, or the current `open: engine` set), enum membership, or
  `bool` — this is the closure that prevents nekg-style open-vocabulary drift the spec
  cites as its control.
- **Quoting.** Both `facts:` entries and `rules:` entries (§7.1) are YAML strings containing
  `:-`/parens; an unquoted entry with a stray space can silently misparse as a YAML mapping
  (`head :- body` reads `head` as a key). Every `facts:`/`rules:` entry in this document —
  and every conforming document — MUST be a quoted string (`- "hostile(harpers, absolute)"`,
  `- "canReach(C, L) :- atLocation(C, L)"`).

### 4.1 Composition (`uses` / `extends`)

`entities:`, `enums:`, `relations:`, and `facts:` compose through the same import DAG as
scalar `state:` (§9.2):

- **`uses`** unions PEER schemas. A relation or `enums:` name declared by two unrelated
  peers is **`E-USES-DUP-RELATION`** (the relational-vocabulary analogue of
  `E-USES-DUP-STATE`); an entity-kind name colliding with another kind or with a relation —
  peer-declared or not — is **`E-KIND-NAME-CLASH`** (§4, checked on the fully merged
  vocabulary). `facts:` lists simply union (duplicates collapse, §4).
- **`extends`** names a lower-precedence BASE a document refines. A relation's
  **signature** — `args` + `tier` + `derive` + `reserved` — MUST match its base exactly; an
  `extends` child MAY ONLY **ADD** `rules:` (and `facts:`) to an inherited relation, and MAY
  ONLY **ADD** members to an inherited entity kind or values to an inherited `enums:` entry.
  It MUST NOT change a relation's class (flipping `derive`/`reserved` orphans base writes
  and rule heads; widening `tier` breaks reset semantics across the merged graph), remove or
  retype an inherited member/enum value, or replace an inherited relation's rules wholesale.
  Any such mismatch is **`E-EXTENDS-RELATION-SIG`** (mirrors `E-EXTENDS-STATE-TYPE`).
- `rules:` always **union** across the import DAG, exactly as multiple `rules:` entries for
  one head union within a single document (§7.1); an `extends` child adds to, never
  replaces, a base-declared head.
- **Load order.** Schema-document `rules:` MAY reference only schema-declared relations;
  scene/quest-tier relations and their rules live in that document's own frontmatter, loaded
  after its imported schemas (§9.2 loads schemas before scenes).
- `E-DATALOG-UNSAFE` and `E-DATALOG-UNSTRATIFIED` are checked on the **merged**, post-
  composition rule set (§7.2) — a negation cycle or unbound variable spanning two documents
  is caught, not missed per-document.

## 5. Asserting & retracting facts

```
Assert         ::= "::assert"  "{" GroundFact "}"
Retract        ::= "::retract" "{" RetractPattern "}"
GroundFact     ::= Ident "(" Arg ( "," Arg )* ")"           (* Ident = relation *)
RetractPattern ::= Ident "(" RetractArg ( "," RetractArg )* ")"
Arg            ::= CelIdent | EnumMember | Bool             (* compile-time-ground literal *)
RetractArg     ::= Arg | "_"                                 (* wildcard: retract-only *)
```

- `::assert{ inParty(shadowheart) }`, `::assert{ atLocation(shadowheart, grove) }`,
  `::retract{ atLocation(shadowheart, grove) }` — single-line leaf directives (`::` form,
  bracket rule §5 / `0.2 §6.4`: a **pure leaf**, never nestable — it keeps the `::`/`:` form
  even where a body-capable sibling construct would self-close). One fact per directive.
- **Args are compile-time-ground.** `Arg` is a concrete entity id / enum member / bool
  literal — **`{{…}}` interpolation is NOT admitted** in fact arguments: assertion is a
  delta over the closed vocabulary, not a template. `validAt`/`holds` patterns (§8)
  likewise take ground literals or `_`, never an interpolation. A base fact's args MUST
  match the relation signature (`E-FACT-DOMAIN` / `E-RELATION-ARITY` otherwise).
- **Wildcard retract.** `::retract{ atLocation(shadowheart, _) }` retracts every
  **currently-valid** fact matching the ground positions — bounded over the finite valid
  set, so still total. `_` is admitted **only** in `::retract` patterns; `::assert` stays
  fully ground (it is a single concrete delta, never a bulk operation). A `_` inside
  `::assert` is **`E-RETRACT-WILDCARD-ASSERT`**. Arity/domain diagnostics on a retract
  pattern's ground positions mirror assert (`E-RELATION-ARITY` / `E-FACT-DOMAIN`).
- **Semantics.** `::assert` adds the fact with `established = now()` (§6), `invalidated =
  unset` (idempotent: re-asserting a currently-valid fact is a no-op). If the relation
  declares a functional `key:` (§4) and a distinct fact already holds on that key, the
  engine **auto-invalidates** the superseded fact at the same token that establishes the
  new one — no gap, no overlap (§6.1). `::retract` sets `invalidated = now()` on every
  currently-valid matching fact (retracting an absent fact is a no-op). The engine owns the
  merge and the timestamps — **authors write deltas; the engine derives the cumulative,
  time-scoped view** (the chronicle model; the nekg anti-pattern was manual accumulation).
- **Where.** `::assert`/`::retract` are admitted wherever `::set` is (scene content,
  `<on>`/quest/objective arms, `0.2 §3.3`). They are the relational analogue of `::set`, and
  share its read/write visibility rules — §8.1.
- **Write policy.** In order of precedence:
  1. A **derived** relation (`derive: true`, §7) MUST NOT be asserted or retracted by
     content — engine-computed (**`E-DERIVED-WRITE`**).
  2. A **reserved** relation (`reserved: true`, §4) MUST NOT be asserted or retracted by
     content — engine-computed by non-Datalog means (**`E-RELATION-RESERVED-WRITE`**).
  3. An **`app`-tier** relation (neither derived nor reserved) is engine-owned/read-only to
     content, exactly as `app.*` scalar state is (§9.5, `E-APP-READONLY`): writing it from
     content is **`E-FACT-TIER-WRITE`**.
  4. Otherwise — a `scene`/`run`/`user`/`quest`-scratch base relation — is author-writable.

  `app`-tier and `reserved` relations are populated only via the schema `facts:` seed block
  (§4, at load/reset) or, for `reserved`, engine-side non-Datalog writes (§7.3).

## 6. Temporal validity & narrative time

- **Narrative-time value.** An OPAQUE, TOTALLY-ORDERED token type, distinct from the three
  scalar types (`number|bool|enum`, §9.3) — surfaced ONLY by the engine. It is: (a) the
  implicit type of every fact's `established`/`invalidated` field (§3.2); (b) obtainable in
  CEL ONLY via the reserved nullary builtin **`now()`**, added to the closed Lute-CEL
  profile (§8.4) — the current narrative-time token; and (c) obtainable via further
  engine-provided anchors (e.g. a quest instance's start tick, an episode boundary) that a
  game's engine→state contract (`0.2 §8`) MAY surface as additional reserved,
  engine-populated, read-only paths of this type — declared and validated exactly as
  capability world events are (`0.2 §4.5`), never author-declared as ordinary
  `number|bool|enum` state.
- **Closed comparison surface.** CEL admits ONLY ordering comparison — the five operators
  `<`, `<=`, `==`, `>`, `>=` — between two narrative-time values. `>`/`>=` are the trivial
  mirror images of `<`/`<=` (no new semantics), and `==` is identity. **`!=` is NOT
  admitted**: inequality is identity-*negation*, a distinct predicate class rather than an
  ordering, so it is **`E-TEMPORAL-ARG`** (an author who genuinely needs it writes
  `!(a == b)`). **No arithmetic, no literal construction** — there is no `time` literal
  syntax, and applying `+`/`-`/indexing to a narrative-time value, or comparing it against a
  non-narrative-time value, is likewise **`E-TEMPORAL-ARG`**.
- This ordering-only value is what makes `established ≤ T ∧ (invalidated = unset ∨ T <
  invalidated)` (§3.2) and the `validAt` guard below well-typed and total: a
  rewind/replay/save-load query is a pure comparison over the append-only delta log, never a
  computed offset.

Queries (§8) default to **valid-now**. A condition MAY ask for historical validity via the
guard form `validAt(rel(args|_), T)` (§8), where `T` is a narrative-time-valued CEL
expression. `holds(rel(args|_))` is the valid-now case of the same query — equivalent to
`validAt(rel(args|_), now())` for a relation `validAt` admits below.

- **Domain restriction.** `validAt` is admitted only over a **BASE** relation, or a
  `derive: true` relation whose rules carry **no** CEL guard (§7.3) in ANY stratum feeding
  it — a scalar CEL guard reads only a valid-now snapshot of global state (§7.3), so a
  point-in-the-past derivation through a guarded rule is ill-defined (scalars keep no
  history). `validAt` over a guarded derived relation is **`E-VALIDAT-DERIVED`**.
- **Reset clears history too.** A tier reset clears the relation's live facts AND its
  interval history for that tier — `validAt` answers do not survive a reset, matching scalar
  reset semantics (§9.3: a reset re-materializes `default`s from the one schema snapshot).

### 6.1 Ordering & granularity (normative)

- The narrative-time token is **strictly monotonic per delta** — one command-stream tick
  advances it — so `established`/`invalidated` totally order every assert/retract,
  including the engine-side `key:`-driven auto-invalidate (§4, §5): the superseded and
  superseding fact never share an open interval.
- Within one token (several deltas emitted by the same command), ordering is **command
  order**; `validAt` itself is **token-granular** — it cannot distinguish two deltas within
  the same token. Combined with the half-open interval (§3.2), a fact retracted in the same
  token it was asserted has a **zero-width** (immediately invalid) interval rather than an
  undefined one — resolving the "asserted-and-retracted within one token" ambiguity.

## 7. Datalog derivation

Derived relations are computed from base (and other derived) relations by **rules** declared
in the schema:

```yaml
entities:
  character: { members: [shadowheart, halsin, player] }
  location:  { members: [grove, moonrise] }
  faction:   { members: [harpers, absolute] }

relations:
  atLocation:     { args: [character, location], tier: run, key: [0] }
  connected:      { args: [location, location], tier: app }
  hostile:        { args: [faction, faction], tier: run }
  ally:           { args: [faction, faction], derive: true }
  hostileEither:  { args: [faction, faction], derive: true }
  canReach:       { args: [character, location], derive: true }
  alerted:        { args: [faction, faction], derive: true }

facts:
  - "connected(grove, moonrise)"
  - "hostile(harpers, absolute)"

rules:
  # symmetric + reflexive-free alliance is the complement of hostility — a DERIVED VIEW over
  # the base relation `hostile` (a relation is either base or derived, never both, §7.1)
  - "ally(A, B)           :- faction(A), faction(B), not hostile(A, B), not hostile(B, A), A != B"
  # the base/derived-view idiom (§7.1): a symmetric read of a directional base relation
  - "hostileEither(A, B)  :- hostile(A, B)"
  - "hostileEither(A, B)  :- hostile(B, A)"
  # transitive reachability (recursion — terminates: finite Herbrand base)
  - "canReach(C, L)       :- atLocation(C, L)"
  - "canReach(C, L2)      :- canReach(C, L1), connected(L1, L2)"
  # a constant head (§7.1): mixes a ground Const (absolute) with a bound Var (F)
  - "alerted(absolute, F) :- hostile(F, absolute)"
```

### 7.1 Rule form

```
Rule ::= Head ":-" Body
Head ::= Relation "(" (Var | Const) ( "," (Var | Const) )* ")"   (* a derived relation *)
Body ::= Literal ( "," Literal )*
Literal ::= Relation "(" Term ( "," Term )* ")"            (* positive atom *)
          | "not" Relation "(" Term ( "," Term )* ")"      (* negated atom (stratified) *)
          | CelGuard                                        (* a CEL condition, §7.3 *)
          | Term ( "=" | "!=" ) Term                        (* (in)equality *)
Term    ::= Var | Const                                     (* Var = uppercase; no function symbols *)
Const   ::= CelIdent | EnumMember | Bool                    (* a declared entity id / enum member / bool *)
CelGuard ::= "cel(" CelString ")"                            (* global scalar state only — §7.3, B2 *)
```

- A `rules:` entry defines one Horn clause. Multiple rules for the same head **union**
  (disjunction). The head relation MUST be declared `derive: true`
  (**`E-DERIVE-UNDECLARED`** for a rule head that isn't; **`E-DERIVED-WRITE`** conversely).
  **A relation is either base or `derive: true`, never both** — a relation's facts have
  exactly one owner (author `::assert`/`::retract`, or `rules:`); there is no third
  "base-with-rules-too" shape. `hostileEither` above is the idiom for the common case where
  a base relation also needs a derived, symmetric (or otherwise reshaped) VIEW: declare a
  second relation and rule it from the first, never mix the two roles on one name.
- **Constants in rule heads.** A head argument MAY be a bound `Var` or a ground `Const`
  (standard Datalog) — e.g. `alerted(absolute, F) :- hostile(F, absolute)` above fires
  `alerted(absolute, harpers)` once `hostile(harpers, absolute)` is a valid fact,
  regardless of any OTHER variable binding for the constant position. A `Const` head
  argument needs no positive-body occurrence — safety (§7.2) is a **variable** rule.
- **Equality binds; inequality never does.** `V = c` (a variable against a ground `Const`)
  or `V = W` where `W` is already bound by a positive atom **binds** `V` for §7.2 safety
  purposes — a legal alternative to a positive-atom binding. `!=` (as in the `ally` rule
  above) never binds a variable; both sides MUST already be bound.
- **No function symbols.** Terms are variables or declared constants only — no compound
  terms, no arithmetic constructing new values. This is the Datalog restriction that keeps
  the Herbrand universe finite and every rule terminating (§10). Using a would-be function
  term is **`E-DATALOG-FUNCTION`**.
- A `derive: true` relation with no matching `rules:` entry is **`W-DERIVE-NO-RULES`**.

### 7.2 Safety, stratification, termination (normative)

- **Safety:** every variable in the **head**, and every variable in a **negated** or
  **CEL-guard** literal, MUST also appear in a **positive** body atom, or be
  **equality-bound** (§7.1) to one that does. An unbound variable is **`E-DATALOG-UNSAFE`**.
  (Guarantees derived facts range only over the finite base.)
- **Guards are fact-query-free.** A rule-body CEL guard MUST NOT contain `holds`/`count`/
  `validAt` (§8) — fact-query functions are condition-surface only; threading one through a
  guard would hide a non-monotonic dependency from this very safety/stratification analysis.
  Details and the diagnostic (**`E-DATALOG-GUARD-FACT`**) are in §7.3.
- **Stratified negation:** the rule set's predicate-dependency graph MUST have no cycle
  through a `not` edge — recursion through negation is **`E-DATALOG-UNSTRATIFIED`**.
  Positive recursion (e.g. `canReach`) is allowed and total. Both this check and the safety
  check above run on the **merged** rule set — after `uses`/`extends` composition (§4.1) —
  so a negation cycle or an unbound variable spanning two documents is caught, not missed
  per-document.
- **Termination:** because the schema is function-free and finite (at every evaluation
  instant, §3.1), the least fixpoint is reached in finitely many steps (bottom-up evaluation
  over a finite Herbrand base). No author-declared recursion cap is required for
  termination; an implementation MAY expose a cost budget, but termination is a
  **structural** property, not a convention (the chronicle lesson: bake the guarantee into
  the checker, don't lint it).
- Derivation is a **pure function** of the currently-valid base facts (§3.2). It is
  recomputed by the engine; content never sees a partially-updated view.

### 7.3 CEL-guard bridge (facts ↔ scalars)

A rule body MAY contain a **CEL guard** over **global scalar state** (§9), plus (in)equality
on the rule's bound variables, bridging the two state models **totally**:

```yaml
entities:
  location: { members: [grove, moonrise] }

relations:
  site:      { args: [location], tier: app }
  act1Site:  { args: [location], derive: true }

facts:
  - "site(grove)"

rules:
  # a site counts as reachable-in-act-1 only once the run has entered act 1 (global scalar gate)
  - "act1Site(L) :- location(L), site(L), cel(\"run.act == 1\")"
```

- The guard is ordinary Lute-CEL (§8.4, closed profile); it reads **global** scalar tiers
  and must be safe (§7.2). It binds no head variable (introduces no term), preserving safety
  and termination, and is read-only/side-effect-free like all CEL.
- **A rule-body CEL guard MUST NOT contain `holds`/`count`/`validAt`** (§8). Fact-query
  functions are a **condition-surface-only** construct, never a rule-body one: threading a
  fact query through a guard would make the rule's OWN dependency on other facts invisible
  to the safety/stratification analysis (§7.2) — an untracked non-monotonic dependency that
  breaks least-fixpoint existence and termination. Violating this is
  **`E-DATALOG-GUARD-FACT`**.
- **Per-entity magnitudes stay out of rules.** A rule guard does NOT index scalars by a rule
  variable (there is no `run.trust[A][B]` form — magnitudes stay as ungrouped sibling paths,
  §11). When a *per-entity* magnitude must become a fact (e.g. "trust ≥ ally threshold"), the
  **engine** asserts the discretized fact as a `reserved: true` relation (§4) as it crosses
  the threshold — the engine→state contract (`0.2 §8`) — and rules/queries read that fact.
  E.g. `trustTier` (§4) is asserted by the engine as `trustTier(shadowheart, halsin, ally)`
  once trust crosses the ally threshold; content MUST NOT `::assert`/`::retract` it
  (`E-RELATION-RESERVED-WRITE`). Facts stay symbolic; magnitudes stay scalar.

## 8. Querying facts in conditions

Conditions — `when`, `<match on>`, `<when test>`, `<on when>`, `::set` RHS guards — read the
fact-DB through a **bounded** predicate surface added to the Lute-CEL profile (§8.4):

```
holds(rel(args|_))              — the fact is valid now (any subset of slots may be `_`)
count(rel(args|_)) OP n          — int count of DISTINCT valid tuples matching the pattern
validAt(rel(args|_), T)          — historical form of holds (§6: base relations / guard-free
                                    derived strata only; T a narrative-time expression)
```

- `holds` is the primary query: `when="holds(inParty(shadowheart))"`,
  `when="holds(hostile(harpers, absolute))"` (real faction ids — `grove` is a `location`
  kind everywhere in this document; under §3.1's one-id-one-kind rule it can never also be a
  `faction`). Ground args = membership; **any subset of argument slots may instead be `_`**,
  each an independent bounded existential over that slot's declared domain (finite → total):
  `holds(rel(a, _))` = "∃ some entity in that slot such that the fact holds";
  `holds(rel(_, _))` (every slot free) is legal too. `validAt` patterns admit the same
  ground-literal-or-`_` shape.
- `count(rel(args|_))` is an **int-valued CEL expression**, not a bare boolean predicate —
  it counts the **distinct currently-valid tuples** matching the pattern (ground positions
  fixed, `_` positions free), and is freely composable in the closed profile
  (`count(...) + 1 <= n`, `count(...) > 0 && …`, not just a top-level `OP n` comparison).
  Bounded → total.
- **Joins under aggregation are expressed as derived relations, not multi-relation
  `count`.** `count(hasClass(_, cleric)) >= 3` counts **every** cleric in the whole
  schema-declared cast — not "clerics in the party." To count a join, name the join as a
  `derive: true` relation and `count` that:
  ```yaml
  entities:
    character: { members: [shadowheart, halsin, player] }
  enums:
    class: [cleric, fighter]
  relations:
    inParty:      { args: [character], tier: run }
    hasClass:     { args: [character, class], tier: run }
    partyCleric:  { args: [character], derive: true }
  rules:
    - "partyCleric(C) :- inParty(C), hasClass(C, cleric)"
  ```
  then `count(partyCleric(_)) >= 3` — "the party has ≥ 3 clerics." This is the *total*
  replacement for the CEL comprehensions the profile forbids (§8.4 `E-CEL-PROFILE`) — the
  engine no longer needs to pre-flatten aggregates into scalars (`0.2 §8` boilerplate goes
  away).
- These predicates are the **only** fact-query functions admitted; a bare relation reference
  or an unbounded/collection-returning form is **`E-CEL-PROFILE`** (still closed).
  `holds`/`count`/`validAt` over an undeclared relation is `E-RELATION-UNKNOWN`; wrong arity
  is `E-RELATION-ARITY`.
- **`<match>` subjects stay scalar/enum.** A fact query MAY appear in a `when`/`test`
  **guard**, but a `<match on>` subject MUST remain an enum/bool/scalar path so
  exhaustiveness (§11.2) stays decidable. Relations are guard-only, never a match subject
  (**`E-MATCH-RELATION-SUBJECT`**). This is the discipline that preserves the crown-jewel
  static guarantees over the new layer (§10).

### 8.1 Evaluation instants (normative)

- `::assert`/`::retract` share `::set`'s **sequential in-flow visibility**: a later
  directive or guard in the SAME arm sees an earlier `::assert`/`::retract` from that arm
  (document order, within one arm — the ordinary `::set` rule extended to facts).
- Every **condition surface** — `when`, `<match on>`, `<when test>`, `<on when>`, quest
  `start`/`fail`/objective `done` — reads the **derived fixpoint of the same state
  snapshot** its scalar reads use: the `<on>` **pre-event snapshot** (`0.2 §4.2`) extends to
  the fact DB and ITS derived fixpoint, not just scalar state — otherwise the all-match
  determinism guarantee (`0.2 §4.2`) would break (a sibling `<on>` arm's fact writes must
  not leak into another arm's guard for the same event).
- Quest lifecycle predicates (`start`/`fail`/objective `done`, `0.2 §6`) re-evaluate at the
  same engine instants scalar-based predicates do, **plus** after any engine-side assert
  (e.g. a `reserved: true` relation crossing a threshold, §7.3) — an engine-driven fact
  change can complete an objective exactly as an engine-driven scalar write already can.

## 9. Epistemic modeling (who knows / believes what)

Perspective and partial knowledge need **no new primitive** — they are the fact-DB used with
a **holder-first convention** plus derivation (the lens/POV lesson: holder = ordinary
subject; visibility = a derived query over one shared DB; never per-character duplicated
tables).

### 9.1 Ground truth vs belief

Model the world's truth and a holder's belief as **separate relations over the same DB**:

```yaml
relations:
  atLocation:       { args: [character, location], tier: run, key: [0] }               # ground truth: subject at location
  heardLocation:    { args: [character, character, location], tier: run, key: [0, 1] } # hearer, subject, location — a percept
  believesLocation: { args: [character, character, location], derive: true }           # holder believes subject at location
rules:
  - "believesLocation(H, S, L) :- heardLocation(H, S, L)"
  - "believesLocation(S, S, L) :- atLocation(S, L)"          # a character knows their own location
```

- The **holder is the first argument**; a belief relation is just a relation whose subject
  is the epistemic holder. `holds(believesLocation(iroha, halsin, grove))` = "Iroha believes
  Halsin is at the grove."
- **Visibility is derived, not duplicated.** `believes`/`knows` relations are `derive: true`
  rules over declared **base** relations (here `heardLocation`, plus self-knowledge from
  `atLocation`; a real schema adds more event sources — e.g. `witnessed`, `overheard` — each
  a declared base relation with its own rule). One shared fact-DB; each viewpoint is a
  derived projection — you can always ask what a *different* holder would know of the same
  base facts. (bard's spaces-lens baked the observer into the row at write time and thereby
  foreclosed that question; do not repeat it.)

### 9.2 Bounds (normative)

- Belief relations are ordinary derived relations → **total** (§10). The design goal of
  *perpetually-non-convergent* belief (the `idola-perception-lens` scratchpad) is a
  **non-goal** (§13): it requires unbounded/unverifiable state. Take the SoT-plus-belief
  *shape*; reject the anti-convergence machinery.
- Epistemic resolution MUST terminate in **mechanically-derived facts** — never an
  LLM-in-the-loop step (the bard `projectMentalModel` anti-pattern). "What X knows" is a
  Datalog query, not a prompt.

### 9.3 Belief revision (normative)

Two conflicting percepts about the same (holder, subject) pair both stay **valid** by
default: nothing in `::assert` implicitly retracts a different fact, and a rule cannot
prefer "the newer" one — rules have **no access to time** (there is no
`established`/`invalidated` read surface in CEL, and a rule-body CEL guard is
fact-query-free by construction, §7.2/§7.3/B2). Without a key, two `heardLocation` percepts
about where Halsin is would both derive a `believesLocation` fact — correctly modeling
"Iroha holds two contradictory beliefs," but useless for a single-answer query.

`heardLocation`'s `key: [0, 1]` (§9.1, on hearer+subject) is exactly §4's **functional
relation** pattern applied to belief: a new percept about the same (holder, subject) pair
**auto-invalidates** the old one (§4, §5), so at most one `believesLocation` derivation is
live per holder/subject at a time — the **RECOMMENDED default** for percept relations. The
alternative — **engine-owned recency** (a `reserved: true` relation the engine itself
asserts/retracts under whatever policy the game wants, §4) — trades the free default for
engine control outside the Datalog layer. Either way, **temporal preference inside a rule is
a non-goal** (§13): admitting it would require rule bodies to read validity timestamps,
breaking the CEL-guard-is-scalar-only discipline (§7.3) that keeps rules total and
evaluation-order-independent.

### 9.4 Reified events

An n-ary happening ("player gave item to npc at time T") is modeled by **reifying** the
event as an entity of a declared `event` kind, with binary facts attached
(`kind(e42, gift)`, `actor(e42, player)`, `target(e42, npc)`, `item(e42, sword)`). Standard
RDF reification; keeps everything binary-decomposable and closed.

## 10. Totality & static guarantees

The relational layer preserves every 0.1.0/0.2.0 invariant because it is **doubly closed
and function-free**:

- **Finite Herbrand base.** No function symbols (§7.1) → the only terms are declared
  entities/enums/bools → the set of derivable facts is finite and bounded by the schema at
  every evaluation instant (§3.1). Bottom-up derivation reaches a least fixpoint in finitely
  many steps → **every derivation terminates** (Datalog, not Prolog).
- **Safety + stratification** (§7.2) make rules range only over the finite base and give
  negation a well-defined meaning.
- **Reduces to data (§3.2).** A document reduces to: its command records + CEL strings
  (unchanged) + a finite set of entity/enum/relation declarations + a finite seed-fact list
  + a finite rule set + a finite stream of assert/retract deltas. The engine evaluates the
  fixpoint deterministically; nothing is author-iterated.
- **Guards stay scalar-subject** (§7.3, B2) → exhaustiveness (§11.2) and definite-assignment
  (§9.4) are unchanged for the scalar layer; fact queries are guard-only reads, never a
  rule-body dependency.
- **Checker obligations (structural, compile-time — never a lint):**
  - relation/entity-kind/enum closure — `E-RELATION-UNKNOWN`, `E-KIND-NAME-CLASH`,
    `E-ENTITY-KIND-SHAPE`;
  - arity/domain — `E-RELATION-ARITY`, `E-RELATION-EMPTY`, `E-RELATION-DOMAIN`,
    `E-FACT-DOMAIN`, `E-ENTITY-KIND-CLASH`;
  - rule safety, stratification, function-freedom — `E-DATALOG-UNSAFE`,
    `E-DATALOG-UNSTRATIFIED`, `E-DATALOG-FUNCTION`;
  - the rule-guard/fact-query firewall — `E-DATALOG-GUARD-FACT` (B2);
  - write policy across all three relation classes — `E-DERIVED-WRITE`,
    `E-RELATION-RESERVED-WRITE`, `E-FACT-TIER-WRITE`;
  - retract-only wildcards — `E-RETRACT-WILDCARD-ASSERT`;
  - composition signature-matching — `E-USES-DUP-RELATION`, `E-EXTENDS-RELATION-SIG`;
  - derived-relation shape — `E-DERIVE-TIER`, `E-DERIVE-UNDECLARED`, `W-DERIVE-NO-RULES`;
  - the narrative-time type — `E-TEMPORAL-ARG`, `E-VALIDAT-DERIVED`;
  - match-subject discipline — `E-MATCH-RELATION-SUBJECT`.

  (The meta-lesson from every surveyed system: convention-only invariants rot; these are
  compile-time.)

## 11. Interaction with scalar tiers & document kinds

- **Coexistence.** Scalar tiers (§9, `quest.*` `0.2 §5`) are unchanged; the fact-DB is a
  parallel state kernel. A relation declares its owning tier (§4) → same reset boundaries.
- **Magnitudes stay scalar.** Continuous quantities (trust, HP, coins) remain `number`
  scalar state; they enter facts only **discretized** via a CEL-guard rule (§7.3). Where a
  scaled ordinal must round-trip, model it as **sibling paths** — `run.trust.v: number`,
  `run.trust.of: number` — the nekg `{v,of}` *shape*, adopted; not a `{v,of}` *literal*.
  0.1.0's state model is `number|bool|enum` only (§9.3), with no record/struct type, so the
  pair is two independently-declared scalar paths sharing a prefix, never one composite
  scalar.
- **Kinds.** Every document kind (`scene`, `quest`, and future reward/achievement/daily) may
  assert/retract and query facts; `<on when>` and quest `start`/`fail`/objective `done`
  predicates MAY use the full §8 query surface — `holds`/`count`/`validAt` — subject to
  §6's base-relation restriction on `validAt`. Quest sequencing (`0.2 §6` start/fail/done
  conditions) may now gate on facts this way; control flow is unchanged — diverts remain
  forward-only (§14).

## 12. Governance & schema evolution

- **Vocabulary provenance.** The relation/entity vocabulary is content the checker treats as
  a closed set (§4). An implementation SHOULD pin the resolved capability + relation
  vocabulary in a **lock file** for reproducible builds (the Terraform-lock lesson; tooling,
  not language).
- **Schema evolution (normative rule, nekg lesson):** a schema carries a version; an
  **additive** relation/arg (optional) is a soft change with a load-time default; a change
  to an existing relation's **arity or arg domains**, an entity kind's `members:`/`open:`
  shape, or a named `enums:` value is a **flag-day hard break**, never a
  nullable/union "compatibility" shim ("that union is the debt"). Renaming a relation, kind,
  or enum is a hard break with a migration note.
- **Diagnostics are a closed, change-controlled set** (Appendix A), each shipping with a
  minimal adversarial fixture (the rnd-flutter one-fixture-per-invariant discipline).

## 13. Non-goals

- **Prolog power** — function symbols, compound-term construction, unbounded recursion. The
  ceiling is **Datalog** (§10). `E-DATALOG-FUNCTION` enforces it.
- **LLM-in-the-loop** resolution of facts/knowledge (§9.2) — violates engine-owns-runtime.
- **Approximate/heuristic scoring** (PageRank, float-threshold bucketing, substring
  matching) as a query primitive — express relevance/depth as bounded rules over declared
  predicates instead.
- **Non-convergent epistemics** (§9.2) — needs unbounded/unverifiable state.
- **Temporal preference inside a rule** (§9.3) — rules have no access to validity
  timestamps; belief revision is modeled at the percept-relation level (`key:` or
  `reserved:`), never inside a rule body.
- **Open/free-text predicate or entity vocabularies** — the nekg failure mode; the schema is
  closed by construction.
- **Numeric value slots inside facts** — facts are symbolic; magnitudes are scalar (§11).
- **Backward diverts** — still permanent (§14); positive Datalog recursion is *derivation*,
  not control-flow divert, and is forward-only in the command stream.

## Appendix B — Worked example (a quest gating on derived + epistemic facts)

Schema (`act1.schema.lute`):

```yaml
entities:
  character: { members: [shadowheart, halsin, player] }
  location:  { members: [camp, grove, moonrise] }

relations:
  inParty:          { args: [character], tier: run }
  atLocation:       { args: [character, location], tier: run, key: [0] }
  captive:          { args: [character], tier: run }
  connected:        { args: [location, location], tier: app }                          # static map topology
  heardLocation:    { args: [character, character, location], tier: run, key: [0, 1] } # hearer, subject, location
  canReach:         { args: [character, location], derive: true }
  believesLocation: { args: [character, character, location], derive: true }           # holder, subject, location

facts:
  - "inParty(shadowheart)"
  - "captive(halsin)"
  - "atLocation(player, camp)"
  - "connected(camp, grove)"
  - "connected(grove, moonrise)"

rules:
  - "canReach(C, L)            :- atLocation(C, L)"
  - "canReach(C, L2)           :- canReach(C, L1), connected(L1, L2)"
  - "believesLocation(H, S, L) :- heardLocation(H, S, L)"
```

Quest pack (`kind: quest`):

```lute
---
kind: quest
luteVersion: "0.3.0"
uses: ../schema/act1.schema.lute
profile: bg-encounter
title: Act 1 — Emerald Grove
---

<quest id="rescueHalsin" title="Rescue the First Druid"
       start="holds(inParty(shadowheart))"
       fail="holds(atLocation(halsin, moonrise))">
  <objective id="reach" title="Reach the grove"
             done="holds(canReach(player, grove))"/>
  <objective id="learn" title="Learn where Halsin is held"
             done="holds(believesLocation(player, halsin, grove))"/>

  <on event="npcSpoke">
    ::assert{ heardLocation(player, halsin, grove) }
    @narrator: "The cages are past the inner grove," Halsin said.
  </on>

  <on event="questComplete">
    ::retract{ captive(halsin) }
    ::assert{ inParty(halsin) }
    ::set{ user.xp += 300 }
  </on>
</quest>
```

`connected` is seeded via the schema `facts:` block (§4), so `canReach` (transitive,
recursive) can actually derive from load — `reach` gates on physically walking the seeded
topology (`atLocation` → `canReach`), independent of ever hearing about Halsin. `npcSpoke`
(a capability world event resolved by `profile: bg-encounter`, `0.2 §4.5`) asserts the
author-side percept `heardLocation`, and does **not** also move the player — `learn` gates
on `believesLocation` alone, so the two objectives are never vacuously co-satisfied by one
action. `questComplete` retracts `captive(halsin)` (seeded, so a valid fact exists to
retract) and asserts `inParty(halsin)` — one `character` id (`halsin`) throughout, never a
second synthetic entity for "captive Halsin" (§3.1's one-id-one-kind rule). All total, all
statically checked.

## Appendix C — Grammar summary (0.3.0 additions)

```
# schema (YAML)
entities:  { <kind>: { members: [<id>…] } | { open: engine }, … }
enums:     { <name>: [<EnumMember>…], … }
relations: { <name>: { args: [<domain>…], tier?: <tier>, derive?: bool, reserved?: bool,
                        key?: [<argIdx>…] }, … }   # tier: forbidden when derive: true
facts:     [ "<Relation>(<Arg>…)", … ]              # seed, checked + materialized at load/reset
rules:     [ "<Head> :- <Body>", … ]                # quoted (§4) — an unquoted `:-` misparses as YAML

Head    ::= Relation( (Var|Const)… )
Body    ::= Literal ("," Literal)*
Literal ::= Relation(Term…) | "not" Relation(Term…) | CelGuard | Term ("="|"!=") Term
Term    ::= Var | Const
CelGuard ::= "cel(" CelString ")"        # global scalar state only — no holds/count/validAt (B2)

# body directives (admitted wherever ::set is)
Assert  ::= "::assert"  "{" Relation "(" Arg ("," Arg)* ")" "}"                (* Arg fully ground *)
Retract ::= "::retract" "{" Relation "(" (Arg|"_") ("," (Arg|"_"))* ")" "}"    (* "_" retract-only *)

# CEL query surface (added to the closed profile, §8.4)
holds(rel(args|_))                 — any subset of slots may be `_`
count(rel(args|_)) — int expr      — distinct valid-tuple count, freely composable
validAt(rel(args|_), T)            — historical holds; base relations / guard-free derived only (§6)
now()                               — the current narrative-time token (§6, §8.4 profile addition)
```
