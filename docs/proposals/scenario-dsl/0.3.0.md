# Lute Scenario DSL — Relational Facts & Datalog Derivation

- **Version:** 0.3.0
- **Status:** Draft (**full-feature** extension of 0.2.0; may later be split into 0.3.x /
  0.4.0 — see §2)
- **Date:** 2026-07-07
- **Scope:** a **relational state kernel** — a closed **SVO / n-ary fact database** over a
  declared finite schema, with **bitemporal validity**, **delta assertion/retraction**, and
  a **total (Datalog) derivation** layer — plus the query surface that lets conditions read
  it, an **epistemic-modeling pattern** over it, and its **governance / schema-evolution**
  rules.
- **Relationship to 0.1.0 / 0.2.0:** an **extension**. [`0.1.0.md`](./0.1.0.md) remains the
  scene kind + shared kernel (logic, Lute-CEL §8, scalar state §9, totality §3);
  [`0.2.0.md`](./0.2.0.md) adds document kinds, `<on>`, `quest.*`. 0.3.0 adds the relational
  fact layer **beside** the scalar tiers. Unqualified `§N` refer to 0.1.0; `0.2 §N` to
  0.2.0.
- **Design provenance:** [`../../superpowers/specs/2026-07-07-lute-0.3.0-prior-art.md`](../../superpowers/specs/2026-07-07-lute-0.3.0-prior-art.md)
  — a survey of `bard` (nekg, chronicle, lens/POV) and `rnd-flutter` (ledger). bard's
  knowledge layer is the blueprint; its open-vocabulary / manual-accumulation /
  convention-only-invariant failures are the control group this spec is designed to prevent.
- **Out of scope:** the runtime engine/SDK; any game-specific relation vocabulary; and
  **Prolog-level power** (function symbols, unbounded term construction) — the ceiling is
  **Datalog** (§10, §13).

The key words **MUST**, **MUST NOT**, **SHOULD**, **MAY** are RFC 2119.

## 1. Abstract

Through 0.2.0, Lute state is **scalar**: typed paths (`number|bool|enum`) in tiered
namespaces (§9). Scalars cannot express *relationships between entities* — "Shadowheart is
in the party," "the player told Halsin about the grove," "faction A is hostile to faction
B" — without the engine pre-flattening each into a bespoke boolean, the pattern the 0.2.0
state-contract (`0.2 §8`) prescribes.

0.3.0 adds a **relational fact database** as a first-class state kernel: a set of **tuples**
over a **declared, closed** vocabulary of **relations** and **entity kinds**, each tuple
carrying a **bitemporal validity interval**. Content **asserts** and **retracts** facts as
deltas; the engine maintains the cumulative view. A **total Datalog derivation** layer
computes new relations from base facts (joins, projection, transitive closure) with
**guaranteed termination** and **static verifiability**. Conditions (CEL, `<match>`,
`<when>`, `<on>`) query the fact set through a **bounded** predicate surface.

The relational layer is **Datalog, not Prolog**: function-free, so the fact universe is the
finite set of declared entities/values (a finite Herbrand base), so every derivation
terminates and every static guarantee Lute already makes (exhaustiveness §11.2,
definite-assignment §9.4, reduction §3.2) extends to it. It is **not** a runtime logic
interpreter authored ad hoc — it reduces, like everything in Lute, to data the engine
evaluates deterministically.

## 2. Status & versioning

0.3.0 is written **full-feature**: the entire relational vision (schema, assert/retract,
temporal validity, Datalog derivation *including stratified recursion*, the query surface,
the epistemic pattern, governance) is specified here as one coherent design. Whether it
ships as one release or is later partitioned (e.g. 0.3.x for the non-recursive core, 0.4.0
for recursive derivation and epistemic tooling) is an **implementation-sequencing decision
deferred to the plan phase**; this document is the design of record for the whole layer.

Relative to 0.2.0 the change is **additive to the state model** (scalar tiers are unchanged;
the fact-DB is new surface) but introduces **new reserved directives and schema forms**
(§4–§7); it is a pre-1.0 minor (breaking allowance per §2 of 0.1.0 is *not* needed —
nothing existing is removed or reinterpreted). A document opts in simply by declaring
relations (§4); a document with no relation declarations behaves exactly as 0.2.0.

## 3. The relational fact model

### 3.1 Entities, relations, facts

```
Entity   ::= a value of a declared entity kind (an id), or an enum member, or a Bool
Relation ::= a declared, named predicate of fixed arity over declared arg domains
Fact     ::= an instance of a relation applied to entities:  rel(a1, …, an)
```

- An **entity kind** is a declared finite domain of ids (e.g. `character`, `faction`,
  `location`, `item`). Entity ids are `CelIdent` (no `-`, §8.4) — they are CEL/path-facing.
  An entity kind `K` doubles as a **unary domain predicate** `K(X)` usable in rule bodies
  ("X is a declared `K`") — the finite domain a variable ranges over, and a valid positive
  safety binding (§7.2).
- A **relation** is a declared predicate with a fixed **arity** and a **typed argument
  signature**; each argument ranges over a declared entity kind, an enum, or `bool`. The
  common binary case is **SVO** (`subject`, `object`); higher arity and reified events
  (§9.3) cover the rest.
- A **fact** is a relation applied to entities of the matching signature. A fact is
  **symbolic** — it either holds or does not. It carries **no numeric value slot**;
  magnitudes live in the scalar tiers (§11.1), discretized into enum-valued relation args
  when they must participate in facts (the nekg `{v,of}` lesson — facts stay symbolic).

The relation vocabulary and every entity kind are **closed**: declared up front, finite,
statically enumerable. This is the doubly-closed property (§10) — Datalog's function-free
closure *and* the game's declared-schema closure — that makes the layer total and checkable.

### 3.2 Bitemporal validity

Every fact carries a validity interval keyed to **narrative time** (the engine-stamped
current position, e.g. an episode/shot token or a monotonic tick — §6):

```
established : <narrative-time>          (* when the fact became true *)
invalidated : <narrative-time> | unset  (* when it ceased to be true; unset = still valid *)
```

A fact is **valid at** query time `T` iff `established ≤ T ∧ (invalidated = unset ∨
invalidated > T)`. **Retraction is a tombstone**, never a deletion: `::retract` sets
`invalidated`; the fact and its history remain (auditability, correct replay). Default
queries (§8) read the **currently-valid** set.

## 4. Fact schema declaration

Relations and entity kinds are declared in the state schema (§9.2 schema documents /
frontmatter `state:`-adjacent), in a new `relations:` block:

```yaml
entities:
  character: { }                 # a declared entity kind (id domain, populated by facts)
  location:  { }
  faction:   { }

relations:
  inParty:   { args: [character], tier: run }                     # unary
  atLocation:{ args: [character, location], tier: run }           # binary (SVO)
  hostile:   { args: [faction, faction], tier: run }              # binary, symmetric by rule
  knowsLoc:  { args: [character, character, location], tier: run } # ternary (holder-first, §9)
```

- **`args`** — the ordered argument signature; each entry names an entity kind, an enum
  type, or `bool`. Arity = `len(args)`. An arg naming no declared domain is
  **`E-RELATION-DOMAIN`**; a relation with no `args` is **`E-RELATION-ARITY`**.
- **`tier`** — the reset boundary that owns this relation's facts (`scene`/`run`/`user`/
  `app`/`quest`, §9.1, `0.2 §5`). The engine clears the relation's facts at that tier's
  reset, exactly as for scalar paths. Default `run` if omitted.
- Relation names and entity-kind names are `CelIdent` (no `-`, §8.4). A duplicate relation
  name is **`E-RELATION-DUP`**.
- The vocabulary is **closed**: asserting/deriving/querying a relation not declared here is
  **`E-RELATION-UNKNOWN`**; applying a relation at the wrong arity/domain is
  **`E-RELATION-ARITY`** / **`E-FACT-DOMAIN`**. (This closure is the fix for nekg's
  open-string-predicate drift; there is no "convention-only" predicate namespace.)

Relation schemas compose through the existing `uses`/`extends` DAG (§9.2): a base may
declare relations a document refines; duplicate-across-unrelated-peers is a conflict, same
as scalar paths.

## 5. Asserting & retracting facts

```
Assert   ::= "::assert" "{" FactPattern "}"
Retract  ::= "::retract" "{" FactPattern "}"
FactPattern ::= Ident "(" Arg ( "," Arg )* ")"      (* Ident = relation; Arg = entity id / enum / bool / {{…}}? no *)
Arg      ::= CelIdent | EnumMember | Bool
```

- `::assert{ inParty(shadowheart) }`, `::assert{ atLocation(shadowheart, grove) }`,
  `::retract{ atLocation(shadowheart, grove) }` — single-line leaf directives (`::` form,
  bracket rule §5 / `0.2 §D4`: pure leaves, never nestable). One fact per directive.
- Args are **ground** (concrete entity ids / enum members / bools) — assertion is a
  **delta**, not a rule. A base (asserted) fact's args MUST match the relation signature
  (`E-FACT-DOMAIN` / `E-RELATION-ARITY` otherwise).
- **Semantics.** `::assert` adds the fact with `established = <now>`, `invalidated = unset`
  (idempotent: re-asserting a currently-valid fact is a no-op). `::retract` sets
  `invalidated = <now>` on the currently-valid matching fact (retracting an absent fact is a
  no-op). The engine owns the merge and the timestamps — **authors write deltas; the engine
  derives the cumulative, time-scoped view** (the chronicle model; the nekg anti-pattern was
  manual accumulation).
- **Where.** `::assert`/`::retract` are admitted wherever `::set` is (scene content,
  `<on>`/quest/objective arms, `0.2 §3.3`). They are the relational analogue of `::set`.
- **Write policy.** A **derived** relation (has `derive` rules, §7) MUST NOT be asserted or
  retracted by content — it is engine-computed (**`E-DERIVED-WRITE`**, the analogue of the
  reserved-state rule `0.2 §5.4`). Base relations are author-writable; derived relations are
  read-only.

## 6. Temporal validity & narrative time

The engine stamps `established`/`invalidated` with the **current narrative-time token**; the
language never writes them directly. Ordering of facts is by this engine-stamped monotonic
token **only** — never by any author-supplied field (the rnd-flutter `run_seq`-vs-`round`
lesson: author-supplied ordinals are non-monotonic and MUST NOT rank state).

Queries (§8) default to **valid-now**. A condition MAY ask for historical validity via the
guard form `validAt(rel(args), T)` where `T` is a narrative-time expression (advanced;
`E-TEMPORAL-ARG` if `T` is not a narrative-time value). The append-only + pure-recompute
model (§3.2) makes rewind/replay/save-load queries trivially correct — derived state is a
pure function of the asserted deltas up to `T`.

## 7. Datalog derivation

Derived relations are computed from base (and other derived) relations by **rules** declared
in the schema:

```yaml
relations:
  ally:      { args: [faction, faction], tier: run, derive: true }
  canReach:  { args: [character, location], tier: run, derive: true }

rules:
  # symmetric + reflexive-free alliance is the complement of hostility, symmetric
  - ally(A, B)      :- faction(A), faction(B), not hostile(A, B), not hostile(B, A), A != B
  # transitive reachability (recursion — terminates: finite Herbrand base)
  - canReach(C, L)  :- atLocation(C, L)
  - canReach(C, L2) :- canReach(C, L1), connected(L1, L2)
```

### 7.1 Rule form

```
Rule ::= Head ":-" Body
Head ::= Relation "(" Var ( "," Var )* ")"                 (* a derived relation *)
Body ::= Literal ( "," Literal )*
Literal ::= Relation "(" Term ( "," Term )* ")"            (* positive atom *)
          | "not" Relation "(" Term ( "," Term )* ")"      (* negated atom (stratified) *)
          | CelGuard                                        (* a CEL condition, §7.3 *)
          | Term ( "=" | "!=" ) Term                        (* (in)equality *)
Term ::= Var | CelIdent | EnumMember | Bool                (* Var = uppercase; no function symbols *)
```

- A `rules:` entry defines one Horn clause. Multiple rules for the same head **union**
  (disjunction). The head relation MUST be declared `derive: true`
  (**`E-DERIVE-UNDECLARED`** for a rule head that isn't; **`E-DERIVED-WRITE`** conversely).
- **No function symbols.** Terms are variables or declared constants only — no compound
  terms, no arithmetic constructing new values. This is the Datalog restriction that keeps
  the Herbrand universe finite and every rule terminating (§10). Using a would-be function
  term is **`E-DATALOG-FUNCTION`**.

### 7.2 Safety, stratification, termination (normative)

- **Safety:** every variable in the **head**, and every variable in a **negated** or
  **CEL-guard** literal, MUST also appear in a **positive** body atom. An unbound variable
  is **`E-DATALOG-UNSAFE`**. (Guarantees derived facts range only over the finite base.)
- **Stratified negation:** the rule set's predicate-dependency graph MUST have no cycle
  through a `not` edge — recursion through negation is **`E-DATALOG-UNSTRATIFIED`**.
  Positive recursion (e.g. `canReach`) is allowed and total.
- **Termination:** because the schema is function-free and finite, the least fixpoint is
  reached in finitely many steps (bottom-up evaluation over a finite Herbrand base). No
  author-declared recursion cap is required for termination; an implementation MAY expose a
  cost budget, but termination is a **structural** property, not a convention (the chronicle
  lesson: bake the guarantee into the checker, don't lint it).
- Derivation is a **pure function** of the currently-valid base facts (§3.2). It is
  recomputed by the engine; content never sees a partially-updated view.

### 7.3 CEL-guard bridge (facts ↔ scalars)

A rule body MAY contain a **CEL guard** over **global scalar state** (§9), plus (in)equality
on the rule's bound variables, bridging the two state models **totally**:

```yaml
rules:
  # a site counts as reachable-in-act-1 only once the run has entered act 1 (global scalar gate)
  - act1Site(L)  :- location(L), site(L), cel("run.act == 1")
```

- The guard is ordinary Lute-CEL (§8.4, closed profile); it reads **global** scalar tiers
  and must be safe (§7.2). It binds no head variable (introduces no term), preserving safety
  and termination, and is read-only/side-effect-free like all CEL.
- **Per-entity magnitudes stay out of rules.** A rule guard does NOT index scalars by a rule
  variable (there is no `run.trust[A][B]` form). When a *per-entity* magnitude must become a
  fact (e.g. "trust ≥ ally threshold"), the **engine** asserts the discretized fact
  (`trustTier(a, b, ally)`) as it crosses the threshold — the engine→state contract
  (`0.2 §8`) — and rules/queries read that fact. Facts stay symbolic; magnitudes stay scalar.

## 8. Querying facts in conditions

Conditions — `when`, `<match on>`, `<when test>`, `<on when>`, `::set` RHS guards — read the
fact-DB through a **bounded** predicate surface added to the Lute-CEL profile (§8.4):

```
holds(rel(args))              — the fact is valid now (args ground → membership test)
holds(rel(a, _))             — bounded existential: ∃ some entity in that slot (a valid fact)
count(rel(a, _)) OP n        — bounded count over a declared finite domain (aggregation)
validAt(rel(args), T)        — historical validity (§6)
```

- `holds` is the primary query: `when="holds(inParty(shadowheart))"`,
  `when="holds(hostile(grove, absolute))"`. Ground args = membership; a `_` slot = bounded
  existential over that arg's declared entity kind (finite → total).
- `count(rel(a, _)) >= n` is **bounded aggregation** over a declared finite domain (e.g.
  "party has ≥ 3 clerics": `count(hasClass(_, cleric)) >= 3`). Bounded → total. This is the
  *total* replacement for the CEL comprehensions the profile forbids (§8.4
  `E-CEL-PROFILE`) — and the engine no longer needs to pre-flatten aggregates into scalars
  (`0.2 §8` boilerplate goes away).
- These predicates are the **only** fact-query functions admitted; a bare relation reference
  or an unbounded/collection-returning form is **`E-CEL-PROFILE`** (still closed). `holds`
  over an undeclared relation is `E-RELATION-UNKNOWN`; wrong arity is `E-RELATION-ARITY`.
- **`<match>` subjects stay scalar/enum.** A fact query MAY appear in a `when`/`test`
  **guard**, but a `<match on>` subject MUST remain an enum/bool/scalar path so
  exhaustiveness (§11.2) stays decidable. Relations are guard-only, never a match subject
  (**`E-MATCH-RELATION-SUBJECT`**). This is the discipline that preserves the crown-jewel
  static guarantees over the new layer (§10).

## 9. Epistemic modeling (who knows / believes what)

Perspective and partial knowledge need **no new primitive** — they are the fact-DB used with
a **holder-first convention** plus derivation (the lens/POV lesson: holder = ordinary
subject; visibility = a derived query over one shared DB; never per-character duplicated
tables).

### 9.1 Ground truth vs belief

Model the world's truth and a holder's belief as **separate relations over the same DB**:

```yaml
relations:
  atLocation:       { args: [character, location], tier: run }              # ground truth: subject at location
  heardLocation:    { args: [character, character, location], tier: run }   # hearer heard subject is at location
  believesLocation: { args: [character, character, location], tier: run, derive: true } # holder believes subject at location
rules:
  - believesLocation(H, S, L) :- heardLocation(H, S, L)
  - believesLocation(S, S, L) :- atLocation(S, L)          # a character knows their own location
```

- The **holder is the first argument**; a belief relation is just a relation whose subject
  is the epistemic holder. `holds(believesLocation(iroha, halsin, grove))` = "Iroha believes
  Halsin is at the grove."
- **Visibility is derived, not duplicated.** `believes`/`knows` relations are `derive: true`
  rules over declared **base** relations (here `heardLocation`, plus self-knowledge from
  `atLocation`; a real schema adds more event sources — e.g. `witnessed`, `overheard` — each
  a declared base relation with its own rule). One shared fact-DB; each viewpoint is a
  derived projection — you can always ask what a *different* holder would know of the same
  base facts. (bard's spaces-lens baked the observer into the row at write time and thereby
  foreclosed that question; do not repeat it.)

### 9.2 Bounds (normative)

- Belief relations are ordinary derived relations → **total** (§10). The design goal of
  *perpetually-non-convergent* belief (the `idola-perception-lens` scratchpad) is a
  **non-goal** (§13): it requires unbounded/unverifiable state. Take the SoT-plus-belief
  *shape*; reject the anti-convergence machinery.
- Epistemic resolution MUST terminate in **mechanically-derived facts** — never an
  LLM-in-the-loop step (the bard `projectMentalModel` anti-pattern). "What X knows" is a
  Datalog query, not a prompt.

### 9.3 Reified events

An n-ary happening ("player gave item to npc at time T") is modeled by **reifying** the
event as an entity of a declared `event` kind, with binary facts attached
(`kind(e42, gift)`, `actor(e42, player)`, `target(e42, npc)`, `item(e42, sword)`). Standard
RDF reification; keeps everything binary-decomposable and closed.

## 10. Totality & static guarantees

The relational layer preserves every 0.1.0/0.2.0 invariant because it is **doubly closed
and function-free**:

- **Finite Herbrand base.** No function symbols (§7.1) → the only terms are declared
  entities/enums/bools → the set of derivable facts is finite and bounded by the schema.
  Bottom-up derivation reaches a least fixpoint in finitely many steps → **every derivation
  terminates** (Datalog, not Prolog).
- **Safety + stratification** (§7.2) make rules range only over the finite base and give
  negation a well-defined meaning.
- **Reduces to data (§3.2).** A document reduces to: its command records + CEL strings
  (unchanged) + a finite set of relation declarations + a finite rule set + a finite stream
  of assert/retract deltas. The engine evaluates the fixpoint deterministically; nothing is
  author-iterated.
- **Guards stay scalar-subject** (§8) → exhaustiveness (§11.2) and definite-assignment
  (§9.4) are unchanged for the scalar layer; fact queries are guard-only reads.
- **Checker obligations (structural, compile-time — never a lint):** relation closure
  (`E-RELATION-UNKNOWN`), arity/domain (`E-RELATION-ARITY`/`E-FACT-DOMAIN`), rule safety
  (`E-DATALOG-UNSAFE`), stratification (`E-DATALOG-UNSTRATIFIED`), function-freedom
  (`E-DATALOG-FUNCTION`), derived-vs-base write policy (`E-DERIVED-WRITE`), and match-subject
  discipline (`E-MATCH-RELATION-SUBJECT`). (The meta-lesson from every surveyed system:
  convention-only invariants rot; these are compile-time.)

## 11. Interaction with scalar tiers & document kinds

- **Coexistence.** Scalar tiers (§9, `quest.*` `0.2 §5`) are unchanged; the fact-DB is a
  parallel state kernel. A relation declares its owning tier (§4) → same reset boundaries.
- **Magnitudes stay scalar.** Continuous quantities (trust, HP, coins) remain `number`
  scalar state; they enter facts only **discretized** via a CEL-guard rule (§7.3). Where a
  scaled ordinal must round-trip, use the `{ v, of }` meter shape (nekg lesson) in scalar
  state, not a fact.
- **Kinds.** Every document kind (`scene`, `quest`, and future reward/achievement/daily)
  may assert/retract and query facts; `<on when>` and quest `start`/`fail`/objective `done`
  predicates MAY use `holds`/`count` (§8). Quest prerequisite graphs (`0.2 §6`) expressed
  over facts remain forward-only (§14).

## 12. Governance & schema evolution

- **Vocabulary provenance.** The relation/entity vocabulary is content the checker treats as
  a closed set (§4). An implementation SHOULD pin the resolved capability + relation
  vocabulary in a **lock file** for reproducible builds (the Terraform-lock lesson; tooling,
  not language).
- **Schema evolution (normative rule, nekg lesson):** a schema carries a version; an
  **additive** relation/arg (optional) is a soft change with a load-time default; a change
  to an existing relation's **arity or arg domains** is a **flag-day hard break**, never a
  nullable/union "compatibility" shim ("that union is the debt"). Renaming a relation is a
  hard break with a migration note.
- **Diagnostics are a closed, change-controlled set** (Appendix A), each shipping with a
  minimal adversarial fixture (the rnd-flutter one-fixture-per-invariant discipline).

## 13. Non-goals

- **Prolog power** — function symbols, compound-term construction, unbounded recursion. The
  ceiling is **Datalog** (§10). `E-DATALOG-FUNCTION` enforces it.
- **LLM-in-the-loop** resolution of facts/knowledge (§9.2) — violates engine-owns-runtime.
- **Approximate/heuristic scoring** (PageRank, float-threshold bucketing, substring
  matching) as a query primitive — express relevance/depth as bounded rules over declared
  predicates instead.
- **Non-convergent epistemics** (§9.2) — needs unbounded/unverifiable state.
- **Open/free-text predicate or entity vocabularies** — the nekg failure mode; the schema is
  closed by construction.
- **Numeric value slots inside facts** — facts are symbolic; magnitudes are scalar (§11).
- **Backward diverts** — still permanent (§14); positive Datalog recursion is *derivation*,
  not control-flow divert, and is forward-only in the command stream.

## Appendix A — Diagnostics (0.3.0 delta)

| Code | Stage | Meaning |
|---|---|---|
| `E-RELATION-UNKNOWN` | check | assert/derive/query of an undeclared relation (§4). |
| `E-RELATION-DUP` | check | duplicate relation declaration (§4). |
| `E-RELATION-ARITY` | check | relation used at the wrong arity (§4, §5, §8). |
| `E-RELATION-DOMAIN` | check | relation arg names no declared entity kind/enum/bool (§4). |
| `E-FACT-DOMAIN` | check | asserted fact's arg not a member of its arg domain (§5). |
| `E-DERIVED-WRITE` | check | `::assert`/`::retract` of a `derive:true` relation (§5). |
| `E-DERIVE-UNDECLARED` | check | a rule head relation not declared `derive:true` (§7.1). |
| `E-DATALOG-FUNCTION` | check | a compound/function term in a rule (§7.1). |
| `E-DATALOG-UNSAFE` | check | head/negated/guard variable not bound by a positive atom (§7.2). |
| `E-DATALOG-UNSTRATIFIED` | check | recursion through negation (§7.2). |
| `E-MATCH-RELATION-SUBJECT` | check | a `<match on>` subject that is a relation query (§8). |
| `E-TEMPORAL-ARG` | check | `validAt(...)` second arg is not a narrative-time value (§6). |

`E-UNDECLARED`, `E-USES-DUP-*` (relation vocabulary composition), and the full kernel
registry.

## Appendix B — Worked example (a quest gating on derived + epistemic facts)

Schema (`act1.schema.lute`):

```yaml
entities:
  character: {}
  location:  {}
relations:
  inParty:          { args: [character], tier: run }
  atLocation:       { args: [character, location], tier: run }
  connected:        { args: [location, location], tier: app }     # static map topology
  heardLocation:    { args: [character, character, location], tier: run }   # hearer, subject, location
  canReach:         { args: [character, location], tier: run, derive: true }
  believesLocation: { args: [character, character, location], tier: run, derive: true } # holder, subject, location
rules:
  - canReach(C, L)        :- atLocation(C, L)
  - canReach(C, L2)       :- canReach(C, L1), connected(L1, L2)
  - believesLocation(H, S, L) :- heardLocation(H, S, L)
```

Quest pack (`kind: quest`):

```lute
---
kind: quest
luteVersion: "0.3.0"
uses: ../schema/act1.schema.lute
---

<quest id="rescueHalsin" title="Rescue the First Druid"
       start="holds(inParty(shadowheart))"
       fail="holds(atLocation(halsin, moonrise))">
  <objective id="reach" title="Reach the grove"
             done="holds(canReach(player, grove))"></objective>
  <objective id="learn" title="Learn where Halsin is held"
             done="holds(believesLocation(player, halsin, grove))"></objective>

  <on event="npcSpoke" when="holds(heardLocation(player, halsin, grove))">
    ::assert{ atLocation(player, grove) }
    :narrator: "The cages are past the inner grove," Halsin said.
  </on>

  <on event="questComplete">
    ::retract{ inParty(halsin_captive) }
    ::assert{ inParty(halsin) }
    ::set{ user.xp += 300 }
  </on>
</quest>
```

Here `canReach` (transitive, recursive) and `believesLocation` (epistemic, holder-first) are
engine-derived; the quest gates on them via `holds`; `<on>` arms assert/retract base facts;
scalar reward stays `::set`. All total, all statically checked.
