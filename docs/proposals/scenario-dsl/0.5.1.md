# Lute Scenario DSL — Trace Preview of Quest-Gated Reads & Authoring-Surface Honesty

- **Version:** 0.5.1
- **Status:** Draft (**compatible refinement** of 0.5.0)
- **Date:** 2026-07-13
- **Scope:** close the one cross-cutting gap the second-round hands-on review surfaced, plus
  the small authoring-surface honesty items alongside it. No new grammar, no changed
  `check`/`compile` verdict, no IR-shape change. This revision is entirely about `trace`
  preview fidelity and the discoverability of already-shipped surface:
  (§1) `trace` can now PREVIEW a document's reads of the reserved quest lifecycle paths
  (`quest.<id>.state`, `quest.<id>.objectives.<oid>.done`) — both by accepting them as
  explicit `--state` mocks when the document references them, and by resolving an un-mocked
  reserved read to its **default** instead of silently matching no arm;
  (§2) `lute context` lists the reserved quest paths a document actually references;
  (§3) the `{mono}` / `{os}` / `{vo}` delivery flags are documented and surfaced;
  (§4) `trace` reports an event that matched no `<on>` handler instead of a silent no-op, and
  the `E-COMPONENT-PARSE` child-diagnostic count is accurate.
- **Relationship to prior revisions:** a **compatible refinement** of
  [`0.5.0.md`](./0.5.0.md) (which added diagnostic specificity, trace honesty, and context
  completeness). Unqualified `§N` refers to **this document**; `0.1 §N` … `0.5 §N` cite prior
  revisions. `0.5.0`'s §3 "trace honesty" framing is the direct motivation: 0.5.0 made a scene
  *read* foreign quest state freely (its D-fix), but `trace` was never extended to let a writer
  *preview* what that read does — this revision finishes that thread.
- **Design provenance:** a second adversarial hands-on review (a narrative writer and a systems
  designer independently authored 0.5.0 scenarios). Both converged on one major finding — a
  quest-gated scene `<match>`/`<when>` cannot be previewed by `trace` — reproduced two distinct
  ways (a refused `--state` reserved mock; an un-mocked reserved read matching no arm). §1
  closes both. The smaller §2–§4 items are the same review's minor findings.
- **Out of scope:** the runtime engine/SDK; any Datalog evaluation in `trace`; new grammar
  productions. A single-file `check`/`trace` cannot verify that a referenced quest `<id>` exists
  (a scene does not import its quests) — existence is validated only where the whole-project
  quest graph is available (`check-project`, §1.4, which this revision adds); single-document
  tools do not guess at cross-document existence, they note it as unverified (§1.3).
- **Inline examples are authoritative.** Every example here is 0.5.1 syntax.

The key words **MUST**, **MUST NOT**, **SHOULD**, **MAY** are RFC 2119.

## 1. `trace` preview of reserved quest-path reads

`0.2 §5.2` reserves `quest.<id>.state` (enum `active | complete | failed`, plus `unset` before
the quest is known) and `quest.<id>.objectives.<oid>.done` (`bool`, **default `false`**) as
implicitly-declared paths any document MAY read (`0.5.0` made cross-document reads work with no
import ceremony). `trace` (`0.4 §4`) previews a document in isolation and never runs the engine
(D1); when the traced document is a *scene* (no `<quest>` construct of its own), it has no walk
that derives those reserved values. 0.5.0 left two observable failures here; §1.1/§1.2 fix them.

### 1.1 Reserved quest paths are seedable when the document references them

`trace` MUST accept a `--state` (or `--mock` `state:`) assignment to a reserved quest path
**when the traced document references that exact path** (in a `<match on=…>`, a `<when>`/`when=`
guard, an interpolation, or any CEL slot), so the author can preview the arm that read selects.
The admitted values are the path's declared domain:

- `quest.<id>.state` — one of `active`, `complete`, `failed`, `unset`.
- `quest.<id>.objectives.<oid>.done` — `true` or `false`.

A `--state` on a reserved path the document does **not** reference remains
`E-TRACE-MOCK-UNDECLARED` (you can only preview a read the document actually makes). Because
standalone `trace` is a single-document preview with no cross-document quest catalog, it does
**not** validate that the quest `<id>` exists; to keep this honest rather than typo-safe,
standalone `trace` MUST attach the "existence unverified" note (§1.3) to **every** admitted
`--state` mock or defaulted read of a reserved foreign `quest.<id>.*` path — an
explicitly-mocked typo is surfaced, never silent. Project-context validation of the `<id>` is
`check-project`'s job (§1.4). A malformed value (outside the domain above) is `E-TRACE-MOCK-*`
as for any typed mock.

```lute
# scene reads quest.kaiFavor.state in a <match>; previewing the "complete" arm:
$ lute trace bar-scene.lute --state quest.kaiFavor.state=complete --choose …
  <match quest.kaiFavor.state>   -> complete   # the arm the author wanted to preview
```

### 1.2 An un-mocked reserved read resolves to its default, never "no arm"

When a reserved quest path is read but neither mocked (§1.1) nor derived by an in-document
`<quest>` walk, `trace` MUST resolve it to the path's **default**, matching the checker's
definite-assignment (`0.2 §5.2`):

- `quest.<id>.objectives.<oid>.done` → **`false`** (its schema default; the checker treats it as
  definitely-assigned, so a `<match>` with exhaustive `is="true"` / `is="false"` arms is
  check-clean and MUST fire the `false` arm at trace, never match no arm).
- `quest.<id>.state` → **`unset`** (its pre-activation value; a `<match on="quest.<id>.state">`
  covering `unset` — required by `0.2 §11.2` — fires that arm).

This removes the 0.5.0 defect where a check-clean, exhaustive objective-`done` match silently
resolved to neither arm (`"outcome": "no arm"`) at trace time.

### 1.3 "Existence unverified" note

Whenever `trace` admits a `--state` mock for (§1.1) or defaults (§1.2) a reserved `quest.<id>.*`
path that is not derived by an in-document `<quest>` (a foreign quest read), it MUST emit one
informational note per such quest `<id>`: that the quest's existence is unverified by `trace`
(run `check-project` to confirm it is defined by a project quest, §1.4), and — for a defaulted
read — that the value is its default (`done=false` / `state=unset`), overridable via `--state`.
`trace` has no cross-document quest catalog, so this note fires on every such read, mocked or
defaulted — an explicitly-mocked typo is therefore never silent. Informational only — never an
error, never a reachability claim, exit code unchanged.

### 1.4 `check-project` validates referenced quest ids

Where the whole-project quest graph is available — `lute check-project <dir>` — the checker MUST
verify that every referenced `quest.<id>` (and `quest.<id>.objectives.<oid>`) resolves to a quest
(and objective) defined by some quest document in the walked project. A reference to a quest
`<id>` (or objective `<oid>`) that no project quest defines is a **`W-QUEST-REF-UNKNOWN`**
warning naming the referencing document + path — the honest catch for the mistyped-quest-id
case (a scene reading `quest.heits.state` for a quest actually named `heist`). It is a WARNING,
not an error: the reference is shape-legal, and the quest may be defined outside the walked
directory or added later. Single-file `lute check` (no project graph) does not and cannot emit
it — which is exactly why standalone `trace` notes such reads as existence-unverified (§1.3).

## 2. `lute context` lists referenced reserved quest paths

`lute context <file>` (`0.5.0 §5`) MUST include, in both human and `--json` output, the reserved
`quest.<id>.state` / `quest.<id>.objectives.<oid>.done` paths the document **references** —
alongside the ordinary `stateSchema`, clearly marked as reserved/implicit. A writer using
`context` to answer "what may I legally read here" (as the tutorial recommends) currently sees a
scene's reserved quest reads omitted, and would wrongly conclude they are invalid. Paths the
document does not reference are not listed (the reserved namespace is unbounded).

## 3. Delivery flags `{mono}` / `{os}` / `{vo}` are documented and surfaced

The content-line delivery flags (`0.2.2`/`0.4` surface, previously demonstrated only for
`{mono}`) are hereby documented normatively:

- **`{mono}`** — interior monologue / thought (not spoken aloud in-scene).
- **`{os}`** — off-screen: the speaker is heard but not currently staged/visible.
- **`{vo}`** — voiceover: narration-style delivery layered over the scene.

They are **mutually exclusive** — at most one per content line (`E-DELIVERY-CONFLICT` on two),
and none is allowed on `narrator` (`E-DELIVERY-NARRATOR`), unchanged from `0.4`. `lute context`
MUST list the available delivery flags in its authoring surface (human + `--json`), and the
`docs/getting-started-first-scene.md` tutorial SHOULD mention `{os}`/`{vo}` alongside its
existing `{mono}` example so all three are discoverable from the documented surface.

## 4. Trace event feedback & accurate component diagnostic count

- **Unmatched `--event`.** A `--event <name>` that matches no `<on>` handler in the traced
  document MUST produce an informational note ("event `<name>` matched no `<on>` handler") rather
  than a silent no-op, so a typo'd event name is distinguishable from an intentionally-unhandled
  one. Informational only; exit code unchanged. (A `--event` naming a built-in lifecycle event is
  still `E-TRACE-EVENT`, `0.4 §4.3`, unchanged.)
- **`E-COMPONENT-PARSE` child count.** The "(N issue(s))" count in an `E-COMPONENT-PARSE` import
  failure MUST equal the number of child diagnostics surfaced (`0.5.0 §2.2`), including children
  that reclassify (e.g. a component-body state error) — no under-count.

## Appendix A — Diagnostic codes

One new **warning** code: **`W-QUEST-REF-UNKNOWN`** (§1.4) — a `check-project` reference to a
quest/objective id that no project quest defines. No new *error* codes. §1.3 and §4's
unmatched-`--event` output are **informational notes** (the same non-error channel as
`0.5.0 §3.1`), not diagnostics. `E-TRACE-MOCK-UNDECLARED` is **narrowed** (it no longer fires on
a reserved quest path the document references, §1.1) but its identity and message are unchanged.
No code is removed; no decided `check`/`compile` verdict of a well-formed document changes.

## Appendix B — Version stamp

`LUTE_LANG_VERSION` and `LUTE_IR_VERSION` advance to `"0.5.1"` (`lute-compile/src/lib.rs`), the
sole permitted compiled-artifact diff for a document using no post-0.5.0 vocabulary. The insta
golden snapshots and version-envelope tests are re-accepted for the version-line change only, in
the same commit as the constant bump.
