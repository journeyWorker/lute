## 1. Abstract

Through 0.3.0 the language invested in **static guarantees**: totality (`0.1 ¬ß3`),
exhaustiveness (`0.1 ¬ß11.2`), definite assignment (`0.1 ¬ß9.4`), the closed CEL profile
(`0.1 ¬ß8.4`), the doubly-closed relational layer (`0.3 ¬ß10`). 0.4.0 invests those guarantees
back into the **writer**. Nothing here adds runtime power, opens the CEL profile, or moves
the engine boundary; every feature is either tooling over the existing semantics or sugar
that reduces to the existing records.

Five deltas:

1. **`lute trace`** (¬ß4) ‚Äî an authoring-time explainer: given a document plus **explicit,
   author-supplied mock state and facts**, it reports which `<match>` arm fires, which
   `<branch>`/`<hub>` choices are enabled, which `<objective>` predicates are satisfied, and
   a deterministic linear trace of one chosen path ‚Äî the Ink/Twine/Yarn ‚Äúplaytest a branch
   without the engine‚Äù workflow. It is a **dev aid under a hard quarantine** (¬ß4.2): it
   feeds neither `check` nor `compile`, proves nothing, and runs no engine machinery.
2. **Reachability diagnostics** (¬ß5) ‚Äî the softlock `0.2 ¬ß6.3` describes in prose but never
   diagnoses becomes a checked error, alongside provably-dead `<match>`/`<branch>`/`<hub>`
   arms. Errors are **provable-only**: a precisely-defined decision fragment (¬ß5.1) with no
   false positives; everything weaker is a warning or a `trace` coverage report.
3. **Param-scoped `<match>` in component bodies** (¬ß6) ‚Äî a component can now *dispatch on
   its own params* (‚Äúthe NPC's response varies with the affinity passed in‚Äù), closing the
   reuse gap that today forces hand-duplicated match blocks across episodes, while keeping
   the component state-purity contract intact ‚Äî `<branch>`/`<hub>`/writes stay forbidden.
4. **Ceremony sugar** (¬ß7) ‚Äî quantified against the flagship examples: the gated line drops
   from 8 lines to 1 (`when=` on a content line). The audit also surfaced a silent trap in
   the remembered-choice form ‚Äî `into=` without `persist=` records nothing today ‚Äî which is
   now *warned* (`W-CHOICE-INTO-NO-PERSIST`), not reinterpreted: implying persist would
   change existing documents' meaning (¬ß7.3). Every old form stays valid; the sugar degrades
   to the same command records; the line-`code` identity invariant (`0.1 ¬ß12`) is untouched.
5. **Diagnostic presentation** (¬ß8) ‚Äî a CEL syntax error MUST be phrased in the writer's
   terms, never leaked backend parser text; and one root cause MUST NOT pile on as N
   diagnostics ‚Äî same-root occurrences collapse to one primary with covered occurrences.

## 2. Status & versioning

0.4.0 is written **full-feature**: all five deltas are specified here as one coherent
revision; implementation sequencing is a plan-phase decision, not a spec split.

Relative to 0.3.0 the change is **additive to the grammar** ‚Äî zero new productions. `<match>`
inside a component body and `when=` on content lines live inside existing forms (`Attrs` is
an open key set, `0.1 ¬ß4.5`); `lute trace` is CLI surface. The
checker gains new codes (¬ß5, ¬ß6, ¬ß7) and two presentation contracts (¬ß8); the compiler
gains only desugaring (¬ß7.4). One
narrow pre-1.0 breaking allowance is exercised for provably-defective documents ‚Äî see ¬ß3.

## 3. Backward compatibility (normative)

0.4.0 makes the following guarantees to the existing corpus:

- **B1 ‚Äî Parse-identical.** There are no new grammar productions and no reinterpreted ones;
  every 0.3.0-valid document parses to the same tree under 0.4.0. New surface is limited to:
  the `when` content-line attr key (¬ß7.2), param-subject `<match>` admission inside
  component bodies (¬ß6), and the `trace` subcommand (¬ß4).
- **B2 ‚Äî Compile-identical (modulo the version stamp, D13).** For any document that uses no
  0.4.0 sugar, the compiled artifact is **byte-identical except its version stamp**: the only
  permitted diff is the `LUTE_LANG_VERSION`/`LUTE_IR_VERSION` fields advancing to `"0.4.0"`
  (`lute-compile/src/lib.rs`); no IR record shape changes (¬ß6.4, ¬ß7.4), and the identity pass
  (`code` back-fill, `lineId`, `voiceKey` ‚Äî `0.1 ¬ß12`) is untouched. A frontmatter `luteVersion`
  is a universal key only, never validated against the consts.
- **B3 ‚Äî Check-clean is preserved for non-defective documents.** A 0.3.0-valid document
  checks clean under 0.4.0 **unless** it contains a defect ¬ß5 newly proves: a provably-dead
  arm/choice, a provably-unsatisfiable objective, or a provably-unreachable quest. Each such
  document was already broken under previously-documented semantics ‚Äî first-match-wins made
  the arm dead (`0.1 ¬ß11.2`); `0.2 ¬ß6.3` explicitly named the unreachable-completion
  softlock ‚Äî 0.4.0 only *reports* it. This is the pre-1.0 breaking allowance (`0.1 ¬ß2`)
  exercised at its narrowest: no construct changes meaning; only provably-authored-in-error
  documents are newly rejected. The shipped example corpus (`docs/examples/`) checks clean
  under ¬ß5.
  (A newly-emitted ¬ß7.3/¬ß8 *warning* never changes a verdict ‚Äî B4 ‚Äî so such a document still
  "checks clean" in the exit-`0` sense; only the ¬ß5 *errors* above can newly fail a verdict.)
- **B4 ‚Äî Warnings and presentation are not compatibility surface.** ¬ß8 changes diagnostic
  *message text*, collapse counts, and adds warning codes. Message text was never stable API
  (`0.1` Appendix E: crates own exact messages); the machine format changes are additive
  (¬ß8.3).
- **B5 ‚Äî `trace` is invisible.** Running or not running `lute trace` MUST NOT affect any
  check verdict or compiled artifact (¬ß4.2). A toolchain without `trace` remains conforming
  for check/compile purposes.

## 4. `lute trace` ‚Äî authoring-time preview

### 4.1 Model

`lute trace` answers the writer's question *‚Äúif the state were X, what would this scene
do?‚Äù* without the engine. The author supplies **all** runtime answers as mocks ‚Äî scalar
state, facts, chosen menu options, fired events ‚Äî and `trace` walks the document once,
deterministically, reporting every decision it takes and **why** (the guard, the values it
read), plus which arms/choices the walk never visited.

`trace` is the third leg of the authoring loop ‚Äî `check` proves, `compile` emits, `trace`
*explains* ‚Äî and the weakest of the three by design: it holds **no** authority (¬ß4.2).

### 4.2 The D1 quarantine (normative)

Lute's foundational boundary (D1) is: **Lute declares; the engine executes** ‚Äî the Datalog
fixpoint, capability bridges, dice, rendering, and every runtime evaluation belong to the
engine (`0.3 ¬ß10`, `0.2 ¬ß8`). `trace` introduces the first expression evaluator anywhere in
the toolchain, so the boundary is restated here as hard conformance rules:

1. **`trace` MUST NOT feed `check` or `compile`.** No diagnostic, no check verdict, and no
   artifact byte may depend on whether `trace` ran or on anything it produced.
2. **`trace` output is NEVER a static guarantee.** Nothing `trace` reports may be cited,
   cached, or consumed as proof of reachability, coverage, or correctness. The ¬ß5 codes are
   the only static reachability surface, and they are computed by the checker alone.
3. **`trace` MUST NOT execute engine machinery.** It runs no Datalog fixpoint (a
   `derive: true` relation is never computed ‚Äî ¬ß4.3), no capability bridge, no dice, no
   scheduler. Every answer the engine would compute is either **supplied as a mock** or
   reported **unknown**. The engine remains the sole runtime authority.
4. **Isolation is structural, not conventional.** The evaluator MUST live in an
   authoring-only module that the checker and compiler cannot import. In the reference
   workspace this is a new terminal crate (e.g. `lute-trace`) wired only into the CLI
   binary: `lute-cel` stays parse-only (it holds no evaluator today and MUST NOT gain one);
   adding the trace crate to `lute-check`'s or `lute-compile`'s dependencies is a
   conformance violation, reviewable as a one-line manifest diff.
5. **The evaluated subset is closed.** ¬ß4.3 defines exactly what `trace` computes; an
   implementation MUST NOT silently widen it (e.g. ‚Äúconveniently‚Äù deriving relations or
   modeling narrative time). Widening the subset is a spec revision.

### 4.3 Mock-input surface & the evaluated subset

```
lute trace <file>
  [--state <path>=<literal>]‚Ä¶        scalar seed: a DECLARED state path and a literal
  [--fact  "<rel>(<arg>‚Ä¶)"]‚Ä¶         a ground fact, valid-now, over the declared vocabulary
  [--choose <branchOrHubId>=<choiceId>[,<choiceId>‚Ä¶]]‚Ä¶   menu selections, in order
  [--event <name>]‚Ä¶                  fire a capability/WORLD event, in CLI order (NOT the
                                     built-in lifecycle events ‚Äî those are engine-derived, ¬ß4.4)
  [--accept <questId>]‚Ä¶              simulate accepting a start-less (engine-driven) quest (¬ß4.4)
  [--mock <file.yaml>]               the same four surfaces as a YAML document
  [--json]                           machine output (¬ß4.5)
  [--providers <DIR>] [--project <DIR>]   resolution flags, exactly as `lute check`
```

The `--mock` file carries `state:` (a map of path ‚Üí literal), `facts:` (a list of quoted
ground facts, the `0.3 ¬ß4` `facts:` shape), `choose:` (a map of branch/hub id ‚Üí choice id or
list), and `events:` (a list). CLI flags compose with the file; on a conflict the flag wins
(facts union; a flag `choose` replaces that id's file entry).

**Mock validation (structural, before the walk).** Trace resolves the document exactly as
`lute check` does ‚Äî same schema/import/capability resolution ‚Äî and MUST refuse a document
with check **errors** (exit 1; run `check` first). Mocks are validated against the resolved
schema: an undeclared `--state` path is **`E-TRACE-MOCK-UNDECLARED`** (state-by-typo MUST
fail in mocks exactly as in documents, `0.1 ¬ß11.1.1`); a literal incompatible with the
declared type is **`E-TRACE-MOCK-TYPE`**; a `--fact` naming an undeclared relation, wrong
arity, or a foreign arg is **`E-TRACE-MOCK-FACT`** (the `0.3` `E-RELATION-UNKNOWN` /
`E-RELATION-ARITY` / `E-FACT-DOMAIN` rules applied at the mock site, under one code); a
`--choose` naming an unknown branch/hub/choice id is **`E-TRACE-CHOICE`**. Pre-walk
validation is **structural only** ‚Äî ids, arity, types, declaredness; a forced choice's
*guard* is deliberately NOT evaluated here, because eligibility depends on in-flow writes
the walk has not applied yet (¬ß4.4). A `--fact` MAY name a
`derive: true` or `reserved: true` relation: a mock is a *supplied answer*, not a content
write ‚Äî `E-DERIVED-WRITE`/`E-RELATION-RESERVED-WRITE` govern documents, not mocks.

A `--event` naming a built-in lifecycle event (`questActive`/`questComplete`/`questFailed`) is
**`E-TRACE-EVENT`** ‚Äî those are engine-derived transitions, not world events a writer fires
(¬ß4.4): a `start`-having quest activates declaratively, a `start`-less one via `--accept`. An
`--accept` naming an unknown quest id, or a quest that carries a `start` predicate (which
activates declaratively and needs no accept), is **`E-TRACE-ACCEPT`**.

**Effective state.** A path reads, in order: a trace-applied write (below) ‚Üí the mock seed ‚Üí
the schema `default` ‚Üí `unset`. Facts read: the mock fact set as modified by trace-applied
`::assert`/`::retract` deltas, all valid-now.

**The evaluated subset.** `trace` evaluates a CEL expression to `true`, `false`, a scalar, or
**`unknown`**, under three-valued (Kleene) logic:

- **Evaluated:** the closed profile operators (`0.1 ¬ß8.4`) over concrete values ‚Äî logic,
  comparison, arithmetic, `?:`, `in` over list literals, indexing; `has()`/`isSet()` are
  **definite** against the effective state ‚Äî `true` iff an effective value exists (trace-write ‚Üí
  mock seed ‚Üí schema `default`), `false` on unset (D19; a bare *value* read of an unset path is
  `unknown`, below); `$` substitution and `@def` macro expansion (`0.1 ¬ß8.1` ‚Äî expansion is
  static, then the expanded text evaluates); `holds(pattern)` / `count(pattern)` as a
  **bounded scan of the supplied fact set** ‚Äî ground positions match, `_` positions are
  existential over the finite supplied set. This is pattern lookup, NOT derivation.
- **Unknown (mock-only):** a read of any path with no effective value; `holds`/`count` over a
  `derive: true` relation **unless** the derived fact itself was supplied via `--fact` (the
  rules are never run); `validAt(‚Ä¶, T)` and `now()` ‚Äî narrative time is engine-minted
  (`0.3 ¬ß6`) and has no mock surface; any engine-populated reserved path not seeded
  (`quest.<id>.state` and `‚Ä¶objectives.*.done` are the exception: trace derives them from
  its own walk, ¬ß4.4).
- **Propagation:** `unknown` propagates by K3 ‚Äî `false && unknown = false`,
  `true || unknown = true`, otherwise `unknown`; a comparison/arithmetic node with an
  `unknown` operand is `unknown`; `?:` with an `unknown` condition is `unknown`.

### 4.4 Walk semantics

The walk is document-ordered and applies writes as it goes (`::set`, and `::assert`/
`::retract` on the mock fact set ‚Äî the sequential in-flow visibility rule, `0.3 ¬ß8.1`). A
`::set` whose RHS is `unknown` marks the path `unknown`.

- **`<match>`** ‚Äî arms top-to-bottom (`0.1 ¬ß11.2`): a `true` arm fires (its body is walked);
  a `false` arm is skipped; an `unknown` arm **halts the trace at that match** ‚Äî trace
  reports the unresolved atoms (which paths/facts a mock would need) and exits incomplete
  (exit 3). Trace never guesses past an unknown guard. A **fourth outcome** exists beyond
  fire/skip/halt: when **every** arm's guard decides **`false`** (all provably ruled out, none
  `unknown`) and there is no `<otherwise>`, **no arm fires** ‚Äî trace records the match as
  `no arm` (coverage `0/total`), annotates it in the transcript, and the walk **continues**.
  This is knowledge, not uncertainty (distinct from an `unknown` halt), and legal: it is the
  same "conditionally nothing" shape the `when=` sugar produces (¬ß7.2), so it is neither an
  error nor a warning.
- **`<branch>`** ‚Äî eligibility is each choice's `when` evaluated **at the presentation
  point**, against the effective state ‚Äî earlier in-flow `::set`/`::assert` writes applied
  (`0.3 ¬ß8.1`) ‚Äî so a choice enabled only by an earlier write is never wrongly refused
  (`unknown` is reported as *unknown, not eligible*). The `--choose` entry for the branch id
  selects the arm; forcing a choice whose guard evaluates **false at that point** is
  `E-TRACE-CHOICE`; forcing past an `unknown` guard is permitted (that is what mocks are
  for) and reported as forced. Absent `--choose`, trace takes the **first eligible choice in
  document order** and marks it `(auto)`; if **no** choice is true-eligible and at least one
  guard is `unknown`, trace **halts incomplete** (exit 3, D20) rather than guess past the
  unresolved eligibility ‚Äî the writer supplies a mock to resolve it. (`E-BRANCH-ALL-GUARDED`
  makes this state unreachable for a check-clean branch; trace degrades honestly regardless.)
- **`<hub>`** ‚Äî with a `--choose` list, the selections are taken in order; eligibility ‚Äî
  `when`, `once`, `exit` ‚Äî is re-evaluated **before each selection** against the
  then-effective state (`0.1 ¬ß11.1.3`), so an arm enabled by a prior arm's write is honored;
  a forced selection ineligible at its presentation point is `E-TRACE-CHOICE`. Absent a
  list, trace visits
  each eligible non-`exit` choice once in document order, re-evaluating eligibility after
  each arm, then takes the first eligible `exit` (or auto-exits) ‚Äî but if no choice is
  true-eligible and at least one guard (a `when`, `once`, or the `exit`'s own guard) is
  `unknown`, trace cannot tell whether the hub continues or auto-exits, so it **halts
  incomplete** (exit 3, D20) rather than guess. Termination is structural
  (`E-HUB-NO-EXIT`, `0.1 ¬ß7.3.2`). Hub/branch recording (`scene.choices.*`,
  `scene.visited.*`) is applied to the effective state as the engine would.
- **`<timeline>`** ‚Äî clips are reported in resolved-offset order (`0.1 ¬ß11.4`); no clock is
  simulated.
- **Quest kinds** ‚Äî `quest.<id>.state` starts `unset`. **Activation has two forms** (`0.2 ¬ß6.3`):
  a quest **with** a `start` predicate activates **declaratively** ‚Äî trace evaluates `start`, and
  `true` transitions it to `active`; a quest **without** `start` is **accept-driven** and stays
  inactive (reported as *awaiting accept*) until a matching **`--accept <questId>`** simulates the
  player/engine accepting it. Declarative auto-activation and accept are distinct paths to the same
  `unset ‚Üí active` transition, which fires the built-in **`questActive`** handlers **exactly once**.
  `questActive`/`questComplete`/`questFailed` are engine-derived lifecycle events, **never**
  user-supplied via `--event` (`E-TRACE-EVENT`, ¬ß4.3). `start` deciding `false` never activates
  (reported); `unknown` leaves the quest and its objectives unresolved. Then each `--event` (a
  capability/world event only), in CLI order: all matching `<on>` guards are evaluated against the
  **pre-event snapshot** (`0.2 ¬ß4.2`), then the matching arms run in document order. After every
  event and after activation, objective `done` predicates are re-evaluated (monotonic ‚Äî once
  `true`, recorded), `fail` is evaluated **before** derived completion (`0.2 ¬ß6.3` precedence), and
  each lifecycle transition (complete/fail) fires its own handlers **once**. The final report tables
  each quest's state and each objective's status (`done` / `pending` / `unknown`).
- **Components** ‚Äî `::use` invocations are expanded exactly as compile expansion binds them
  (`0.1 ¬ß13.2`, ¬ß6.4); a param-scoped `<match>` (¬ß6) dispatches on the bound args.

### 4.5 Output contract

- **Deterministic.** For identical inputs (document, mocks, flag order), output MUST be
  byte-identical. All iteration is document-ordered; JSON keys are emitted in a fixed order.
- **Human form** ‚Äî an indented, ordered transcript: emitted content lines (interpolations
  substituted where decided, kept verbatim `{{‚Ä¶}}` where unknown), staging directives, state
  writes, and one line per **decision** ‚Äî the construct, the winning arm/choice, and the
  guard with its read values. A trailing summary reports decisions taken, arms/choices
  visited vs. total (coverage), and unresolved atoms if any.
- **Machine form (`--json`)** ‚Äî one JSON object; the top-level keys `file`, `seeds`,
  `steps`, `decisions`, `unresolved`, and `coverage` are normative; implementations MAY add
  keys. `decisions[]` entries carry the construct kind, its id/span, the outcome, and the
  evaluated guard bindings; `unresolved[]` carries the span, expression, and the atoms that
  need mocks.
- **Exit codes** ‚Äî `0` complete trace; `1` refused (document has check errors, or invalid
  mocks ‚Äî the `E-TRACE-*` codes render exactly as check diagnostics do); `2` I/O; `3` trace
  incomplete (an `unknown` guard halted the walk).

### 4.6 Worked example

Against [`docs/examples/choice-persist.lute`](../../examples/choice-persist.lute) (the
52-line remembered-choice example):

```
$ lute trace docs/examples/choice-persist.lute --choose sofaHelp=help
trace: choice-persist.lute  (seeds: 0 paths, 0 facts; 1 selection)
  ## Shot 1.
  @narrator   Sofia had slipped on the wet step. You could help ‚Äî or walk on.
  <branch sofaHelp>   eligible: help, warmly, tip   ‚Üí help (--choose)
    @sofia    Thank you. I won't forget this.
    ::set     run.metHelpfully = true          (persist sugar)
  ## Shot 2.
  <match run.metHelpfully>   = true ‚Üí arm 1 ($ == true)
    @sofia    You helped me back then. I've been meaning to thank you again.
trace complete: 2 decisions; choices visited 1/3 (sofaHelp), arms 1/2 (match run.metHelpfully)
```

The same document with no selection auto-picks `help` `(auto)`; with
`--state run.metHelpfully=true --choose sofaHelp=tip`, Shot 2 still takes arm 1 ‚Äî the trace
shows the `::set` from `tip`'s arm writing `run.tip = 5` and the seeded `true` surviving,
which is precisely the ‚Äúwhy did this arm fire?‚Äù explanation the critique asked for. A quest
document traces as:

```
$ lute trace quests/act1.lute --fact "inParty(shadowheart)" --event npcSpoke
trace: act1.lute
  <quest rescueHalsin>  start: holds(inParty(shadowheart)) = true ‚Üí active
  <on npcSpoke>  guard: (none) ‚Üí fires
    ::assert  heardLocation(player, halsin, grove)
    @narrator  "The cages are past the inner grove," Halsin said.
  objective learn   done: holds(believesLocation(player, halsin, grove)) = unknown
    unresolved: believesLocation is derive:true ‚Äî supply it as a mock, e.g.
    --fact "believesLocation(player, halsin, grove)"
trace incomplete: 1 unresolved atom (exit 3)
```

The derived relation is **not** computed (¬ß4.2 rule 3, ¬ß4.3): the writer previews the
derivation's *consequence* by mocking its output, never by running rules.

## 5. Provable reachability & softlock diagnostics

`0.2 ¬ß6.3` documents the softlock ‚Äî ‚Äúa required objective whose `when` can be permanently
false makes completion unreachable‚Äù ‚Äî but nothing diagnoses it; and first-match-wins
(`0.1 ¬ß11.2`) makes a subsumed later arm silently dead. Today the checker's only
reachability-adjacent checks are local and structural (`E-HUB-NO-EXIT`,
`E-BRANCH-ALL-GUARDED`, and the literal-overlap warning `W-OVERLAP-ARMS`); no dead-arm,
unsatisfiable-objective, or unreachable-quest analysis exists. 0.4.0 adds one, under a hard
boundary:

> **Boundary (normative).** An **error** in this section fires only on a **provable** defect
> ‚Äî provable within the decision fragment of ¬ß5.1, which is sound in every runtime state.
> Anything requiring runtime knowledge, or that could false-positive, is a **warning** or is
> surfaced only by `lute trace` coverage (¬ß4.5) ‚Äî never an error. A ¬ß5 error emitted for a
> document that has a satisfying run is a conformance **bug**, not a tuning matter.

### 5.1 The decided fragment (normative)

A condition expression is **statically decided** to a constant iff the rules below reduce it
to one. `decide()` runs after `@def` macro expansion (`0.1 ¬ß8.1` ‚Äî expansion is a static,
hygienic text substitution the checker performs itself) and consults only the merged,
post-composition schema snapshot:

- **R1 (literals).** A literal decides to itself.
- **R2 (domain membership).** Let `S` be a **finite-domain subject**: a declared `enum` or
  `bool` state path, the `$` token bound to one, an implicit `scene.choices.*` /
  `quest.<id>.state` domain, or an `enum`/`bool` component param (¬ß6.3). Then `S == lit`
  with `lit` outside `S`'s domain decides **false**; `S != lit` with `lit` outside the
  domain decides **true**; `S in [‚Ä¶]` with an empty intersection between the list and the
  domain decides **false**. (`unset` is a domain member only where the subject is
  maybe-unset, `0.1 ¬ß11.2`; a defaulted path and a bound param are never `unset`.)
- **R3 (ground operations).** An operator applied to decided operands decides by the
  operation's semantics: numeric comparison/arithmetic over literals, string/bool/enum
  equality.
- **R4 (connectives).** `!d` decides by negation; `a && b` decides **false** if either
  conjunct decides false, **true** if both decide true; `a || b` dually; `c ? x : y` with a
  decided `c` decides to the decision of the chosen branch.
- **R5 (everything else is undecided).** A state-path read, `isSet()`/`has()`, any fact
  query (`holds`/`count`/`validAt`), `now()`, an engine-reserved path, an interpolation ‚Äî
  all undecided. No assumption about runtime values is ever made.

**Soundness.** Every rule preserves the expression's value in **every** reachable runtime
state (R2 by domain closure ‚Äî the schema is the single source of truth for both checker and
engine, `0.1 ¬ß9.3`; the rest by literal semantics). A decided constant therefore *is* the
runtime value, on every run: the fragment cannot false-positive.

**Closure.** A conforming checker MUST implement exactly R1‚ÄìR5 and MUST NOT emit any ¬ß5
error on the strength of reasoning outside them (no SAT, no interval or path-sensitive
narrowing, no cross-shot state flow). This pins behavior identically across implementations
‚Äî stronger analyses are a spec revision, not an implementation liberty. (This is the same
discipline the existing exhaustiveness engine already follows: its arm-coverage analysis is
deliberately conservative, under-covering rather than guessing.)

**Corpus fixture (normative).** The implementation of this section MUST run the existing
example corpus (`docs/examples/`) through the ¬ß5 checks as a fixture. Any hit is triaged on
its merits ‚Äî a genuine latent softlock fixes the *example*; an over-aggressive check fixes
the *check* ‚Äî an example is never silenced or special-cased to force a corpus-clean result.

### 5.2 Dead arms, dead choices, dead `<otherwise>`

Within one `<match>` / `<branch>` / `<hub>` (all analysis in this section is local to one
construct ‚Äî no cross-construct graph):

- **`E-WHEN-LITERAL-DOMAIN`** (error) ‚Äî an `is` literal outside the subject's decided finite
  domain: a foreign enum member (a typo ‚Äî `is="platnum"` against `[fail, bronze, silver,
  gold]`), a number/bool literal against a mismatched domain, or `unset` on a subject that is
  never unset (a defaulted path; a component param, ¬ß6.3). Such an arm can never fire, and
  the cause is the literal itself. Today this typo surfaces ‚Äî if at all ‚Äî as a confusing
  `E-NONEXHAUSTIVE` on the whole match; it now gets its own diagnostic at the literal.
- **`E-ARM-DEAD`** (error) ‚Äî an arm or choice that can provably never fire, for either of two
  causes, each named in the message:
  1. **Decided-false guard.** The arm's `test` (or a `<choice when>`, in `<branch>` or
     `<hub>`) decides **false** under ¬ß5.1 ‚Äî e.g. `test="1 > 2"`, or
     `when="$ == 'gone'"` against a domain without `gone`.
  2. **Subsumption.** A `<when>` whose entire `is` set is contained in the union of the
     `is` sets of **earlier, unguarded** (`test`-less) sibling arms ‚Äî first-match-wins
     (`0.1 ¬ß11.2`) means it can never be reached: `<when is="gold|silver">` followed by
     `<when is="gold">` (with or without a `test` ‚Äî a guard cannot resurrect a subsumed
     pattern). Guarded earlier arms never count toward subsumption (their guards might be
     false), and partial overlap stays the existing warning `W-OVERLAP-ARMS`.
- **`W-OTHERWISE-DEAD`** (warning) ‚Äî an `<otherwise>` that is provably unreachable because
  earlier **unguarded** `is` arms already cover the subject's whole domain (including
  `unset` where applicable). This is provable, but a defensive `<otherwise>` is a legitimate
  hedge against schema evolution (a widened enum makes it live again, `0.3 ¬ß12`) ‚Äî hence a
  warning, the boundary discipline in action.

No other dead content exists in a document, by construction: control flow is forward-only
with no diverts (`0.1 ¬ß14`), so guarded arms/choices and the quest lifecycle (¬ß5.3) are the
only places unreachability can hide.

Interaction: an arm flagged `E-ARM-DEAD` MUST NOT additionally produce `W-OVERLAP-ARMS` or
coverage-derived diagnostics ‚Äî the dead-arm error is the root (¬ß8.2 C4). `E-ARM-DEAD` on a
choice does not relax `E-BRANCH-ALL-GUARDED`/`E-HUB-NO-EXIT`, which keep their existing,
purely-structural definitions.

### 5.3 Unsatisfiable objectives, unreachable quests

- **`E-OBJECTIVE-UNSATISFIABLE`** (error) ‚Äî an `<objective>` whose `done` predicate decides
  **false** (¬ß5.1). The objective can never complete on any run. If the objective is
  required (non-`optional`), the enclosing quest can never complete either
  (`0.2 ¬ß6.3` derived completion); that consequence is reported as a **note on this
  diagnostic**, not as a second error (¬ß8.2 C4).
- **`E-QUEST-UNREACHABLE`** (error) ‚Äî a `<quest>` that can provably never complete, for
  either of two standalone causes, each named in the message:
  1. `start` decides **false** ‚Äî the quest never activates; every objective, handler, and
     arm inside it is dead content.
  2. `fail` decides **true** ‚Äî `fail` takes precedence over completion (`0.2 ¬ß6.3`), so an
     activated instance fails at the first evaluation instant; completion is unreachable.
- **`W-OBJECTIVE-HIDDEN`** (warning) ‚Äî a **required** objective whose `when` (visibility
  gate, `0.2 ¬ß6.4`) decides **false**. This is exactly the `0.2 ¬ß6.3` softlock prose made
  checkable ‚Äî the objective is provably never visible or tracked, yet still gates
  completion. It is a warning, not an error, because `done` is evaluated independently of
  visibility: completion may still be reachable, so unreachability is not proven. The
  message carries `0.2 ¬ß6.3`'s own advice: mark it `optional` or fix the gate.

What stays outside the static surface ‚Äî deliberately: ‚Äúno engine event sequence ever
satisfies this `done`‚Äù, ‚Äúthis guard is never true in any reachable playthrough‚Äù, ‚Äúthese two
quests deadlock each other‚Äù. All of these require runtime knowledge (engine writes,
capability events, fact assertion order) and would false-positive as static claims; they are
`lute trace` territory ‚Äî the coverage summary (¬ß4.5) reports never-visited arms/choices
under the author's own mock scenarios, with no pretense of proof.

### 5.4 Worked example

```lute
<match on="scene.serve.debut.rank">          /* enum [fail, bronze, silver, gold] */
  <when is="gold | silver">  @bianca: A fine service.          </when>
  <when is="gold">           @bianca: Perfect, even!           </when>
  <when is="platnum">        @bianca: ...                      </when>
  <otherwise>                @bianca: Practice makes rhythm.   </otherwise>
</match>
```

```
ep.lute:3:3: error [E-ARM-DEAD] arm can never fire: its pattern `gold` is fully covered
  by the earlier unguarded arm at 2:3 (`gold | silver`) ‚Äî first-match-wins (dsl 0.4 ¬ß5.2)
ep.lute:4:9: error [E-WHEN-LITERAL-DOMAIN] `platnum` is not a member of the subject's
  domain [fail, bronze, silver, gold] (dsl 0.4 ¬ß5.2)
```

```lute

  <objective id="catch" done="run.rank == 'platnum'"/>
</quest>
```

```

  activate (dsl 0.4 ¬ß5.3)
q.lute:2:3: error [E-OBJECTIVE-UNSATISFIABLE] `done` is provably false: `'platnum'` is not
  a member of `run.rank`'s domain [fail, bronze, silver, gold]; the objective ‚Äî and, being
```

## 6. Param-scoped `<match>` in component bodies

### 6.1 The reuse gap

A component body is **presentational** (`0.1 ¬ß13.4`): lines, staging, `@param` refs ‚Äî no
state read/write, no logic blocks. The checker enforces this as `E-COMPONENT-BODY` over
every logic/write construct (`<branch>`, `<match>`, `<timeline>`, `<hub>`, `<on>`,
`<objective>`, `::set`, `::assert`, `::retract`). ‚ÄúStaging‚Äù is itself qualified by
**effects**: a directive whose *resolved* capability declares state or bridge-result writes
(an `effects.writes[]` entry, a bridge that fills an `into=`-style result path, or a
lowering that desugars to a state write) IS a state write under this contract, whatever its
name ‚Äî only a directive with no declared writes is presentational.

The critique's finding: the most common reusable beat ‚Äî *‚Äúask the NPC something; the
response varies with the affinity you pass in‚Äù* ‚Äî is exactly a `<match>` over a value the
caller already has, and the ban forces every episode to hand-duplicate the dispatch. The
flagship corpus demonstrates it: the rank-tier dispatch in
`docs/examples/showcase/episode01.lute` (lines 98‚Äì110) and the affect-tier dispatch in
`docs/examples/idola-project/date-minigame.lute` (lines 53‚Äì65, 69‚Äì76) are the same
three-arm reaction block, re-typed with different thresholds ‚Äî precisely what one component
taking a `tier` param would express once.

0.4.0 relaxes the ban for **exactly one construct in exactly one shape**: a `<match>` whose
subject and guards range over **the component's own params**. Dispatch on a param is a pure
read of an invocation argument ‚Äî it touches no ambient state and records nothing.
`<branch>` and `<hub>` remain forbidden **on principle**, not caution: presenting a menu
records the selection into `scene.choices.*` / `scene.visited.*` (`0.1 ¬ß11.1`, `¬ß9.6`) ‚Äî a
state **write** ‚Äî so a choice construct inside a component can never satisfy the purity
contract. That is a language boundary, not a v1 gap (¬ß9).

### 6.2 Admitted form (normative)

Inside a component body, a `<match>` is admitted iff:

- **Subject.** `on` is exactly a bare param reference ‚Äî `on="@tier"` ‚Äî naming a declared
  param. Any other subject is rejected: a subject reading ambient state is
  **`E-COMPONENT-STATE`** (below); any other expression shape (a compound over params, a
  literal) is **`E-COMPONENT-BODY`** (not an admitted form ‚Äî dispatch needs a domain).
- **Arms.** `<when>` arms use `is` patterns and/or `test` guards as everywhere
  (`0.1 ¬ß7.3.1`); `$` is in scope bound to the param. Every reference inside a `test` MUST
  resolve to `$` or a declared param (`@budget`, expanded as 0-arity refs); the profile
  (`0.1 ¬ß8.4`) applies unchanged. Any state-path read, fact query (`holds`/`count`/
  `validAt`), or `now()` inside a subject, guard, arm body slot, or `{{‚Ä¶}}` interpolation of
  the component is **`E-COMPONENT-STATE`**.
- **Arm bodies.** Exactly the node set a component body admits elsewhere: content lines,
  **purely-presentational** staging directives (next bullet), `::use` (acyclic,
  `E-COMPONENT-CYCLE`), and ‚Äî recursively ‚Äî further param-scoped `<match>` blocks. A param
  MAY be forwarded to a nested component (`::use{component="x" tier=@tier}`).
- **Directive effects.** Anywhere in a component body ‚Äî an ordinary position or a
  param-scoped `<match>` arm ‚Äî a directive whose resolved manifest/lowering declares state
  writes or bridge-result writes (`effects.writes[]`, bridge semantics, or a desugaring to a
  state write) is **`E-COMPONENT-STATE`**: it depends on or affects ambient state and is NOT
  covered by the staging allowance (¬ß6.1). A directive with no declared writes stays
  admitted.
- **Invocation surface (caller side).** Purity constrains the component **body**, never the
  value a caller binds. A `::use` arg MAY be any type-correct caller-side value: a literal,
  a forwarded `@param` (inside another component), or a `@def` ref whose CEL reads the
  **caller's** own declared state ‚Äî the caller owns its state, and binding a state-derived
  value to a param is precisely how ‚Äúthe reaction varies with the caller's affinity‚Äù is
  expressed dynamically (¬ß6.4 case 2, ¬ß6.5). The existing arg checks apply unchanged
  (`E-COMPONENT-ARG`, `0.1 ¬ß13.3`).
- **Everything else is unchanged.** `<branch>`, `<hub>`, `<timeline>`, `<on>`,
  `<objective>`, `::set`, `::assert`, `::retract` in a component body remain
  **`E-COMPONENT-BODY`**. The code *narrows* ‚Äî it no longer fires for the admitted `<match>`
  shape ‚Äî and its message now names the admitted exception.

`E-COMPONENT-STATE` is new and deliberately distinct from both `E-COMPONENT-BODY` (a
construct that is never admitted) and the incidental `E-UNDECLARED` a component-body state
read produces today (misleading ‚Äî the path may be perfectly declared *in the consuming
scene*; the defect is that a component may not depend on ambient state at all). With ¬ß6 in
place, an ambient reference anywhere in a component body reports `E-COMPONENT-STATE`, and
the purity contract finally has its own name.

### 6.3 Exhaustiveness over a param domain (normative)

A param-subject `<match>` carries the full `0.1 ¬ß11.2` exhaustiveness obligation, with the
domain induced by the param's declared type (`0.1 ¬ß8.1` / `¬ß13.1` types):

| param type | domain | obligation |
|---|---|---|
| `bool` | `{true, false}` | coverable by `is` arms; else `<otherwise>` |
| `{ enum: [m‚Ä¶] }` | the declared members | coverable by `is` arms; else `<otherwise>` |
| `number`, `string` | infinite | `<otherwise>` REQUIRED (`E-NONEXHAUSTIVE`) |

An enum param always declares its members inline (a member-less `enum` is a malformed
`params:` type ‚Äî `E-COMPONENT-PARSE`, unchanged). A param is **never `unset`**: every
invocation binds every param (`E-COMPONENT-ARG` enforces count/type, `0.1 ¬ß13.3`), so
`unset` is not a member of any param domain and `is="unset"` on a param subject is
`E-WHEN-LITERAL-DOMAIN` (¬ß5.2). `E-NONEXHAUSTIVE`, `E-MATCH-DUP-OTHERWISE`,
`W-OVERLAP-ARMS`, `E-ARM-DEAD`, and `E-WHEN-LITERAL-DOMAIN` all apply inside component
bodies exactly as at scene level.

### 6.4 Expansion & reduction (normative)

`::use` expansion binds args to params (`0.1 ¬ß13.2`). For a param-scoped `<match>`, the
expansion-time environment decides lowering, using the ¬ß5.1 fragment with bound literal args
substituted as literals:

1. **Static selection.** If, walking arms top-to-bottom, every arm condition **decides**
   under ¬ß5.1 (the common case ‚Äî args are literal at almost every call site), expansion MUST
   replace the whole `<match>` with the selected arm's nodes (or the `<otherwise>` body).
   No match record is emitted; the component costs nothing at runtime. Selection always
   exists ‚Äî exhaustiveness was proven statically (¬ß6.3).
2. **Residual dispatch.** Otherwise (an arg bound to a non-literal value ‚Äî e.g. a def
   `@ref` whose CEL reads the caller's own declared state),
   the `<match>` lowers to an **ordinary match command record** with the substituted subject
   ‚Äî the same record kind a scene-level `<match>` produces, arms and `<otherwise>` intact.

Either way the emitted artifact contains only record shapes that already exist ‚Äî no new IR,
no new engine obligation, and the reduction invariant (`0.1 ¬ß3.2`, `¬ß11.5`) holds. The
choice between 1 and 2 is pinned (MUST), so two conforming implementations emit identical
artifacts.

### 6.5 Worked example

The deduplicated affinity reaction ‚Äî `reaction.component.lute`:

```lute
---
component: reaction
params:
  tier: { enum: [cold, warm, fond] }
---

## Scene 1.

<match on="@tier">
  <when is="fond">
    @bianca{emotion="delighted"}: You remembered! You actually remembered.
  </when>
  <when is="warm">
    @bianca{emotion="content"}: Not bad at all, Mr. Fixer.
  </when>
  <when is="cold">
    @bianca{emotion="neutral"}: ...Shall we begin?
  </when>
</match>
```

No `<otherwise>`: the three arms cover the declared enum, and a param is never `unset`
(¬ß6.3). Each episode now maps its own scalar to the shared vocabulary at the call site ‚Äî the
only part that ever differed between the duplicated blocks:

```lute
<match on="scene.affect.bianca">
  <when test="$ >= 3"> ::use{component="reaction" tier="fond"} </when>
  <when test="$ >= 1"> ::use{component="reaction" tier="warm"} </when>
  <otherwise>          ::use{component="reaction" tier="cold"} </otherwise>
</match>
```

Every `tier` arg here is a literal, so each `::use` expands to exactly one `@bianca` line
(¬ß6.4 case 1) ‚Äî the compiled artifact is indistinguishable from the hand-duplicated
original. Purity is a **body** property (¬ß6.2), not a call-site one: the component itself
may name only its params ‚Äî adding `test="run.affection > 1"` to an arm **inside**
`reaction.component.lute` is `E-COMPONENT-STATE`, and `<branch>` there is still
`E-COMPONENT-BODY` ‚Äî but the **caller** owns its own state and MAY bind a state-derived
value to the param. The `<match>` wrapper above is one way; binding a caller-side def
directly is the other:

```lute
::use{component="reaction" tier=@currentTier}   /* @currentTier: a caller def ‚Äî enum-typed CEL over the caller's state */
```

A non-literal binding like `@currentTier` cannot fold, so the component's `<match>` lowers
to an ordinary residual match record on the substituted subject (¬ß6.4 case 2). Dynamic
dispatch on a caller-supplied value is exactly the supported case ‚Äî still with no new IR.

## 7. Ceremony sugar

### 7.1 The ceremony audit (what actually costs)

The critique's ‚Äútoo much typing‚Äù complaint, audited against the flagship examples, does
**not** land where folklore puts it: `code=` is already optional (every content-line attr
is ‚Äî the checker's known-attr set is `code`, `emotion`, `variant`, `action`, `dialogMotion`,
the delivery flags `mono`/`os`/`vo`, and `as`, none required; missing codes are back-filled
deterministically at compile and persisted by `lute tag`, `0.1 ¬ß12`). The real, mandatory
ceremony concentrates in three everyday beats:

1. **The gated line.** Showing ONE line only when a condition holds costs a minimum of
   **8 lines** today ‚Äî `<match>` + `<when>` + the line + closers + the mandatory
   `<otherwise>` ‚Äî for every single occurrence.
   [`choice-persist.lute`](../../examples/choice-persist.lute) lines 45‚Äì52 spend 8 lines to
   react to one bool; four of `showcase/episode01.lute`'s seven `<match>` blocks are this same
   two-arm gate shape. There is no shorter form anywhere in the corpus.
2. **The remembered choice.** A choice persisted then read back costs **13‚Äì29 lines and
   14‚Äì19 attributes** across the corpus: `choice-persist.lute`'s minimal round-trip is a
   3-choice `<branch>` (11 lines, 15 attrs ‚Äî `persist="run" into="run.‚Ä¶"` on every choice,
   `value=` on two) plus an 8-line read-back `<match>`; `showcase/episode01.lute`'s richest
   (`<branch id="approach">`, lines 122‚Äì134, read back at 146‚Äì180) is 13 lines / 15 attrs of
   branch plus **three** separate read-back matches (27 lines). The `persist="run"` attr is
   pure ritual: `run` is the **only** legal value (`E-PERSIST-TARGET`).
3. **The repeated attr block.** Consecutive lines by one speaker repeat identical
   presentational attrs ‚Äî `bianca-s01ep02.lute` sets `emotion=`/`variant=` on effectively
   every `@bianca` line, unchanged for lines at a stretch.

0.4.0 answers with one sugar and one guard-rail. The gated line gets a first-class form
(¬ß7.2). The remembered choice keeps its explicit `persist=` ‚Äî the tempting implied form
turns out to CHANGE the meaning of documents that are valid today, so it is rejected and the
trap is named instead (`W-CHOICE-INTO-NO-PERSIST`, ¬ß7.3). The third beat is deliberately
deferred (¬ß9). The sugar keeps every old form valid and degrades to the same command records
(¬ß7.4): the precedent is `persist=`/`into=` itself, which desugars to an ordinary `::set`
before any staging or addressing runs (`0.1 ¬ß11.1.1`) ‚Äî desugar early, reuse existing
records, never mint new IR.

### 7.2 The gated line: `when=` on a content line

```
@sofia{when="run.metHelpfully"}: You helped me back then. I've been meaning to thank you.
```

- **Surface.** `when` joins the content-line attr vocabulary (a `CelString` guard ‚Äî the same
  key, type, and closed profile as `<choice when>` and `<on when>`; `$` is NOT in scope,
  matching `<on when>`). It is admitted on any content line, in any kind/context that admits
  the line. Inside a component body the guard may reference only params (¬ß6.2, else
  `E-COMPONENT-STATE`).
- **Semantics.** The line is emitted iff the guard holds at that point in flow ‚Äî exactly
  equivalent to wrapping the single line in a one-arm guard block with an empty alternative.
  Canonically: `@s{when="G"}: T` ‚â° `<match on="G"><when test="$">@s: T</when><otherwise/>
  </match>` ‚Äî and it MUST lower to that same match record: one arm whose test is the guard,
  whose `otherwise` target is the empty join point. (The match record's `otherwise` is
  already an optional jump target; an implicit empty `<otherwise>` is a target with no body
  ‚Äî an existing shape, not a new one.) The exhaustiveness obligation (`0.1 ¬ß11.2`) is
  satisfied by that implicit empty alternative ‚Äî by construction, ‚Äúshow nothing‚Äù is the
  author's stated intent for the else-case of a gated line, which is what distinguishes this
  sugar from an unguarded non-exhaustive `<match>`.
- **Static analysis.** The guard is an ordinary guard: reads inside the line's text
  (`{{‚Ä¶}}`) and attrs are analyzed under it (`0.1 ¬ß9.4` ‚Äî so `when="isSet(run.tip)"` guards
  a `{{run.tip}}` interpolation); the line itself is a may-emit, never a definite one; a
  guard that decides false is `E-ARM-DEAD` (¬ß5.2). All existing attr checks (delivery-flag
  exclusivity, `emotion` domain, ‚Ä¶) apply unchanged.
- **Identity.** The line is the same line: `code` back-fill, `lineId`, `voiceKey`, and
  `lute tag` behavior are byte-for-byte unaffected (¬ß7.4).

Before/after ‚Äî the `choice-persist.lute` read-back, when only the positive beat is wanted:

```lute
/* 0.3.0 ‚Äî 8 lines */                          /* 0.4.0 ‚Äî 1 line */
<match on="run.metHelpfully">
  <when test="$ == true">
    @sofia: You helped me back then.            @sofia{when="run.metHelpfully"}: You helped me back then.
  </when>
  <otherwise>
  </otherwise>
</match>
```

A gated beat with BOTH a then- and an else-line stays a `<match>` ‚Äî the sugar deliberately
covers only the show-or-skip case (two gated lines with complementary guards would evaluate
the guard twice; the block form remains the right tool, ¬ß9).

### 7.3 The bare-`into=` trap (`W-CHOICE-INTO-NO-PERSIST`)

The audit's obvious second sugar ‚Äî let `into=` alone imply `persist="run"`, since `run` is
the only legal `persist` value (`E-PERSIST-TARGET`, `0.1 ¬ß11.1.1`) ‚Äî is **REJECTED on
backward-compatibility grounds** (¬ß3): today the persist checks engage only when `persist=`
is present, so a 0.3.0 `<choice>` carrying a bare `into=` is a **silent no-op** ‚Äî it parses,
checks clean, and records nothing. Reinterpreting it would silently start writing `run.*`
state in existing documents ‚Äî exactly the meaning-change class B1‚ÄìB3 forbid.

0.4.0 instead names the trap:

- **`W-CHOICE-INTO-NO-PERSIST`** (warning) ‚Äî a `<choice>` (branch or hub) carries `into=`
  without `persist=`: it records nothing. The message states that and offers both exits ‚Äî
  add `persist="run"` to persist the fact, or remove the dead `into=`. Both remedies change
  what the author meant, so **`lute fix` (an automatic, unprompted codemod in this toolchain)
  MUST NOT apply either**: inserting `persist="run"` is a semantic migration (it starts
  writing state), and removing `into=` is an intent judgment. They are surfaced only as
  author-chosen LSP code-actions, or under an explicit opt-in migration flag; the default
  `lute fix` leaves the choice untouched, merely warned.
- **No behavior changes.** `persist=` stays mandatory for persistence, exactly as in 0.3.0;
  every `E-PERSIST-*` check is untouched; the bare-`into=` document still compiles to the
  same (recording-nothing) records ‚Äî it merely stops being *silent*.

An implied-persist form stays open to a future major that can afford the reinterpretation;
0.4.0 deliberately does not take it (¬ß9).

### 7.4 Reduction & identity (normative)

For the ¬ß7.2 sugar:

- It MUST desugar **before** staging/addressing, into a construct that already exists
  (a one-arm match record, ¬ß7.2). The compiled artifact contains no new record kinds and no
  new fields.
- The **line-`code` invariant is untouched**: the sugar never adds, removes, renames, or
  reorders content lines relative to its explicit equivalent, so authored codes, `lute tag`
  back-fill (`max-authored + 10` per speaker, per identity scope), `lineId` derivation, and
  `voiceKey` are identical between a sugared document and its hand-expanded form. VO and
  localization pipelines cannot observe the difference.
- A document using no sugar compiles byte-identically to 0.3.0 (¬ß3 B2).

### 7.5 Quantified

| beat | 0.3.0 cost | 0.4.0 cost | delta |
|---|---|---|---|
| gated single line (`choice-persist` 45‚Äì52) | 8 lines, 2 attrs | 1 line, 1 attr | **‚àí87% lines** |
| bool read-back pair (`episode01` 164‚Äì171) | 8 lines | 2 gated lines | ‚àí75% lines |
| full remembered-choice round-trip (`choice-persist`) | 19 lines / 17 attrs | 12 lines / 16 attrs | **‚àí37% lines** |

The round-trip row applies ¬ß7.2 alone: the 11-line branch is unchanged (its `persist=` stays
mandatory, ¬ß7.3); the 8-line read-back becomes one gated line (the example's `<otherwise>`
beat, if wanted, keeps the block form ‚Äî the table's first row is the honest unit).

## 8. Diagnostic presentation

Two semi-normative quality contracts ‚Äî ‚Äúsemi‚Äù only in that they constrain *presentation*,
not language semantics; the MUSTs below are conformance requirements for any shipping
checker/CLI/LSP.

### 8.1 CEL error translation (the `E-CEL-PARSE` message contract)

Today a CEL syntax error reaches the writer as the embedded backend parser's own text,
verbatim (‚Äúno viable alternative at input ‚Ä¶‚Äù, ‚Äútoken recognition error‚Äù), or ‚Äî on the
backend's panic path ‚Äî as a bare ‚Äúinvalid CEL expression‚Äù. Every other diagnostic in the
toolchain speaks the document's language and cites the spec; the single most common error a
writer makes speaks ANTLR. The contract:

- **T1 (no leakage).** An `E-CEL-PARSE` message MUST be phrased in the document's terms: it
  names the offending token (or the slot, when no token is recoverable) and what a condition
  expected there. It MUST NOT contain backend parser vocabulary ‚Äî grammar rule names,
  ‚Äúviable alternative‚Äù, ‚Äútoken recognition‚Äù, ‚Äúmismatched/extraneous input‚Äù ‚Äî in any output
  surface (CLI human, CLI JSON `message`, LSP).
- **T2 (required translations).** The following writer mistakes MUST be detected and
  answered with their canonical fix (each SHOULD also attach a machine-applicable fixit ‚Äî
  the diagnostic `fixits` field exists and is nearly unused today):

  | authored | message essence |
  |---|---|
  | `run.act = 1` | `=` assigns; comparison is `==` ‚Äî did you mean `run.act == 1`? |
  | `a & b` / `a \| b` | `&`/`\|` are not operators here ‚Äî use `&&`/`\|\|` (or `is="a\|b"` for a literal alternation) |
  | `a and b` / `or` / `not` | words are not operators in a condition ‚Äî use `&&` / `\|\|` / `!` |
  | `x =< 1` / `x => 1` | did you mean `<=` / `>=`? |
  | `"unterminated` | unclosed quote in the condition |
  | *(empty slot)* | the condition is empty |

  The six detections are performed by a **pre-parse lexical scan** of the raw slot text
  (string-mask aware), independent of the backend parser's error taxonomy ‚Äî that is what
  makes this MUST reliably implementable without depending on backend error strings. T3
  remains the fallback for anything the scan does not catch.

- **T3 (fallback).** When no T2 pattern applies ‚Äî including the backend-panic path ‚Äî the
  message MUST be the neutral form ‚Äúnot a valid condition expression‚Äù plus the slot text,
  and SHOULD point at the narrowest span the parser recovered. Never the raw backend text.
- **T4 (unchanged mechanics).** Code, severity, span behavior, and the parse-failure
  suppression of downstream per-slot analyses (¬ß8.2 C2) are unchanged ‚Äî this is a message
  contract only.

### 8.2 Root-cause collapse (the pile-on contract)

One mistake MUST NOT read as N mistakes. Today a single undeclared path used five times
yields five identical errors; a failed schema import yields an error *plus* an `E-UNDECLARED`
for every read the schema would have satisfied. The existing discipline is piecemeal (span-
overlap dedupe for `E-UNDECLARED` producers; exhaustive-subject suppression; the project
pass's covered-occurrence handling for `E-QUEST-ID-DUP`). 0.4.0 makes it a uniform contract:

- **C1 (same-root collapse).** Diagnostics with the same code AND the same root subject ‚Äî
  the undeclared path (`E-UNDECLARED`), ref name (`E-UNDECLARED-REF`), relation name
  (`E-RELATION-UNKNOWN`), reserved path (`E-CHOICELOG-READ`), or component name
  (`E-COMPONENT-UNDECLARED`) ‚Äî MUST collapse to **one primary** diagnostic at the first
  occurrence in document order, carrying every further occurrence as a covered span.
  Site-specific analyses (e.g. `E-MAYBE-UNSET`, whose verdict depends on each site's
  dominators and guards) are exempt ‚Äî each site is independently meaningful.
- **C2 (parse-failure suppression).** A CEL slot that fails to parse produces its
  `E-CEL-PARSE` and NOTHING else: every downstream per-slot analysis (profile, refs, paths,
  temporal, domain) MUST be suppressed for that slot. (Practice today; normative now.)
- **C3 (import-failure suppression).** When a `uses:`/`extends:`/`components:` import fails
  to resolve (`E-USES-NOT-FOUND`/`-PARSE`/`-CYCLE`, `E-COMPONENT-PARSE`), the checker MUST
  NOT emit absence-of-declaration diagnostics whose claim depends on the merge it failed to
  build ‚Äî `E-UNDECLARED`, `E-UNDECLARED-REF`, `E-MAYBE-UNSET`, `E-RELATION-UNKNOWN`,
  `E-COMPONENT-UNDECLARED` for the affected namespaces. ‚ÄúNot declared‚Äù is unproven while a
  declaration source is missing; the import error is the root, and fixing it MUST be the
  only demand the checker makes.
- **C4 (implied-consequence suppression).** A diagnostic strictly implied by an
  already-emitted diagnostic on the same construct MUST NOT be emitted separately: the
  quest-level consequence of `E-OBJECTIVE-UNSATISFIABLE` rides as a note, never as a second
  `E-QUEST-UNREACHABLE` (¬ß5.3); an `E-ARM-DEAD` arm produces no `W-OVERLAP-ARMS` (¬ß5.2).
- **C5 (presentation).** The primary carries its covered occurrences: in human output as a
  trailing `(+N more: 12:3, 47:9, ‚Ä¶)`; in JSON as an additive `covered: [Span]` field.
  Collapse never changes ordering or determinism (primaries sort exactly as today), and
  never crosses files.

### 8.3 Machine surface

The JSON diagnostic object gains one additive field (`covered`, C5) and wider use of the
existing `fixits` (T2). No field is removed or retyped; `ok`/error counting is by primaries
(a collapsed occurrence does not inflate the count ‚Äî five reads of one typo are ONE error).
Tools MUST key success on `ok` (or the exit status), never on parsing a printed count.

## 9. Non-goals

- **`trace` is not a runtime.** It asserts nothing about the engine's behavior ‚Äî no parity
  guarantee, no QA substitute, no source of truth. The quarantine (¬ß4.2) is permanent.
- **No Datalog evaluation in Lute** ‚Äî anywhere, including `trace`: derived relations are
  mocked, never computed. The engine owns the fixpoint (`0.3 ¬ß10`).
- **No SAT / model checking.** The ¬ß5.1 fragment is the ceiling for error-grade reachability
  claims; path-sensitive, cross-shot, or cross-document reachability is out (per-document
  analysis only; the project-wide surface remains quest-id uniqueness, `0.2 ¬ß6.3`).
- **No `<branch>`/`<hub>` ‚Äî and no new local-choice form ‚Äî in component bodies.** Choice
  presentation *records* (`scene.choices.*`/`scene.visited.*`, `0.1 ¬ß9.6`): it is a state
  write and can never satisfy component purity (¬ß6.1). A future parameterized-menu design
  would need its own recording story first; 0.4.0 deliberately does not open it.
- **No interactive trace / REPL / watch mode** ‚Äî `trace` is batch and deterministic; an
  interactive playtest UI is downstream tooling over `--json` (¬ß4.5).
- **No multi-line gated block sugar.** `when=` covers the single line; a guarded group
  remains `<match>` (¬ß7.2) ‚Äî one construct per job.
- **No implied `persist=`.** A bare `into=` stays what it is today ‚Äî a no-op, now warned
  (`W-CHOICE-INTO-NO-PERSIST`, ¬ß7.3); reinterpreting it as an implicit `persist="run"` is
  left to a future major that can afford the meaning change.
- **No block-scoped sticky defaults (`::defaults`) ‚Äî deferred.** A mid-body sticky
  directive is action-at-a-distance: a reader of `@bianca: hi` must scroll up to learn its
  emotion ‚Äî hidden positional state, against the clarity this release serves. If the
  repeated-attr cost (¬ß7.1, beat 3) earns a sugar, the better direction is **declarative
  per-speaker defaults in frontmatter** ‚Äî visible and non-positional ‚Äî not a mid-body
  directive.
- **No grammar / parser productions** ‚Äî 0.4.0 adds no syntax tree shapes; editors and
  tree-sitter grammars need no structural update (highlighting the new attr keys/directive
  is cosmetic vocabulary).
- **Still out, unchanged:** ICU MessageFormat (`0.1 ¬ß14`), backward diverts (`0.1 ¬ß14`),
  Lute-CEL profile expansion (`0.2 ¬ß9`), open vocabularies (`0.3 ¬ß13`).
