# Lute Scenario DSL — 0.5.2: Unset-Sentinel Spelling Diagnostic

- **Version:** 0.5.2
- **Status:** Draft (**compatible refinement** of 0.5.1)
- **Date:** 2026-07-13
- **Scope:** a single new authoring-feedback diagnostic (`E-UNSET-LITERAL`) that
  catches the most common way authors misspell the *unset* sentinel in a CEL
  guard. No language grammar productions, no IR shape change, no §5.1 decide
  semantics change. The existing golden snapshots and version-envelope tests are
  re-accepted for the version-line change only, in the same commit as the
  constant bump.

## 1. Motivation

A round-3 hands-on review surfaced a real friction (not a soundness bug): an
author who wants "when this quest has not started yet" writes

```
<choice when="quest.riverDebt.state == 'unset'"> … </choice>
```

The DSL's *unset* sentinel is **not** the string `'unset'` — it is the CEL
`null` literal (`0.1 §11.2`), matched in a `<match>` by the literal arm
`<when is="unset">` or tested with `!isSet(path)`. So
`quest.riverDebt.state == 'unset'` compares the path to a foreign **string**
that it can never equal, and §5.1 R2 correctly **decides it `false`**
(`0.4.0 §5.1`: a finite-domain `S == lit` with `lit` outside the domain decides
false; the domain of a maybe-unset `S` is `{enum members} ∪ {Unset}`, and the
foreign string `'unset'` is in neither). Reachability then reports `E-ARM-DEAD`
("guard provably false"), which — piled on the independent, also-correct
`E-MAYBE-UNSET` for the raw read of a no-default path — reads as two
contradictory errors to an author who simply spelled the sentinel wrong.

This revision keeps every decision the checker already makes (R2 still decides,
`E-MAYBE-UNSET` still fires; see §4) and adds ONE targeted, actionable
diagnostic that names the mistake and the supported forms.

## 2. `E-UNSET-LITERAL` (new diagnostic)

### 2.1 Trigger

`E-UNSET-LITERAL` fires for a CEL guard slot (a `<when test>`, `<choice when>`,
`<match on>` subject, `<objective when/done>`, or `<quest start/fail>` — any
profile CEL slot) that contains a comparison

```
S == 'unset'      'unset' == S      S != 'unset'      'unset' != S
```

where **all** of:

1. `S` resolves to a **finite-domain** subject (a dotted state path via
   `infer_domain`, the match `$`, or a bound component param) — the SAME
   `resolve_domain` used by §5.1 R2;
2. that domain is **maybe-unset** (`DomainInfo.maybe_unset == true` — an
   undefaulted enum path, or the reserved `quest.<id>.state`, `0.2 §5.2`);
3. the string `'unset'` is **foreign** to the domain (NOT a declared enum
   member literally named `unset`).

The comparison may be **nested** inside a larger boolean expression
(`a && quest.q.state == 'unset'`); the lint scans every comparison sub-expression,
not only a top-level guard. **Both operand orders** and **both `==` and `!=`**
trigger it — critically, `!= 'unset'` decides `true` (always) and therefore
never reaches the dead-arm path, yet it is the identical sentinel mistake, so
`E-UNSET-LITERAL` is an **independent AST lint**, not a swap on a decided-false
guard.

### 2.2 Message

> `E-UNSET-LITERAL` — comparing `quest.riverDebt.state` to the string
> `'unset'`, which is never equal to the *unset* sentinel. Test for unset with
> `!isSet(quest.riverDebt.state)`, or in a `<match on="quest.riverDebt.state">`
> use `<when is="unset">` (dsl 0.2 §5.2).

Severity **error** (the guard is still a real authoring defect — the arm never
fires, or always fires, contrary to intent). `Layer::Logic`.

### 2.3 Ownership — suppress the derivative reachability diagnostic

When `E-UNSET-LITERAL` fires for a `S == 'unset'` comparison, that comparison is
its **root**: reachability MUST NOT ALSO emit `E-ARM-DEAD` for a guard whose
decided-`false` is caused by that same comparison (mirrors the existing D4 rule
that a foreign `is=` literal is owned by `E-WHEN-LITERAL-DOMAIN` and excluded
from the dead-arm pass, `reachability.rs`). The `S != 'unset'` form decides
`true` and never produced `E-ARM-DEAD`, so nothing is suppressed there; if it
makes an `<otherwise>` provably dead, that `W-OTHERWISE-DEAD` is likewise the
sentinel comparison's derivative and is suppressed. `E-MAYBE-UNSET` (§4) is NOT
a derivative of the comparison — it is about the raw read of a no-default path —
and continues to fire.

## 3. Non-goals (unchanged from 0.5.1)

- **No §5.1 change.** `decide()` still decides `S == 'unset'` `false` and
  `S != 'unset'` `true`; R2's domain `{members} ∪ {Unset}` and the rule that a
  foreign literal is outside it (`0.4.0 §5.1`) are untouched. `E-UNSET-LITERAL`
  is a reporting-layer lint that reads the same domain, never a decision.
- **Sentinel stays distinct.** `unset` is NOT added to any enum, and
  `quest.<id>.state` is NOT given a `default` — the lifecycle enum
  `active|complete|failed` and the maybe-unset sentinel remain separate
  (`0.2 §5.2`).
- **Other foreign literals unchanged.** A foreign typo that is NOT the string
  `'unset'` (e.g. `run.rank == 'legendary'`) still decides `false` and still
  reports `E-ARM-DEAD` (in a guard) or `E-OBJECTIVE-UNSATISFIABLE` (in an
  objective `done`), on both defaulted and undefaulted enums. A legitimate enum
  member literally named `unset` (in-domain) is a normal comparison — no lint,
  no dead-arm.

## 4. `E-MAYBE-UNSET` is retained and correct

A bare CEL read of `quest.<id>.state` (a no-default path, `0.2 §5.2`) with no
dominating `::set` and no `has`/`isSet` guard is `E-MAYBE-UNSET` (`§9.4`),
independent of this revision. Following `E-UNSET-LITERAL`'s advice to use
`!isSet(quest.<id>.state)` clears BOTH diagnostics at once (the `isSet` guard
proves the path for its arm, `defassign.rs`), which is why the message names
that form first.

## Appendix A — Diagnostic codes

One new **error** code: `E-UNSET-LITERAL` (§2). No codes removed; no decided
`check`/`compile` verdict of a well-formed document changes except that a
`S == 'unset'` guard now reports `E-UNSET-LITERAL` in place of `E-ARM-DEAD`
(same error count contribution, clearer message), and a `S != 'unset'` guard
that previously reported nothing now reports `E-UNSET-LITERAL`.

## Appendix B — Version stamp

`LUTE_LANG_VERSION` and `LUTE_IR_VERSION` advance to `"0.5.2"`
(`lute-compile/src/lib.rs`); the ONLY permitted golden diff is the
`lute`/`irVersion` envelope line, in the same commit as the constant bump.
