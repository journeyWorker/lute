# Lute 0.3.0 — Relational Facts & Datalog Derivation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the Lute 0.3.0 relational state kernel — closed entity/relation/enum schema blocks, seed facts, Datalog rules, `::assert`/`::retract` body directives, the `holds`/`count`/`validAt`/`now()` CEL query surface, and the opaque narrative-time type — as a **static layer only**: parse → statically check → emit JSON for a downstream engine.

**Tech Stack:** Rust workspace crates (`lute-syntax`, `lute-manifest`, `lute-check`, `lute-cel`, `lute-compile`, `lute-cli`, `lute-lsp`), tree-sitter (`grammar.js` + corpus), cel-parser 0.10.1, serde_yaml 0.9, insta 1 (yaml), TextMate JSON, Neovim `.scm` queries.

## Decisions & Spec Deviations (READ FIRST — every task honors these)

- **D1 — No evaluator.** See Global Constraints. If any step below seems to require evaluating a rule, you are misreading it: every check is a syntactic/graph property of the DECLARED rule set.
- **D2 — Diagnostic-code reassignment for project schema vocabulary collisions (BREAKING, no codemod).** 0.2.2 landed `E-DOMAIN-DUP` for ANY cross-file project `enums:`/`entities:` name collision (`schema_import.rs:296-305`), with an explicit doc-comment deferral ("0.3.0 leaves additive extends growth for entity/relation composition"). 0.3.0 resolves that deferral per spec §4.1: a `uses`-PEER collision on an `enums:` name → **`E-USES-DUP-RELATION`**; a peer collision on an entity-KIND name → **`E-KIND-NAME-CLASH`**; an `extends` child may ADDITIVELY grow an inherited kind/enum (superset re-declaration, see D5). `E-DOMAIN-DUP` REMAINS for plugin/core-vs-project collisions (`merge_domains`, unchanged) and cross-plugin collisions (`assemble.rs`, unchanged). Existing tests in `crates/lute-check/tests/domains.rs` asserting `E-DOMAIN-DUP` for project-project collisions are UPDATED in Task 6.
- **D3 — One new diagnostic outside Appendix A: `E-DATALOG-PARSE`** (check stage; "a `facts:`/`rules:` entry or `::assert`/`::retract` payload that is not a string / does not parse per the Appendix C grammar"). Appendix A has no code for a structurally malformed fact/rule string (e.g. the unquoted-YAML misparse §4 warns about, or `"canReach(C, L) : atLocation(C, L)"`). Diagnostics are a change-controlled set (§12): this plan ADDS the code and flags it for a spec Appendix A amendment; it ships with adversarial fixtures like every other code. Compound/function terms keep their own spec code `E-DATALOG-FUNCTION`.
- **D4 — `E-RELATION-DOMAIN` is the residual bucket for malformed relation-decl FIELDS** (non-bool `derive:`/`reserved:`, non-string/unknown `tier:` value, unknown decl keys, non-string `args` entries), matching its Appendix A scope ("relation arg names no declared entity kind/enum/bool, or `key:` names an out-of-range/duplicate arg index") in spirit. A decl that is not a mapping at all, or has no/empty `args`, is `E-RELATION-EMPTY`.
- **D5 — `extends` growth semantics.** A child re-declaring an inherited entity kind or `enums:` entry MUST re-declare a SUPERSET of the base's members (add-by-re-listing, mirroring how an `extends` child re-declares a state path); a missing base member (or a `members:`↔`open:` shape flip) is `E-EXTENDS-RELATION-SIG`; the merged entry is the child's list. A child re-declaring an inherited RELATION must match the FULL decl — `args` + `tier` + `derive` + `reserved` + `key` (spec §4.1 names the first four; `key:` is included because a differing functional key silently changes engine auto-invalidation semantics — flagged as a deliberate strictness extension). Rules and facts always UNION across the DAG.
- **D6 — Rule constants are domain-checked under `E-FACT-DOMAIN`.** Appendix A scopes `E-FACT-DOMAIN` to asserted/seeded/retract-pattern args; this plan also applies it to ground `Const` terms in rule heads/bodies (e.g. `alerted(absolute, F)` — `absolute` must be a `faction`). Same closure principle, deliberate extension.
- **D7 — `now()` inside a rule-body CEL guard is `E-DATALOG-GUARD-FACT`** alongside `holds`/`count`/`validAt`: §9.3 is explicit that "rules have no access to time", and a time read through a guard is exactly the hidden non-monotonic dependency the firewall exists to stop.
- **D8 — `E-TEMPORAL-ARG` admits five comparison operators** (`<`, `<=`, `==`, `>`, `>=`) between two narrative-time values; **`!=` is REJECTED** (→ `E-TEMPORAL-ARG`). §6 lists `<`/`<=`/`==` (ordering + identity); `>`/`>=` are the trivially-normalizable ordering mirrors (`a > b` ≡ `b < a`) and are admitted for author convenience with zero soundness cost. `!=` is identity-NEGATION — a broader non-order predicate NOT in §6's surface; it stays rejected unless the spec is amended (an author needing it writes `!(a == b)`, which uses the admitted `==`). Controller-tightened from an earlier "all six" reading. Flagged as an interpretation for spec confirmation.
- **D9 — `::assert`/`::retract` are NOT admitted inside `<track>` clips.** Spec §5 admits them "wherever ::set is (scene content, `<on>`/quest/objective arms)" — the enumeration excludes timeline tracks, and `ClipNode` stays `Directive | Set`. An `::assert` inside a `<track>` parses as a generic directive and is flagged by the existing unknown-directive path.
- **D10 — `open: engine` kinds are NOT membership-checkable at compile time** (spec §3.1: an id "becomes a valid K(X) member once the engine mints it"). For an arg whose domain is an open kind, `E-FACT-DOMAIN` checks ONLY: (a) the literal is a `CelIdent`-shaped ident (not a bool where an id is required), and (b) the id is NOT a declared member of a DIFFERENT closed kind (one-id-one-kind, §3.1). Actual membership is engine-deferred. Implementers MUST NOT try to enforce open-kind membership statically.
- **D11 — `narrativeTime` is not author-declarable state.** `Type::NarrativeTime` exists so plugin capability exports (`state_shapes`, the engine→state contract) can surface anchor paths; an author `state:` decl (inline or schema doc) with `type: narrativeTime` is `E-TEMPORAL-ARG` at the decl. - **D12 — Seed `facts:` may not contain `_`.** Seeds are ground (§4, "checked as for ::assert"), so a wildcard in a seed fact reuses `E-RETRACT-WILDCARD-ASSERT` with a seed-specific message.
- **D13 — Malformed-parse sentinel.** When an `::assert`/`::retract` payload fails to parse, the parser emits `E-DATALOG-PARSE`/`E-DATALOG-FUNCTION` and still produces the node with `pattern.relation == ""` (the sentinel).
```rust
/// A fact pattern: `rel(a, b)` / `rel(a, _)` (spec §5 GroundFact/RetractPattern).
#[derive(Clone, Debug, PartialEq)]
pub struct FactPattern {
    pub relation: String,
    /// Byte range of the relation ident, relative to the parsed input.
    pub relation_span: (usize, usize),
    pub args: Vec<FactArg>,
    /// Byte range of the whole pattern, relative to the parsed input.
    pub span: (usize, usize),
}
#[derive(Clone, Debug, PartialEq)]
pub struct FactArg { pub term: FactTerm, pub span: (usize, usize) }
#[derive(Clone, Debug, PartialEq)]
pub enum FactTerm { Ident(String), Bool(bool), Wildcard }

/// One Horn clause: `Head :- Body` (spec §7.1).
#[derive(Clone, Debug, PartialEq)]
pub struct Rule { pub head: RuleAtom, pub body: Vec<BodyLiteral> }
#[derive(Clone, Debug, PartialEq)]
pub struct RuleAtom {
    pub relation: String,
    pub relation_span: (usize, usize),
    pub terms: Vec<RuleTerm>,
    pub span: (usize, usize),
}
/// Var = leading ASCII uppercase ident; `true`/`false` = Bool; other idents = Const (§7.1).
#[derive(Clone, Debug, PartialEq)]
pub enum RuleTerm { Var(String), Const(String), Bool(bool) }
#[derive(Clone, Debug, PartialEq)]
pub enum BodyLiteral {
    Pos(RuleAtom),
    Neg(RuleAtom),
    /// `cel("…")` — the raw CEL string, unescaped (§7.3). Span covers the whole literal.
    Guard { cel: String, span: (usize, usize) },
    /// `Term = Term` (negated: false) / `Term != Term` (negated: true) (§7.1).
    Cmp { lhs: RuleTerm, rhs: RuleTerm, negated: bool, span: (usize, usize) },
}

#[derive(Clone, Debug, PartialEq)]
pub enum DatalogError {
    /// Anything that violates the Appendix C grammar other than a function term.
    Malformed { at: usize, msg: String },
    /// A compound/function term `f(g(x))`, or arithmetic between terms (§7.1, E-DATALOG-FUNCTION).
    FunctionTerm { at: usize, name: String },
}

pub fn parse_fact(input: &str) -> Result<FactPattern, DatalogError>;
pub fn parse_rule(input: &str) -> Result<Rule, DatalogError>;
```

Grammar implemented (Appendix C, byte-level recursive descent over `input.as_bytes()`, same idiom as `parser.rs`):

```text
FactPattern ::= Ident "(" FactArg ("," FactArg)* ")"
FactArg     ::= Ident | "true" | "false" | "_"
Rule        ::= Atom ":-" Literal ("," Literal)*
Literal     ::= "not" WS Atom | "cel(" CelString ")" | Term ("="|"!=") Term | Atom
Atom        ::= Ident "(" Term ("," Term)* ")"
Term        ::= Ident | "true" | "false"           (* "_" is FACT-pattern-only; in a rule it is Malformed *)
Ident       ::= [A-Za-z][A-Za-z0-9_]*               (* CelIdent — no "-" *)
CelString   ::= "\"" ([^"\\] | \\.)* "\""
```

Disambiguation rules (implement exactly): after an ident inside an atom's parens, a `(` means a nested call → `FunctionTerm` (the `E-DATALOG-FUNCTION` seed). `+`/`-`/`*`/`/` adjacent to a term inside an atom or comparison → `FunctionTerm` with `name` = the operator char. A body literal starting with ident then `(` is an Atom; ident/bool then `=`/`!=` is a `Cmp`; `not` must be followed by an Atom (else `Malformed`). `=` inside `!=` handled by trying `!=` first. Zero args (`rel()`) is `Malformed` (arity-0 relations cannot be declared — `E-RELATION-EMPTY` — so a 0-ary use is a grammar error here). Trailing junk after the closing shape is `Malformed`. Whitespace (spaces/tabs) is free between every token.
