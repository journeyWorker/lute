# Lute DSL 0.2.0 — Document Kinds & the Quest Kind (approved design)

- **Date:** 2026-07-07
- **Status:** approved design; spec-first (documents revised before any implementation)
- **Drives:** `docs/proposals/scenario-dsl/0.2.0.md` (new, **additive** extension of 0.1.0),
  errata to `docs/proposals/plugin-system/0.0.1.md` (an `events` capability export),
  addendum to `docs/superpowers/specs/2026-07-04-lute-compile-json-ir-design.md` (kind
  envelope + `on`/`quest`/`objective` records).
- **Provenance:** a domain-generalization evaluation — can the 0.1.0 declarative kernel
  (tiered state + Lute-CEL + reduce-to-records totality + plugin/capability + stable
  `lineId`) carry non-VN authoring domains (quests, rewards, achievements, daily quests)?
  Cross-checked against the most extreme interwoven case, **Baldur's Gate 3 / Larian's
  Osiris** — a *declarative, event-driven, Event-Condition-Action* story-scripting layer
  sitting over a separate simulation engine. Lute occupies Osiris's role, deliberately
  constrained to be **total** (statically verifiable, reduces to JSON) where Osiris is
  Prolog-like. Conversation-approved decisions D1–D10 below.

## 0. Versioning & sequencing

- 0.2.0 is an **additive minor** over 0.1.0 — **not breaking**. Every existing 0.1.0
  `.lute` scene remains valid unchanged (it acquires an implicit `kind: scene`, D8).
- The 0.1.0 spec is **NOT frozen**: `scenario-dsl/0.1.0.md` remains the normative source
  for the **scene (VN) kind** and for the **shared kernel** (logic layer, Lute-CEL, state
  tiers, i18n). `scenario-dsl/0.2.0.md` is a NEW file written as an **extension**: it
  references 0.1.0 for the kernel rather than restating it, and specifies only the kind
  system + the new primitives + the quest kind.
- **Scope this pass:** the kind framework + the two new primitives (`<on>` ECA trigger,
  `quest.*` instance tier) + **the quest kind fully specified as the exemplar**. The
  reward / achievement / daily-quest kinds are deferred to follow-on 0.2.x specs; each is
  a thin vocabulary pack over this framework. The engine/SDK and any BG3-scale capability
  pack are out of scope (downstream, separate efforts).
- No implementation in this pass; the 0.2.0 spec carries its own inline examples.
  `docs/examples/*.lute` are not migrated.

## D1. `.lute` becomes kind-polymorphic — NEW

- A `.lute` document is no longer implicitly a VN episode. A **document kind** selects the
  required frontmatter keys, the admitted grammar subset (D3), the per-kind vocabulary
  (D2), and the identity rule (D7).
- The checker's `MetaKind` (today `Scene | Schema | Component`, `crates/lute-check/
  src/meta.rs`) gains **`Quest`**. `Schema`/`Component` are *import-role* documents
  (reached via `uses:`/`components:`) and are orthogonal to the **domain kind** of a
  *root/authored* document; only root documents carry `kind:` (D8).
- Rationale (BG3/Osiris): the interweaving of quests↔story↔rewards↔achievements is
  **data** — shared tiered state + CEL gates + ECA triggers — which is exactly the 0.1.0
  kernel. The simulation (combat, dice, physics) is **code**, and stays in the engine
  (D10). The language generalizes; the engine does not move into the language.

## D2. Shared logic kernel / per-kind vocabulary

- The 0.1.0 three-layer model (§5) is the seam. The **logic layer** — `<branch>`,
  `<match>`/`<when>`/`<otherwise>`, `<hub>`, `::set`, CEL, `defs`, the state schema,
  totality, and the `lineId`/i18n machinery — is the **kind-neutral kernel**.
- The **content layer** (`:speaker: text` grammar shape) and the **staging layer**
  (`::`-directives) are **per-kind vocabulary**. The i18n/`lineId`/`{{…}}` *plumbing*
  under content is kernel (re-skinned per kind, D7); the content-line *grammar shape*
  (speaker/`delivery`/`voiceKey`) is scene-kind vocabulary.
- Payoff (normative motivation): the static guarantees (exhaustiveness §11.2,
  definite-assignment §9.4, totality §3) live entirely in the kernel and are
  **kind-independent**. Quest/reward/achievement logic inherits them for free; swapping
  vocabulary per kind never touches them.

## D3. Closed grammar, open vocabulary, per-kind admission

- The grammar is a **fixed, closed set** of constructs. Plugins extend **vocabulary**
  (directive names, attrs, assetkinds, state shapes, and now `events`, D5/D10) — **never
  grammar**. Plugin-defined block grammar would demote the static analyses to fixpoint
  computations (same reason backward diverts are a non-goal, §14).
- Each **kind admits a subset** of the closed grammar. This is the existing per-context
  restriction (0.1.0 §7.4 already forbids content/logic inside `<timeline>`) lifted one
  level, to the document kind. A construct that appears but is not admitted by the
  document's kind is **`E-GRAMMAR-NOT-ADMITTED`** (not a generic parse fallthrough).
- Closed grammar set (0.2.0): `content-line`, `::directive`, `::set`, `<branch>`,
  `<match>`, `<hub>`, `<timeline>`, and NEW `<on>`, `<quest>`, `<objective>`.
  - **scene** admits: content-line, `::directive`, `::set`, `<branch>`, `<match>`,
    `<hub>`, `<timeline>`, `# `/`## ` headings. (= 0.1.0 unchanged.)
  - **quest** admits: `<quest>`, `<objective>`, `<on>`, `<match>`, `<branch>`, `::set`,
    content-line, `::directive`. **Forbids** `<hub>`, `<timeline>`, `# `/`## ` headings.
- `<timeline>` remains an **opt-in kernel module** — a `<…>` kernel structure whose
  clips are per-kind vocabulary (the same container/payload shape as `<branch>` arms), not
  universal: the quest kind does not admit it.

## D4. Syntax stays on the nesting axis (`::set` unchanged)

- The 0.1.0 bracket rule (§5) is retained on its axis: `<…>` = **has children
  (nesting)**; `::`/`:` = **single-line leaf**. `<on>`/`<quest>`/`<objective>` have
  children ⇒ `<…>`; `::set` remains a leaf `::`-form.
- The **kernel-vs-vocabulary** distinction is carried **semantically** — by the reserved
  built-in ident list (§10: `set`, `use`; NEW reserved tags `on`, `quest`, `objective`)
  plus the per-kind admission (D3) — **not** by lexical form. `::set` is not promoted to a
  `<set/>`: it is a genuine leaf, a self-closing tag would be the language's first and
  would erode "`<` ⇒ nesting follows," and uniformity (`::set` reads identically in every
  kind) beats local tidiness.

## D5. `<on>` — the ECA trigger primitive — NEW

```
On ::= "<on" Attrs ">" Node* "</on>"     (* requires event and/or when *)
```

- The one genuinely new execution primitive. `event` = a capability-declared event-kind
  id (engine-fired); `when` = a CEL guard over declared state (the same closed Lute-CEL,
  §8.4). **Neither present is `E-ON-EMPTY`** (mirrors `E-WHEN-PATTERN`, D-D of 0.1.0). An
  undeclared `event` is **`E-UNKNOWN-EVENT`**.
- Semantics (Event-Condition-Action): when its `event` fires (for a `when`-only rule, at
  engine-defined state-change re-evaluation points), if `when` holds, the arm's records
  execute. Canonical example (author-supplied): *"combat's activation condition is this;
  if the NPC Halsin dies, the scenario played after combat is this."*

  ```lute
  <on event="combatEnd" when="run.npc.halsin.dead">
    ::bg{location="grove" time="aftermath"}
    :narrator: Halsin had fallen. The grove went silent.
    ::set{run.groveOutcome = "halsinDead"}
  </on>
  ```
  Engine: runs combat, detects the death, writes `run.npc.halsin.dead`, fires `combatEnd`.
  Lute: matches, emits the authored records. The engine is an **opaque event source** —
  Lute never expresses *how* combat runs.
- **Firing & determinism:** `<on>` rules are **all-match** (every rule whose event+guard
  holds fires — unlike `<match>`'s first-wins), and fire in **document order** for a
  deterministic reduction. Cross-document priority is engine policy but MUST be
  deterministic.
- **Totality:** identical argument to `<hub>` (§3, hub totality) — each `<on>` reduces at
  build time to finitely many records; the engine's event loop *consumes events* each
  cycle, introducing no unbounded author computation and no backward divert. The "reduces
  to data" invariant (§3.2) is preserved; `<on>` is **forward-only**.
- **Definite assignment:** `<on>` arms have no dominance relation among one another (same
  join rule as `<match>`/`<hub>` arms); an arm write is a may-write.

## D6. `quest.*` per-instance state tier — NEW (scheduled tiers reserved)

- The 0.1.0 tiers (`scene`/`run`/`user`/`app`, §9.1) are named by their reset boundary.
  A quest is a **run-to-completion instance** with a lifecycle that the 4 tiers cannot
  scope. 0.2.0 adds an **instance-scoped tier `quest.*`**.
- **Reset boundary:** the engine clears `quest.<id>.*` on that quest instance's terminal
  transition (complete / failed / abandoned), per engine policy. Instance-scoped and
  **bounded above by the run** (a new run clears all quest instances); it is a
  run-subordinate, instance-keyed tier, not a new global point on the linear ladder.
  As with every tier, **the engine owns and fires the reset** (§9.1) — the language never
  triggers it.
- **Path shape:** `quest.<questId>.<field>`. Reserved, engine-populated, implicitly
  declared sub-namespaces (like `scene.choices.*`, §9.6):
  - `quest.<id>.state` — lifecycle enum `active | complete | failed`
    (∪ `unset`); content MAY read it in `<match>`/`<when>`; content MUST NOT write it.
  - `quest.<id>.objectives.<oid>.done: bool` (default `false`) — per-objective completion,
    populated by the objective's `done` predicate (D9). Content MAY read it.
  Authors MAY additionally declare ordinary `quest.<id>.*` scratch fields (schema §9.3),
  writable via `::set` inside that quest's arms.
- **Scheduled tiers `daily.*` / `weekly.*` — reserved, not implemented in 0.2.0.** The
  tier system is designed to admit a **time-scheduled reset boundary** (a cron edge,
  engine-owned) so the daily-quest kind lands as a pure state-model extension with no
  language change. 0.2.0 pins the extension point only.

## D7. Per-declaration identity (lineId generalization)

- 0.1.0 keys `lineId` on `meta.episodeId` (per **file**, §12). 0.2.0 generalizes: a
  `lineId`'s prefix is the **enclosing stable declaration id**, not the file.
  - **scene** kind: unchanged — `{character}.{episodeId}.{speaker}_{code}`; choice/hub
    label `{character}.{episodeId}.{branchOrHubId}.{choiceId}`.
  - **quest** kind: quest title `{questId}.title`; objective description
    `{questId}.{objectiveId}`; an `<on>`-arm content line `{questId}.{speaker}_{code}`.
- The rule is a **generalization, not a change** to scene: identity prefix = the enclosing
  declaration's stable id (the episode is that declaration for a scene; the quest is that
  declaration for a quest). All the 0.1.0 i18n contracts (assign-once, placeholder-set
  equality `E-L10N-PLACEHOLDER`, translation sidecar) apply verbatim.

## D8. `kind:` frontmatter discriminator

- A root document declares its domain kind with a top-level **`kind: <ident>`** frontmatter
  key. **Omitted ⇒ `scene`** (existing 0.1.0 scenes are valid with zero edits — the source
  of the additive/non-breaking guarantee, §0).
- 0.2.0 defines two core kinds: **`scene`** (the 0.1.0 VN document) and **`quest`**. An
  unknown value is **`E-UNKNOWN-KIND`**.
- Per-kind **required keys** (a missing one reuses `E-META-MISSING`):
  - `scene`: `character`, `season`, `episode` (= 0.1.0).
  - `quest`: none of the VN triad; a quest **pack** carries `uses:` (the schema it gates
    on), and MAY carry `profile`/`plugins` (the event + state-shape capability), `title`,
    `contentLang`. Quest **identity** is per-`<quest id>` (D7), so no file-level id key is
    required.
- An explicit `kind:` enum scales better than N presence-keys (`component:`-style) as
  kinds accrue (quest/reward/achievement/daily); it also keeps `MetaKind` selection a
  single lookup.

## D9. Quest kind — body & vocabulary

```
QuestDoc  ::= Meta QuestDecl+
QuestDecl ::= "<quest" Attrs ">" QuestBody "</quest>"      (* requires id; optional title, start, fail *)
QuestBody ::= ( Objective | On | Match | Branch | Set | Directive | Line )*
Objective ::= "<objective" Attrs ">" Node* "</objective>"  (* requires id, done (CEL); optional when, title, optional *)
```

- A quest-kind document is a **pack** of one or more `<quest>` declarations (realistic:
  quest files hold many quests). Each `<quest id>` is independently identified (D7); a
  duplicate id in a pack is **`E-QUEST-ID-DUP`**.
- **Lifecycle** lives in `quest.<id>.state` (D6), engine-derived from pure predicates:
  `unset → active` on the quest's `start` CEL; `active → failed` on `fail`; `active →
  complete` when all non-`optional` objectives are `done`. Content MUST NOT write the
  state; it reacts via `<match on="quest.<id>.state">` and via `<on>` lifecycle events.
- **`<objective id done="CEL">`**: an objective whose `done` CEL predicate (over declared
  state) completes it; completion is recorded in `quest.<id>.objectives.<id>.done` (D6).
  Optional `when` gates availability; optional `title` (localizable, D7).
- **`<on>`** (D5): quest reactions to built-in lifecycle events (`questActive`/
  `questComplete`/`questFailed`) and world events — grant a reward on completion, branch on
  an NPC death. This is where the Halsin pattern lives.
- **Rewards** reuse the existing mechanism — `::set{run.* / user.* = …}` or the 0.1.0
  `persist` sugar — appended to an `<on>`/objective arm. A richer `::grant` is capability
  **vocabulary** (engine-provided), not new core grammar.
- **Admitted grammar:** `<quest>`, `<objective>`, `<on>`, `<match>`, `<branch>`, `::set`,
  content-line, `::directive`. **Forbidden:** `<hub>`, `<timeline>`, `# `/`## ` headings
  (`E-GRAMMAR-NOT-ADMITTED`, D3).

## D10. State-contract & engine-computed facts

- **The boundary is normative:** anything the **simulation** produces — dice rolls,
  combat outcomes, positions, and any **aggregate over a collection** (CEL forbids
  `map`/`filter`/`exists`/`all`/`size`, §8.4 `E-CEL-PROFILE`) — is **computed by the
  engine and surfaced as declared state**. Lute *reads* it via CEL; Lute never computes it.
  This is the same split BG3 draws between Osiris and the Divinity engine.
- **`::check`-style capability directives** illustrate the contract: a directive whose
  bridge performs an engine computation and writes a declared `into` path, which content
  then branches on — e.g. `::check{skill="persuasion" dc="15" into="scene.shCheck"}`
  followed by `<match on="scene.shCheck">`. The directive is capability **vocabulary**,
  not core grammar.
- **Events & state shapes are capability-provided** (plugin exports; the `events` export
  is the 0.2.0 plugin-system erratum). This is the integration seam with the engine; a
  BG3-scale game supplies a large capability pack, while the **language stays small**.
- The closed Lute-CEL profile (§8.4) is **unchanged**. Opening CEL (comprehensions) is a
  non-goal (§NG): it trades away the static guarantees and still cannot produce simulation
  facts. The default lever is the engine→state contract, not CEL expansion.

## NG. Non-goals (0.2.0)

- **Reward / achievement / daily-quest kinds** — full specs deferred to follow-on 0.2.x.
  Each is a thin vocabulary pack over this framework; achievement is nearly a `<quest>`
  with one objective and no instance lifecycle.
- **Scheduled tiers** (`daily.*`/`weekly.*`) — extension point pinned (D6), not implemented.
- **Engine / SDK / BG3-scale capability packs** — downstream, out of the language.
- **CEL profile expansion** — the environment stays closed (§8.4).
- **Backward diverts** — still a permanent non-goal (§14). `<on>` is forward-only; the ECA
  event loop is the total, DAG-preserving form of reactivity.

## New/changed diagnostics (registry delta)

NEW: `E-UNKNOWN-KIND` (frontmatter `kind:` unknown), `E-GRAMMAR-NOT-ADMITTED` (construct
not admitted by the document's kind), `E-ON-EMPTY` (`<on>` with neither `event` nor
`when`), `E-UNKNOWN-EVENT` (undeclared event kind), `E-QUEST-ID-DUP` (duplicate `<quest
id>` in a pack), `E-OBJECTIVE-MISSING-DONE` (`<objective>` without `done`).
Reused (unchanged): `E-META-MISSING` (per-kind required key), `E-UNDECLARED`,
`E-MAYBE-UNSET`, `E-CEL-PROFILE`, `E-PATH-IDENT`, `E-CHOICE-DUP`, `E-L10N-PLACEHOLDER`,
and the whole 0.1.0 kernel registry (Appendix E), which the quest kind inherits verbatim.
