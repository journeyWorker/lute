## Important — spec-internal contradictions / real gaps

### I1 · §6 grammar contradicts §6.2 (Title placement + cardinality)

`§6` gives `Document ::= Meta DocItem*`, `DocItem ::= Title | ShotBlock`. That permits **multiple**
`Title`s and a `Title` **after a shot**. But `§6.2` says "An optional document-level H1 title MAY
appear after `Meta` and **before the first shot**" (singular, positioned). The EBNF does not encode
the prose. Additionally, a `# ` line appearing after the first shot is accepted as a title node by
the `§4.3` classifier (rule 2), and **no diagnostic** is defined to reject it.

**Fix:** `Document ::= Meta Title? ShotBlock*` (encodes "at most one title, only before the first
shot"), and define a diagnostic (e.g. `E-TITLE-PLACEMENT`) for a late/duplicate `# ` title.

### I2 · §11.2 — "guard that excludes `unset`" contradicts the `test`-coverage MAY-downgrade

`§11.2` says a conforming checker is **not required to reason about `test`-only coverage**
(recognizing CEL-guard narrowing is downgraded to MAY). The **same** paragraph then lists, as a valid
way to cover a nullable subject's `unset` member: "an `is="unset"` arm, **a guard that excludes it**,
or `<otherwise>`." Recognizing that a `test` guard excludes `unset` **is** exactly the `test`
reasoning the paragraph just excused checkers from. The two statements are mutually exclusive.

**Fix (pick one):**
- (a) `unset` must be covered by `is="unset"` or `<otherwise>` **only** — drop "a guard that excludes
  it"; or
- (b) promote "a guard that provably excludes `unset`" to a **narrow MUST-recognized** case, stated
  separately from the general `test`-coverage MAY.

### I3 · IR addendum: `<when is>` → arm `expr` lowering is unspecified (and `match.subject` has no `expr`)

The 0.1.0 headline construct `<when is>` (literal patterns, `|`-alternation, `unset` matching) has
**no lowering rule** in the compile-IR addendum. Addendum sections A1–A12 add `hub`, `costume`,
`placeholders`, `expr`, `effects` — but the `match` record's arm schema is still
`arms:[{ test:<cel>, target }]` (§4.4, 0.0.1-era). Meanwhile **A7** ("no runtime CEL parser") gives
`expr` to arm `test`, `choice`/`hub` `when`, and `set.value` — but **not to the `match` subject**
(§4.4 `subject` is a bare CEL string). So either the runtime must evaluate `subject` (needs a CEL
parser → violates A7), or every arm inlines the subject (as the §4.5 worked example shows) and
`subject` is debug-only — but that path, and the `is`→`expr` desugaring, are never stated.

**Fix:** add an addendum section that specifies the `is`-pattern lowering into the executable arm
`expr`:
- subject inlined into each arm (matching the worked example), and `subject` **declared
  debug-only** (string, non-authoritative) — OR give `subject` its own `expr`;
- `is="a | b"` → `(subj)=='a' || (subj)=='b'`;
- `is="unset"` → `!isSet(<path>)` / null-check, **including the compound-subject case** (a
  maybe-unset compound subject where `isSet` on a path does not apply);
- confirm `arms[]` carries `expr` for the synthesized test just like authored `test`.

This is the one gap that most directly blocks this worktree — the syntax→IR path needs it defined.

### I4 · Voiced line + `{{interpolation}}` interaction is undefined

`§7.6` permits `{{…}}` inside content `Text`; `§12` emits a `voiceKey` on **voiced** lines
(dialogue / voiceover). A voiced line carrying a runtime placeholder (`{{run.coins}}`,
`{{userName}}`) cannot match a pre-recorded VO clip. Nothing in `§7.6`, `§12`, IR addendum **A3**, or
**A11** reconciles this — it is a genuine content-authoring hazard, not a wording nit.

**Fix:** state the policy explicitly. Options: reject interpolation in a voiced line
(`E-VOICE-INTERP`); or suppress `voiceKey` for interpolated lines (+ a warning); or define a
partial-VO contract. Pick one and normatively document it in §12 (+ IR addendum note).

## Minor — clarity / consistency

- **M1 · §7.6** `{{@fn(args)}}`: `@fn` arguments are CEL expressions, yet the same section says
  "arbitrary CEL is not [legal inside `{{…}}`]; use a `@def`." fn-args reintroduce arbitrary CEL.
  Either restrict interpolation `Ref` to zero-arg `@def`, or explicitly permit fn-args-as-CEL and
  soften the wording. Also state that an interpolated `@ref` must produce a renderable type
  (number/bool/enum, per the rendering rule).
- **M2 · §9.6 / §11.1.2** `run.choiceLog.*` read is called "a static (**lint**) error" — ambiguous
  severity (build-failing error vs. warning). Pin it.
- **M3 · §7.4 vs §11.4** — §7.4's comment says a track requires "one of `subject|channel|property`",
  but §11.4 requires a `property` track to **also** carry `subject` (`E-TRACK-KEY`). Correct §7.4 to
  "`channel` | `subject` | `subject`+`property`".
- **M4 · §11.2** "an age-gated `<match on="app.rating">` MUST cover `teen` …" — a normative MUST that
  names a single enum member without defining the `rating` domain. Generalize to "MUST explicitly
  handle the safe/minimum rating tier (or carry `<otherwise>`)."
- **M5 · Diagnostic registry incompleteness** — codes used in the spec body but absent from the
  design-doc / Appendix D "registry delta": `E-TRACK-KEY`, `E-REF-ARITY`, `E-REF-ARG-TYPE`,
  `E-REF-TYPE`, `E-COMPONENT-{UNDECLARED,ARG,CYCLE,DUP,PARSE}`, `E-PERSIST-{TARGET,VALUE,CONFLICT}`.
  Add a single consolidated diagnostic-registry appendix (new + reused-from-0.0.1).
- **M6 · §7.3.1** The lexical class of an `is` `EnumMember` literal is unpinned (may it contain `-`?).
  Enum members are not CEL-facing, but they are the `is` equality operands and render verbatim
  (§7.6). State the class explicitly.
