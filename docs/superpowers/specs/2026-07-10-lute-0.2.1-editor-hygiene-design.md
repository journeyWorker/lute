# Lute 0.2.1 — Editor Fidelity, Content-Line Validation & Example Hygiene (approved design)

- **Date:** 2026-07-10
- **Status:** approved design; spec-first (documents/decisions before implementation)
- **Drives:** `editors/vscode/syntaxes/lute.tmLanguage.json`, `editors/vscode/package.json`,
  `editors/nvim/queries/lute/*.scm`, `crates/lute-lsp/src/features/{mod,completion}.rs`,
  `crates/lute-check/src/{check,meta}.rs`, `crates/lute-compile/src/lower.rs`, and the
  `docs/examples/` tree. No change to the normative language grammar.
- **Provenance:** diagnosis of ten reported problems (this session). The problems split into a
  **non-breaking editor/hygiene + enforce-what-the-spec-already-says** half (this doc, 0.2.1) and a
  **breaking data-catalog + `.yaml` format + `@` speaker sigil** half (the follow-on **0.2.2**
  foundation spec, see §0). This doc is the first half only.

## 0. Scope & sequencing (normative)

- **0.2.1 is NON-BREAKING.** No language grammar changes; the content-line speaker syntax stays
  `:speaker:` / `:speaker{attrs}:`. 0.2.1 (a) makes the two editor integrations (VS Code, Neovim)
  faithfully render what the parser/LSP already produce, (b) enforces the content-line attribute
  domains the language spec **already** mandates but no code enforces, and (c) removes the
  false-diagnostic and confusion friction in `docs/examples/`.
- **DEFERRED to the follow-on new-minor spec (breaking):** the closed-vocabulary **data-catalog
  unification** (plugin `enums`/`providers` ∪ project-authored domains, one merged namespace), the
  **`.schema.lute` → `.yaml`** declaration-file format split with a shipped **JSON Schema**
  (structural lint) + **LSP claiming declaration YAML** (semantic lint), per-character/per-game
  **`emotion`/`action` catalogs**, and the **speaker sigil `:` → `@`** swap (with a `lute fix`
  codemod). 0.2.1 must not pre-empt those; where a 0.2.1 fix touches a surface the new minor will
  revisit, it is keyed on stable properties (frontmatter shape, import reachability) so it composes
  rather than conflicts.
- **Decision locked (for the new minor, recorded here):** the speaker sigil will become `@speaker:`
  (not `>` / not sigil-less). 0.2.1 does not implement it.

## Root cause (one line)

Everything works at the *core* — the tree-sitter grammar, canonical `queries/*.scm`, and the LSP's
`semtok.rs`/completion all handle 0.2.0 quest/on/objective and attribute-enum validation correctly.
The failures live in **stale/incomplete downstream consumers** (VS Code TextMate grammar, VS Code
semantic-token contribution, the hand-mirrored Neovim queries), in **two validation gaps** (content-
line attributes are never checked; `delivery` is never enforced), and in **example ergonomics** (no
project wiring, inconsistent naming, false `E-KIND-MISSING` on import-only fragments).

## D1. VS Code TextMate grammar — quest tags + modern content line

`editors/vscode/syntaxes/lute.tmLanguage.json` is a hand-written TextMate grammar (independent of
tree-sitter) and is stale:

- **D1a.** `repository.tag` `begin` enumerates only `timeline|track|branch|choice|match|when|
  otherwise`. Add `quest`, `on`, `objective` so the 0.2.0 tags get scoped. (The canonical
  `tree-sitter-lute/queries/highlights.scm` already scopes them — this only mirrors it into the
  TextMate world VS Code actually uses.)
- **D1b.** `repository.line` matches only the **deprecated** `:line[name]:` bracket form
  (`^[ \t]*(:line)(\[)…`). Replace it with the current content line: `^[ \t]*:` *speaker*
  `({attrs})?` `:` *text*. This (i) colors modern dialogue lines, and (ii) makes the TextMate
  grammar **claim the `:…:` span**, which — together with D4 — stops the host editor's generic
  `:word:` emoji provider from owning that span (partial mitigation of the emoji collision until the
  `@` swap lands in the new minor).

## D2. VS Code semantic-token contribution

`editors/vscode/package.json` `contributes` declares `languages` + `grammars` only. The LSP emits
correct semantic tokens for a custom legend (`content`/`staging`/`logic`/`cel`/`ref`/`statePath`,
`crates/lute-lsp/src/features/semtok.rs`) **including** quest/on/objective, but VS Code cannot theme
non-standard token types without a client-side declaration, so they render invisibly and VS Code
falls back entirely to the (stale) TextMate grammar.

- Add `contributes.semanticTokenTypes` for the legend and `contributes.semanticTokenScopes` mapping
  each to a standard TextMate scope, so the LSP's (correct) semantic tokens become the primary
  coloring source and the TextMate grammar is only the pre-LSP/inert-file fallback.

## D3. Neovim query mirror + drift guard

`editors/nvim/queries/lute/{highlights,folds,tags}.scm` are hand-copied mirrors of the canonical
`tree-sitter-lute/queries/*.scm` and were never updated after the 0.2.0 quest work, so Neovim gets
zero quest/on/objective highlighting, folding, or go-to-definition.

- **D3a.** Regenerate the three nvim query files from the canonical ones (add the quest/on/objective
  sections).
- **D3b.** Add a **drift guard**: a test (or `lute`/CI check) that fails when
  `editors/nvim/queries/lute/*.scm` diverge from `tree-sitter-lute/queries/*.scm` (byte-identical, or
  identical after a documented transform). Convention-only mirrors rot; make the mirror mechanical.

## D4. LSP speaker cursor + speaker completion

`crates/lute-lsp/src/features/mod.rs` `resolve_line` inspects only `attrs` and `{{interp}}` spans —
there is **no `Cursor` variant for the speaker-name position** — so the LSP offers nothing when the
cursor sits on `:narrator`/`:bianca`.

- Add `Cursor::Speaker` and resolve it in `resolve_line`.
- Provide `speaker` completion: character ids from the resolved capability/cast catalog (the
  character-cast provider, when active); absent a cast catalog, the completion is empty but the
  cursor case still **claims** the position so the editor's generic `:word:`/emoji behavior does not.
- Non-goal for 0.2.1: validating the speaker id (that is the cast-catalog work in the new minor);
  0.2.1 only adds the cursor case + completion source.

## D5. Content-line attribute validation + `delivery` enforcement

Today content-line attributes are **entirely unchecked**: `crates/lute-check/src/check.rs`'s
`Node::Line` arm runs only `check_attr_refs`/`check_interps` (reference/interp scanning), never
key/value validation; `crates/lute-lsp/src/features/completion.rs` short-circuits
`Cursor::AttrKey{directive:None,..} | Cursor::AttrValue{directive:None,..} => Vec::new()`. The
language spec (`scenario-dsl/0.1.0.md` §7.1, §12.1) defines the content-line attribute set and
`delivery`'s domain, but none of it is enforced.

- **D5a — wire content-line attrs through the existing attr-schema path.** Give the content line a
  built-in attribute schema and validate/complete it via the same `check_attr_value`/`check_enum_member`
  machinery directives already use (`crates/lute-check/src/directives.rs`), reusing existing
  diagnostics (`E-UNKNOWN-ATTR`, `E-ATTR-TYPE`, `E-BAD-ENUM`). Schema for 0.2.1:
  - `code` — string; `variant` — number; `dialogMotion` — string; `as` — string;
  - `emotion` — enum over the **core** `emotion` domain (`lute.core/enums.yaml`) **for now**; the
    per-character/per-costume emotion catalog is new-minor cast work — 0.2.1 keeps the flat core enum;
  - `action` — string **for now**; the character-action catalog is new-minor work (this doc does NOT
    convert it to an enum/assetKind, to avoid pulling in the catalog design);
  - `delivery` — **fixed core enum** `spoken | thought | voiceover` (D5b).
- **D5b — enforce `delivery` (fixed core enum, §12.1).** Add `E-DELIVERY-VALUE` (a `delivery` value
  outside `{spoken,thought,voiceover}`) and `E-DELIVERY-NARRATOR` (`delivery` present on a `narrator`
  speaker). Today `crates/lute-compile/src/lower.rs:20-24` maps any unrecognized `delivery` silently
  to `Role::Dialogue` and never flags `delivery` on `narrator`, contradicting §12.1. The checker
  gains the two diagnostics; `lower.rs` may keep its mapping (the checker is now the gate).
- **D5c — surface unchanged in 0.2.1 (forward note).** 0.2.1 enforces the *current*
  `delivery="spoken|thought|voiceover"` attribute form (non-breaking) — it fixes the silent
  miscompile now. The follow-on new-minor reshapes this surface into terse **bare flags**
  (`@bianca{mono}:` / `{os}` / `{vo}`, at-most-one) with a `lute fix` codemod — a breaking surface
  change, so it lives in the foundation spec (§D7 there), not here.
- Rationale for the fixed-vs-catalog split: `delivery` is **load-bearing** — the compiler branches
  on the specific member (`thought`→monologue, `voiceover`→voiceover) — so it is a closed core enum
  that no one extends, and belongs in 0.2.1. `emotion`/`action` are **pure vocabulary** the game
  defines; their authorable catalog is the new minor. (This is exactly the "fixed core enum vs
  author vocabulary" line the new-minor spec formalizes.)

## D6. Example hygiene (`docs/examples/`)

Three distinct frictions, three fixes:

- **D6a — project wiring.** Loose top-level examples that use plugin directives/providers
  (`date-minigame.lute` → `::minigame`, others → provider ids) emit `E-UNKNOWN-DIRECTIVE` /
  unresolved-id when opened without a project, because `docs/examples/` has no `lute.project.yaml`.
  Add project wiring so every example resolves a project when opened (a top-level project and/or the
  existing `idola-project`/`showcase` subprojects; each loose example lands under a project that
  activates the plugins it uses). The LSP already discovers `lute.project.yaml` by walking up
  (`crates/lute-lsp/src/backend.rs:513`), so wiring is sufficient — no LSP change needed for this
  part.
- **D6b — consistent naming.** Rename the topic-prefixed `extends-*.lute` to the role-suffix
  convention every other declaration file uses: `extends-base.lute` → `*.schema.lute`,
  `extends-child.lute` → `*.schema.lute` (both are schemas — state/defs, no `kind:`), and
  `extends-scene.lute` → an ordinary scene name (it is a `kind: scene` doc). Format stays `.lute` in
  0.2.1; the new minor migrates declaration files `.schema.lute` → `.yaml`. Standardizing the naming
  now makes that migration a clean rename.
- **D6c — kill the false `E-KIND-MISSING` on fragments.** `crates/lute-check/src/check.rs`
  (`fold_env`, ~L195-215) unconditionally calls `resolve_doc_kind` (`meta.rs` L143-190) on whatever
  document is checked and emits `E-KIND-MISSING` when there is no `kind:` — so opening an import-only
  `*.schema.lute`/`*.component.lute` fragment standalone red-underlines it, even though it is valid
  and validated transitively via `uses:`/`components:`. Fix by **frontmatter-shape role inference on
  the root document**: no `kind:` +
  - `component:`/`params:` shape → validate as `MetaKind::Component`;
  - schema shape (`state:`/`defs:`/… only, no body nodes) → validate as `MetaKind::Schema`;
  - otherwise (has body nodes but no `kind:`) → keep `E-KIND-MISSING` (a real scene that forgot its
    kind).

  Routing a fragment to `MetaKind::Schema`/`Component` also skips the scene-required-key check
  (`character`/`season`/`episode`), so this **additionally** clears the `E-META-MISSING` the
  `showcase/README.md` caveat currently warns about — the two false positives were always the same
  root cause (a fragment judged as a root scene). This suppresses both while preserving the errors
  for a genuine mistake, and is keyed on frontmatter shape (survives the future `.yaml` format change).
- **D6d — layout/naming reference.** Add/refresh `docs/examples/README.md` documenting the roles and
  the naming convention — the single "schema vs component vs scene vs project vs catalog vs plugin,
  which do I write?" reference (extending the existing `showcase/README.md`, which already documents
  the standalone-check caveat for `E-META-MISSING` but predates `E-KIND-MISSING`).

## D7. Diagnostics (0.2.1 delta)

| Code | Stage | Meaning |
|---|---|---|
| `E-DELIVERY-VALUE` | check | a content-line `delivery` value outside `{spoken, thought, voiceover}` (§12.1). |
| `E-DELIVERY-NARRATOR` | check | a `delivery` attribute present on a `narrator` speaker (§12.1). |

Content-line attribute key/type/enum errors reuse the existing directive diagnostics
(`E-UNKNOWN-ATTR`, `E-ATTR-TYPE`, `E-BAD-ENUM`) via the shared attr-schema path (D5a). `E-KIND-MISSING`
is unchanged in meaning; D6c narrows *when the root check runs it*, not the diagnostic itself.

## Testing (per area; run only these + existing suites)

- **Checker:** `delivery="thouhgt"` → `E-DELIVERY-VALUE`; `:narrator{delivery="thought"}:` →
  `E-DELIVERY-NARRATOR`; each legal `delivery` value → clean; a bad content-line `emotion`/attr →
  the reused directive diagnostic; a `*.schema.lute`/`*.component.lute` fragment checked standalone →
  **no** `E-KIND-MISSING`; a real scene missing `kind:` → still `E-KIND-MISSING`.
- **LSP:** cursor on a speaker name resolves `Cursor::Speaker`; speaker completion returns cast ids
  when a cast catalog is active; content-line attr key/value completion is non-empty for the built-in
  schema (esp. `delivery`).
- **Editors:** a TextMate scope test (or a fixture) asserting `<quest>`/`<on>`/`<objective>` and a
  modern `:speaker{…}: text` line receive scopes; the D3b drift guard fails on a desynced nvim query.
- **Examples:** every `docs/examples/**/*.lute` root scene checks clean under its wired project
  (`--project`), and opening any fragment standalone produces no `E-KIND-MISSING`.

## Non-goals (this pass)

The data-catalog unification, the `.schema.lute` → `.yaml` declaration-file format split, the shipped
JSON Schema + LSP-claims-declaration-YAML semantic lint, per-character/per-game `emotion`/`action`
catalogs, and the `:` → `@` speaker-sigil swap are all **out of scope for 0.2.1** and specified in
the follow-on new-minor design. 0.2.1 introduces **no** language grammar change.
