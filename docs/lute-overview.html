Lute는 대사·연출·퀘스트·세계 상태를 선언하는 저작 언어다. 실행은 엔진의 몫 — Lute는 총계적으로 검증된 JSON을 뱉는다.

문법 보기 왜 Lute인가
lute — 저작episode01.lute
## Shot 3. 노을
::bg{location="cafe" time="evening"}
@bianca{mono emotion="delighted"}: 드디어… 완벽한 서빙이었어.
@narrator: 창밖으로 빛이 천천히 기울었다.

<match on="run.affection">
  <when is="warm">
    @bianca: 같이 있어줘서, 고마워요.
  </when>
  <otherwise> @bianca: 그럼, 다음에 봐요. </otherwise>
</match>
§ 01

무엇인가

한 줄 한 줄이 커맨드 레코드 + CEL로 환원되는, 줄 단위 저작 언어.

기획팀이 의도를 적고, 엔진이 실현한다.

Lute 문서는 세 겹이 엮인다 — 콘텐츠(대사·나레이션), 연출(단일 라인 디렉티브), 로직(분기·매치·허브·타임라인·퀘스트). 조건은 전부 CEL 표현식이다.

모든 구성물은 빌드 타임에 유한하고 순서화된 커맨드 레코드 + CEL 문자열로 환원된다. 발더스게이트의 Osiris가 시뮬레이션 위에 얹힌 선언적 스크립트 층이듯, Lute는 그 자리를 차지하되 의도적으로 총계적(total)이다 — 무한 반복도, 재귀도, 뒤로 가는 점프도 없다. 스프라이트를 어떻게 그리고 주사위를 어떻게 굴릴지는 Lute가 정하지 않는다. 그건 엔진의 계약이다.

§ 02

왜 써야 하나

"닫힌" 언어가 주는 다섯 가지 — 전부 컴파일 타임에.

01

버그가 컴파일에서 죽는다

망라성(모든 분기 커버), 확정 할당(읽기 전 반드시 씀), 스테이지 조인, 종료 보장이 정적으로 검사된다. 런타임에 "이 분기 빠졌네"가 없다.

// total · non-Turing · 정적 검증
02

저작과 실행의 분리

작가는 "이 대사는 속마음", "전투 발동 조건은 이것", "죽으면 이 시나리오"를 선언한다. 실행·연출·전투는 엔진이 맡는다. QA·배틀팀과 깔끔히 갈린다.

// .lute → JSON IR → engine
03

자유 텍스트가 아니라 데이터 카탈로그

cast·action·emotion·delivery가 아무 문자열이나 아니다. 선언된 닫힌 어휘에 타입 체크되고, 에디터가 자동완성하며, 오타는 즉시 에러다.

// { domain: action } · providers
04

관계형 사실 DB (Datalog)

"파티에 클레릭 ≥ 3?", "이로하는 할신이 어디 있다고 믿는가?"를 파생 규칙으로. Prolog가 아니라 Datalog — 함수기호 없음, 유한 Herbrand, 항상 종료.

// holds() · count() · rules :-
05

문법은 고정, 어휘는 확장

플러그인은 데이터만 더한다 — 디렉티브·속성 도메인·id 레지스트리·에셋 종류. 문법도 동작도 건드리지 않는다. k8s CRD가 API를 넓히듯, capability가 Lute를 넓힌다.

// capability snapshot · profiles
06

진짜 툴링

Rust 체커·LSP·tree-sitter·VS Code·Neovim·lute fix 코드모드. 선언 .yaml은 JSON Schema(구조) + LSP(의미) 두 겹으로 린트된다.

// check · lsp · fix · schema
§ 03

어떻게 쓰나

화자는 @, 연출은 ::, 로직은 <>.

대사 · 연출

화자와 전달 방식

@narrator는 화자, {mono}는 속마음, {os}는 화면 밖. 짧은 bool 플래그는 JSX처럼.

content
@bianca{emotion="shy" code="0075"}: …누구세요?
@fixer{mono}: 조심스럽게 접근해야겠군.
@halsin{os}: (문 너머) 우리를 안쪽 숲으로.
::auto{character="bianca" action="fade-in-up"}
로직

분기와 값 매칭

<match>/<when>는 if/switch, is=는 등호 슈가. 모든 갈래가 커버돼야 통과한다.

logic
<branch>
  <choice into="run.metHelpfully">친절하게 묻는다</choice>
  <choice when="@fond">농담을 건넨다</choice>
</branch>

<match on="scene.affect.bianca">
  <when test="$ >= 3 && @veteran"> @fixer{mono}: 이제 익숙하다. </when>
  <otherwise/>
</match>
퀘스트 · 이벤트

조건으로 게이팅

<quest>는 start/fail, <objective>는 done을 사실 질의로 건다. <on>은 이벤트 트리거.

quest — kind: quest
<quest id="rescueHalsin" title="첫 드루이드를 구하라"
       start="holds(inParty(shadowheart))"
       fail="holds(atLocation(halsin, moonrise))">
  <objective id="reach" done="holds(canReach(player, grove))"/>
  <on event="npcSpoke">
    ::assert{ heardLocation(player, halsin, grove) }
    @halsin: "고맙네. 이제 길을 안내하지."
  </on>
</quest>
스키마 · 사실

세계를 선언한다

닫힌 entities/enums 위에 relations와 rules. 파생은 항상 유한하게 종료한다.

act1.schema.yaml
entities:
  character: { members: [shadowheart, halsin, player] }
  location:  { members: [camp, grove, moonrise] }
enums:
  action: [fade-in-up, wave, bow]        # 작가가 정의하는 어휘
relations:
  atLocation: { args: [character, location], tier: run, key: [0] }
  canReach:   { args: [character, location], derive: true }
rules:
  - "canReach(C, L)  :- atLocation(C, L)"
  - "canReach(C, L2) :- canReach(C, L1), connected(L1, L2)"
§ 04

파이프라인

저작물은 데이터로 환원되고, 엔진이 실체화한다.

01 · PARSE

.lute / .yaml

tree-sitter 문법 + 프론트매터. 줄 단위, 세 겹.

02 · CHECK

정적 검증

망라성·확정할당·도메인·Datalog 안전성·종료.

03 · COMPILE

JSON IR + CEL

커맨드 레코드 · role · lineId · 사실 델타.

04 · ENGINE

실체화

렌더·음성·전투·주사위. Lute 밖. 게임의 몫.

스냅샷-first: 컴파일은 핀된 카탈로그만 참조하고, 라이브/원격 소스에 절대 블로킹하지 않는다 — 재현 가능한 빌드.
