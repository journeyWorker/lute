## What Lute does, what your engine does

Lute is the author-time half: a grammar, a checker, and a compiler. It never
runs your game. Everything it knows becomes a versioned JSON IR plus CEL that
your engine loads through a documented [runtime contract](/tooling/runtime-contract/) —
engines never embed the Lute parser or compiler.

```text
writer
  │  writes
  ▼
.lute source ──▶ Lute [ parse · check · analyze · trace · compile · LSP ]
                   │  emits
                   ▼
        versioned JSON IR + CEL
                   │  loaded by
                   ▼
   engine adapter [ command loop · CEL eval · facts · quest lifecycle · bridges ]
                   │  drives
                   ▼
        UI · animation · save · gameplay
```

| Lute (author + toolchain) | Your engine (runtime) |
| --- | --- |
| Grammar + type checking | Rendering + input |
| Choice / `<match>` control data | Command execution |
| Quest declarations | Quest ticking |
| Facts + Datalog rules as data | Fact store + rule evaluation |
| Timeline data | Animation playback |
| Bridge signature checking | Minigame / QTE execution |
| Connectivity analysis | Scene entry + saves |

Read the full split on the [runtime contract](/tooling/runtime-contract/) page.

<span class="lute-kicker">The language</span>

## Scenes that read like scripts

Dialogue, player choice, and state-driven callbacks — the branch below always
gets the number; asking gently just warms her. `<match>` is first-match-wins
with a mandatory `<otherwise>`, so every branch is total.

```lute
## The Hostess with a Name
::camera{focus="bianca" zoom="1.1" duration="0.5"}

@bianca{emotion="surprised"}: Oh!
@bianca{emotion="delighted"}: Mr. Fixer!
@bianca{emotion="delighted"}: From the ramen counter — you eat at Mechageki too, isn't it wonderful?

/* a choice of TONE — both paths get the number; the gentle ask warms her */
<branch id="number">
  <choice id="blunt" label="Just ask, flatly">
    @fixer: Bianca. Your number.
  </choice>
  <choice id="soft" label="Ask gently">
    @fixer: Bianca — would you mind terribly if I had your number?
    ::set{scene.affect.bianca += 1}
  </choice>
</branch>

/* state-driven callback — first-match-wins; <otherwise> is mandatory */
<match on="scene.choices.number">
  <when test="@fond">
    @fixer{mono}: I asked nicely, which I am electing not to examine.
  </when>
  <otherwise>
    @fixer{mono}: I am, apparently, taking that woman somewhere this evening.
  </otherwise>
</match>
```

One choreographed beat, many concurrent tracks — split a single subject across
property-scoped tracks (`pos`, `opacity`) alongside an independent camera track:

```lute
<timeline duration="1.2">
  <track subject="bianca" property="pos">
    ::auto{character="bianca" anchor="left" action="slide-in-left"}
  </track>
  <track subject="bianca" property="opacity">
    ::auto{character="bianca" action="fade-in-up"}
  </track>
  <track subject="camera">
    ::camera{focus="bianca" zoom="1.2" duration="0.6"}
    ::camera{shake="0.4" duration="0.3" at="0.7"}
  </track>
</timeline>
```

## Investigation, checked

Story-driven RPGs run on world state, not just lines. A `<hub>` re-presents its
choices until you leave; a choice is offered only when its `when` guard holds; a
clue is a relational fact you `::assert`; and a closing `<match>` reacts to how
far the case has moved. The checker validates every relation signature, guard,
and match arm before a player ever sees it.

```lute
## The Study

@narrator: The lamp still burned. The ledger had already given you one name.

<hub id="investigate">
  <choice id="desk" label="Search the writing desk" once>
    @detective: A cufflink, wedged behind the drawer.
    ::assert{ found(cufflink) }
  </choice>
  <choice id="confront" label="Confront Mireille with the ledger" when="holds(found(ledger))">
    @mireille: ...You've read it, then. Fine.
    ::assert{ accused(mireille) }
    ::set{run.caseProgress += 1}
  </choice>
  <choice id="leave" label="Step back into the hall" exit>
    @detective: Enough for now.
  </choice>
</hub>

<match on="run.caseProgress">
  <when test="$ >= 1">
    @detective{mono}: A name, a motive. The case has a spine now.
  </when>
  <otherwise>
    @detective{mono}: Still only shadows. Keep looking.
  </otherwise>
</match>
```

<div class="rpg-outcome" role="group" aria-label="Investigation outcome">

**Clue discovered** — `found(cufflink)` lands in the run fact store; the `confront`
choice unlocks because `holds(found(ledger))` already holds.

**Quest updated** — `run.caseProgress` advances, and the closing `<match>` flips to
*"A name, a motive. The case has a spine now."*

</div>

## Built for scenarios you can trust

<CardGrid stagger>
  <Card title="Total, not Turing-complete" icon="approve-check">
    Every scenario provably terminates — no infinite loops, no unbounded
    recursion. That decidability is the reason the checker can analyze paths
    at all instead of guessing.
  </Card>
  <Card title="Typed plugin capabilities" icon="puzzle">
    Directives and providers are declared with typed manifests, so the
    checker knows exactly what each plugin can emit and consume.
  </Card>
  <Card title="CEL + Datalog facts" icon="list-format">
    Guards are plain <code>CEL</code> expressions; a Datalog fact layer lets
    you derive higher-order truths and query them across a scene.
  </Card>
  <Card title="Simulate before you ship" icon="rocket">
    <code>lute trace</code> runs one deterministic, mock-driven path — state,
    facts, choices, events — so you watch a scenario play out before anyone
    does. It walks that path, not every path.
  </Card>
  <Card title="Graph-checked connectivity" icon="random">
    Reachability runs over your declared <code>after:</code> routes and returns
    Reachable / Unreachable / Unknown — relational gates surface as human-review
    boundaries, never false certainty.
  </Card>
  <Card title="AI-ready authoring surface" icon="comment">
    <code>lute context</code> emits a structured view of a project built for
    agents — the language was designed to be written with a model, not despite one.
  </Card>
  <Card title="LSP + editors" icon="laptop">
    A language server ships diagnostics, hovers, and completions to your
    editor, so mistakes surface as you type, not at runtime.
  </Card>
  <Card title="Compiles to a versioned JSON IR" icon="document">
    Engines don't embed the Lute parser or compiler; they implement the
    documented <a href="/tooling/runtime-contract/">runtime contract</a> against
    a versioned JSON IR plus CEL — any engine, any language.
  </Card>
</CardGrid>
